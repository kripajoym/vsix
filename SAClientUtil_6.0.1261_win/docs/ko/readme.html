Static Analyzer 시작하기

정적 분석을 사용하여 소스 코드를 스캔해서 보안 취약점이 있는지 검사합니다. 이를 수행하려면 소형 유틸리티를 다운로드해야 합니다. 로컬 디스크로 유틸리티를 추출하는 경우에는 해당 명령행 인터페이스(CLI)를 사용하여 모든 지원 언어에서 보안 분석을 수행할 수 있습니다. 또는 정적 분석 플러그인을 Eclipse, Maven, IntelliJ IDEA 또는 Visual Studio에 추가하는 포함된 설치 프로그램을 실행할 수 있습니다. Eclipse 및 IntelliJ IDEA에서는 Java 프로젝트를 스캔할 수 있습니다. 그리고 Visual Studio에서는 .NET (C#, ASP.NET, VB.NET)를 스캔할 수 있습니다.

프로시저

  • IBM Cloud Marketplace의 Application Security on Cloud 서비스 사용 방법에 대해 알아보려면 http://www.ibm.com/support/knowledgecenter/SSYJJF_1.0.0/ApplicationSecurityonCloud/Welcome.html을 참조하십시오.
  • Application Security on Cloud for Bluemix의 사용 방법에 대해 알아보려면 https://console.ng.bluemix.net/docs/services/ApplicationSecurityonCloud/index.html을 참조하십시오.
  • 이 readme 파일의 나머지 부분에서는 Static Analyzer Client Utility 설치 및 사용법에 대해 다룹니다.

최근의 정적 분석 업데이트

Application Security on Cloud 서비스에 최근 추가된 정적 분석 기능에 대해 알아봅니다.

Static Analyzer Client Utility 설정

정적 분석의 경우 소형 Client Utility를 다운로드합니다. 이 유틸리티를 로컬 디스크에 추출하는 경우 해당 명령행 인터페이스(CLI)를 사용하여 보안 분석을 수행할 수 있습니다. 또는 정적 분석 플러그인을 Maven 빌드 환경이나 Eclipse, IntelliJ IDEA 또는 Visual Studio IDE(Integrated Development Environment)에 추가하는 포함된 설치 프로그램을 실행할 수 있습니다.

이 태스크 정보

정적 분석을 사용하면 소스 코드를 스캔하여 클라우드에 업로드되는 파일을 생성합니다. 추적 분석으로 암호화된 IRX (.irx) 파일인 해당 파일을 스캔하여 보안 취약성을 찾습니다. 컴파일된 언어에 대해 IRX 파일이 생성되면 애플리케이션 바이트 코드 파일이 코드의 Intermediate Representation으로 변환됩니다. 스크립팅 언어에 대해 IRX 파일이 생성되면 소스 파일이 암호화된 IRX 파일에 포함됩니다.

Eclipse, IntelliJ IDEA 또는 Visual Studio 플러그인을 사용하는 경우 정적 분석 조치를 사용하면 IDE에서 IRX 파일 생성 및 업로드가 자동으로 수행됩니다.

팁: 설치 프로그램을 실행하는 경우 특정한 IDE에서 정적 분석을 사용할 계획은 없지만 해당 IDE가 설치되어 있고 나중에 사용할 예정이라면 설치 프로그램을 나중에 다시 실행할 필요가 없도록 지금 IDE에 플러그인을 설치할 수 있습니다.
빠른경로: 설치 프로그램에서 yes 또는 no를 입력하는 경우 대신 약어로 y 또는 n을 입력할 수 있습니다. 영어가 아닌 언어를 사용하는 시스템에서 실행 중인 설치 프로그램에 약어를 입력하는 경우 입력하는 약어는 y 또는 n이어야 합니다.
중요사항: Client Utility를 비ASCII 문자가 포함된 디렉토리에 배치해서는 안됩니다.

프로시저

Client Utility를 설정하려면 다음을 수행하십시오.

  1. Client Utility를 실행할 플랫폼을 선택하고 다운로드를 클릭하십시오. 이렇게 하면 SAClientUtil_<version>_<os>.zip 파일이 다운로드됩니다. 여기서 <version>은 Client Utility의 현재 버전이고 <os>는 Client Utility의 운영 체제입니다.
    참고: IRX 파일을 선택하기 위한 양식으로 IRX 파일을 작성하는 방법을 선택하여 시작 양식을 다시 열 수 있습니다.
  2. 로컬 드라이브에 파일의 압축을 푸십시오.
  3. IRX 파일 생성, IRX 파일 업로드 또는 스캔 관리를 위해 CLI를 사용할 경우: 추출된 SAClientUtil_<version>_<os>.zip 파일의 \bin 디렉토리 위치를 PATH 환경 변수에 추가하십시오. 이를 수행하지 않으면 명령이 실행될 때마다 추출된 SAClientUtil_<version>_<os>.zip 파일의 \bin 디렉토리를 사용하여 모든 명령을 규정해야 합니다.
    중요사항: CLI 또는 Microsoft Visual Studio로 .NET을 스캔할 경우 다음과 같은 세 개의 .bat 파일(추출된 SAClientUtil_<version>_<os>.zip 파일)을 관리자로 실행해야 합니다.
    • bin\register.bat
    • walalib\dotnet.libs40\RegisterInGAC.bat
    • walalib\dotnet.libs35\RegisterInGAC.bat

    이러한 파일을 수동으로 실행할 수도 있고 관리자로 install-plugins.bat 파일을 실행할 수도 있습니다. CLI만 사용할 경우 install-plugins.bat를 실행하지만 정적 분석 플러그인을 설치하라는 프롬프트가 표시되면 no(아니오)를 입력하십시오.

    이러한 파일을 관리자로 실행하지 않은 경우 CLI를 사용할 때마다 경고 메시지가 콘솔에 나타납니다. Java 대상을 스캔하는 경우에는 이 메시지를 무시할 수 있습니다.

    팁: PATH를 변경한 후 명령 프롬프트에서 appscan version (Windows) 또는 appscan.sh version (Linux 및 macOS)을 실행하십시오. Static Analyzer Client Utility 버전, 홈 및 기타 정보가 리턴되면 PATH가 올바르게 설정된 것입니다.
  4. 정적 분석 플러그인을 Eclipse, Microsoft Visual Studio(Windows만 해당), IntelliJ IDEA 또는 Maven에 추가할 경우: 추출된 Client Utility에서 install-plugins.bat(Windows) 또는 install-plugins.sh(Linux 및 macOS)를 찾아서 실행하십시오. Windows에서는 관리자로 파일을 실행하십시오. Linux 및 macOS에서 Eclipse에 플러그인을 추가할 경우 설치 중인 Eclipse 인스턴스를 수정할 권한이 있는 계정에 로그인되었는지 확인하십시오.
    1. 기본적으로 Client Utility는 IBM Cloud Marketplace의 ASoC 서비스에 연결됩니다. 대신 IBM Bluemix®의 서비스에 연결할 경우 프롬프트가 표시되면 yes를 입력하십시오.
    2. Eclipse에서 소스 코드를 스캔할 경우 설치 프로그램의 프롬프트를 따르십시오. Eclipse 플러그인을 설치하도록 선택하면 Eclipse 설치 위치를 입력하라는 프롬프트가 표시됩니다. eclipse.exe(Windows) 또는 eclipse(Linux 및 macOS) 파일이 포함된 디렉토리의 전체 경로를 입력하십시오.

    4. IntelliJ IDEA에서 소스 코드를 스캔할 경우 설치 프로그램의 프롬프트를 따르십시오. IntelliJ IDEA 플러그인을 설치하도록 선택할 경우, IntelliJ IDEA 구성 디렉토리의 위치를 입력하라는 프롬프트가 표시됩니다. 이 디렉토리의 전체 경로를 입력하십시오.
    5. Windows에만 해당: Visual Studio에서 소스 코드를 스캔할 경우 설치 프로그램의 프롬프트를 따르십시오.Visual Studio 플러그인을 설치하도록 선택하는 경우 플러그인을 설치해야 하는 Visual Studio의 버전을 선택하라는 프롬프트가 표시됩니다. 플러그인을 설치할 Visual Studio 버전에 해당하는 버전을 입력하십시오.
    6. 아무 키나 눌러 명령 프롬프트를 닫으십시오.
    7. 플러그인을 설치한 IDE의 경우 IDE를 다시 시작하십시오.

      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export LD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${LD_LIBRARY_PATH}

      여기서 <client_utility>는 Client Utility의 경로입니다. 이러한 행을 추가한 후 정적 분석 플러그인을 사용하기 전에 쉘을 다시 로드하십시오.

    9. macOS 전용: 명령 프롬프트에서 클라이언트 유틸리티가 포함된 디렉토리를 찾은 후 다음 행을 사용자의 쉘 자원 파일(예: .bash_profile 또는 .shrc)에 추가하십시오. 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export DYLD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${DYLD_LIBRARY_PATH}

      여기서 <client_utility>는 Client Utility의 경로입니다. 이러한 행을 추가한 후 정적 분석 플러그인을 사용하기 전에 쉘을 다시 로드하십시오.

      중요사항: 쉘 자원 파일에 환경 변수를 설정하면 터미널에서 실행된 앱에만 영향을 미칩니다. 따라서 정적 분석을 실행할 IDE(Integrated Development Environment)를 실행하기 위한 터미널 창을 사용해야 합니다.

프록시 뒤에 있는 컴퓨터에서 Client Utility를 실행 중인 경우 -다음 방법 중 하나를 사용하여 Client Utility에서 클라우드에 연결할 수 있도록 프록시를 지정하십시오.

  • Client Utility CLI 및 지원되는 IDE(Integrated Development Environment): 프록시가 자동으로 인식되도록 다음 글로벌 또는 시스템 환경 변수를 설정하십시오.
    • Windows: APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>
    • Linux and macOS: APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"

    여기서 <proxy>는 프록시 서버의 호스트 이름이고 <port>는 프록시 서버에서 사용 중인 포트 번호입니다.

    또는 (명령 프롬프트 또는 터미널에서 실행된) CLI 또는 IDE를 사용할 때마다 명령을 실행하여 Client Utility가 프록시를 사용하도록 설정할 수 있습니다.

    • Windows: set "APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
    • Linux and macOS: export APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
  • Maven: 정적 분석 플러그인을 Maven에 추가한 경우 MAVEN_OPTS 환경 변수에 특성을 글로벌로 추가하거나 각 명령에 특성을 추가할 수 있습니다.

Static Analyzer Client Utility 제거

이 태스크에서는 IDE(Integrated Development Environment)에서 Static Analyzer Client Utility 플러그인을 제거하는 방법을 설명합니다. 또한 Client Utility 제거를 위한 지시사항이 제공됩니다. 시스템에서 Static Analyzer Client Utility 컴포넌트를 제거하려는 경우 해당 지시사항만 따라야 합니다. 해당 컴포넌트를 새 Client Utility 버전으로 업그레이드하는 경우 새 버전을 설치하기 전에 플러그인 설치가 이전 컴포넌트를 자동으로 제거합니다.

시작하기 전에

중요사항: Client Utility 플러그인을 IDE에 설치한 경우 추출된 SAClientUtil_<version>_<os>.zip 위치에서 하나 이상의 파일 또는 명령을 실행하여 IDE 플러그인을 완전히 제거해야 합니다. 해당 파일은 다음과 같습니다.
  • Static Analyzer Client Utility 버전 6.0.865 및 이전 버전:
    • walalib\dotnet.libs35\UnRegisterInGAC.bat
    • walalib\dotnet.libs40\UnRegisterInGAC.bat
  • 버전 6.0.865 및 이전 버전을 포함한 모든 Static Analyzer Client Utility 버전: 관리자로 명령 프롬프트를 열고 SAClientUtil_<version>_<os>.zip을 추출한 위치로 디렉토리를 변경하십시오. bin\ 폴더로 디렉토리를 변경한 후 이 명령을 실행하십시오. 
    regsvr32 /u msdia80.dll

해당 파일 및 명령을 실행하려면 모든 IDE 플러그인이 제거될 때까지 추출된 SAClientUtil_<version>_<os>.zip 컨텐츠를 삭제하지 마십시오.

IDE에서 플러그인을 제거하기 위한 아래의 지시사항을 따르기 전 또는 후에 해당 파일 및 명령을 실행할 수 있습니다.

    4. 지금 다시 시작을 선택하여 Visual Studio를 다시 시작하거나 이후에 Visual Studio를 다시 시작하려는 경우 닫기를 선택하십시오.
  • Client Utility: Client Utility를 제거하려면 시스템에서 추출된 SAClientUtil_<version>_<os>.zip 컨텐츠만 삭제해야 합니다. Client Utility \bin 디렉토리의 위치를 PATH 환경 변수에 추가한 경우 이를 제거하십시오.

  • appscan prepare 명령(Windows) 또는 appscan.sh prepare 명령(Linux 및 macOS)을 실행하십시오.

더 이상 지원되지 않는 Client Utility 버전에서위의 조치들을 수행하면, Client Utility를 업데이트해야 한다는 메시지를 계속 수신하게 됩니다. 이런 경우, 최신 Client Utility를 다운로드 및 설정하십시오.

명령행 인터페이스(CLI)를 사용하여 IRX 파일 생성

파일의 분석을 시작하려면 먼저 클라우드에 제출할 IRX 파일을 생성해야 합니다. CLI를 사용하려면 다음 지시시항에 따라 해당 파일을 작성하십시오.

프로시저

IRX 파일을 생성하려면 다음을 수행하십시오.

  1. 새 명령 프롬프트 창을 열었는지 확인하십시오. Client Utility를 설치하는 데 사용된 명령 프롬프트 창을 사용하지 마십시오.
  2. 명령 프롬프트에서, 파일이 포함된 위치로 디렉토리를 변경하는 명령을 실행하십시오. 지원되는 파일 유형에 대해 파악하려면 언어 지원을 참조하십시오. Eclipse에서 Java 프로젝트를 스캔 중인 경우에는 Eclipse 작업공간으로 디렉토리를 변경하십시오.
    참고:
    • 구성 파일을 사용하여 IRX 파일을 생성할 경우 이 단계가 필요하지 않습니다. CLI로 IRX 파일 생성 구성을 참조하십시오.
    • CLI를 사용하여 Eclipse 작업공간에서 개별 프로젝트를 스캔할 수 없습니다. 전체 작업공간만 스캔할 수 있습니다. 개별 프로젝트를 스캔하려는 경우 Eclipse에서 IRX 파일을 생성해야 합니다.
    • .dll 및 .exe 파일이 올바른 .NET 어셈블리여야 합니다.
  3. appscan prepare (Windows) 또는 appscan.sh prepare (Linux 및 macOS) 명령을 실행하십시오. 이 명령은 구성 명령(Windows) 및 구성 명령(Linux 및 macOS) 주제에 나열된 옵션과 함께 사용됩니다. 예를 들어, -d 및 -n 옵션을 사용하여 IRX 파일을 저장할 특정 디렉토리를 지정하고 특정 파일 이름으로 파일을 저장할 수 있습니다. 또한 구성 파일을 사용하여 명령을 실행하면 완전한 IRX 파일을 생성하는 데 도움이 될 수 있는 추가 정보를 지정할 수 있습니다(CLI로 IRX 파일 생성 구성 참조).
    팁: 기본적으로, 써드파티 Java 및 .NET 코드는 IRX 파일 생성 중에 스캔되지 않습니다. 써드파티 Java 및 .NET 제외 관리의 지시사항을 따라 써드파티 코드 제외 설정을 수정할 수 있습니다. 써드파티 코드를 포함하려면 prepare 명령을 실행할 때 -t 또는 --thirdParty 옵션을 지정하십시오.

    정상적으로 스캔에 포함되는 써드파티 코드의 개발자인 경우 설정을 사용하여 써드파티 코드를 포함시켜야 합니다.

    참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.
  4. 명령이 실행된 후 Client Utility가 명령이 실행된 디렉토리 및 모든 해당 서브디렉토리에서 지원되는 파일에 대한 정보를 수집합니다. IRX 파일이 디렉토리에 작성됩니다. 이 파일의 이름은 명령이 실행된 디렉토리를 기반으로 하며 날짜 및 시간소인을 포함합니다(다른 파일 이름을 지정하기 위해 구성 옵션을 사용하지 않았다고 가정함).
  5. 분석을 위해 클라우드에 IRX 파일을 제출하십시오. 이를 수행하려면 appscan queue_analysis 명령((Windows)) 또는 appscan.sh queue_analysis 명령((Linux 및 macOS))을 실행하십시오. 이 명령은 분석 명령 (Windows) 또는 분석 명령 (Linux 및 macOS) 주제에 나열되어 있는 옵션과 함께 사용됩니다.

Maven 프로젝트를 위한 IRX 파일 생성

파일의 분석을 시작하려면 먼저 클라우드에 제출할 IRX 파일을 생성해야 합니다. Maven 프로젝트를 스캔하려면 다음 지시사항에 따라 해당 파일을 작성하십시오.

프로시저

IRX 파일을 생성하려면 다음을 수행하십시오.

  1. 새 명령 프롬프트 창을 열었는지 확인하십시오. Client Utility를 설치하는 데 사용된 명령 프롬프트 창을 사용하지 마십시오.
  2. 명령 프롬프트에서 명령을 실행하여 Maven 프로젝트가 포함된 위치(pom.xml 파일이 포함된 폴더)로 디렉토리를 변경하십시오.
  3. 다음 명령 중 하나를 실행하십시오.
    • Maven 프로젝트를 아직 빌드하지 않은 경우 mvn package com.ibm.appscan:appscan-maven-plugin:prepare 명령을 실행하십시오. 명령이 실행된 후 IRX 파일이 Maven 프로젝트의 target 디렉토리에 작성됩니다. 이 파일의 이름은 다른 파일 이름을 지정하기 위한 구성 옵션을 사용하지 않았다고 가정하여 최상위 레벨 Maven 프로젝트(pom.xml 파일에 있는 프로젝트 이름)에 따라 지정됩니다.
    • Maven 프로젝트를 이미 빌드한 경우 mvn com.ibm.appscan:appscan-maven-plugin:prepare 명령을 실행하십시오. 명령이 실행되면 IRX 파일이 Maven 프로젝트의 target 디렉토리에 작성됩니다. 이 파일의 이름은 다른 파일 이름을 지정하기 위한 구성 옵션을 사용하지 않았다고 가정하여 최상위 레벨 Maven 프로젝트(pom.xml 파일에 있는 프로젝트 이름)에 따라 지정됩니다.
      참고: Maven 프로젝트가 동일한 리액터 빌드에 있는 다른 프로젝트에 대한 컴파일 시간 종속성을 가지는 경우 기존 Maven 버그(https://issues.apache.org/jira/browse/MNG-3283)로 인해 mvn com.ibm.appscan:appscan-maven-plugin:prepare 명령이 실패합니다. 이 문제를 해결하려면 다음 단계 중 하나를 수행하십시오.
      • Client Utility prepare 명령을 실행하기 전에 mvn package 대신 mvn install 명령을 실행하십시오.
      • mvn package 명령과 Client Utility prepare 명령을 동시에 실행하십시오(mvn package com.ibm.appscan:appscan-maven-plugin:prepare 실행).
    • IRX 파일을 다른 디렉토리에 작성하거나 다른 파일 이름을 지정하려는 경우 -Doutput 옵션을 사용하여 명령을 실행하십시오.

      Windows에서는 다음 명령을 실행하십시오.

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<output_location>\<filename>.irx

      Linux 및 macOS에서는 다음 명령을 실행하십시오.

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<output_location>/<filename>.irx

      여기서, <output_location>은 파일을 저장하려는 디렉토리이고 filename은 파일의 이름입니다.

    • Client Utility에는 Tomcat JSP 컴파일러가 포함되어 있습니다. 이 JSP 컴파일러는 기본적으로 IRX 파일 생성 중에 사용됩니다. WAR 또는 EAR 파일을 위한 IRX 파일을 생성 중이고 다른 JSP 컴파일러가 사용되도록 하려면 -DjspCompiler 옵션을 사용하여 명령을 실행하십시오. 예: 
      mvn com.ibm.appscan:appscan-maven-plugin:prepare -DjspCompiler=<jsp_compiler>

      여기서, <jsp_compiler>는 사용하려는 JSP 컴파일러의 경로입니다(서버의 루트 디렉토리를 지정하십시오).

    팁: 기본적으로 써드파티 코드는 IRX 파일 생성 중에 스캔되지 않습니다. 써드파티 Java 및 .NET 제외 관리의 지시사항을 따라 써드파티 코드 제외 설정을 수정할 수 있습니다. 써드파티 코드를 포함하려면 -DthirdParty 옵션을 지정하여 명령을 실행하십시오.

    정상적으로 스캔에 포함되는 써드파티 코드의 개발자인 경우 설정을 사용하여 써드파티 코드를 포함시켜야 합니다.

    참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.
  4. 분석을 위해 클라우드에 IRX 파일을 제출하십시오. 이를 수행하려면 appscan queue_analysis 명령((Windows)) 또는 appscan.sh queue_analysis 명령((Linux 및 macOS))을 실행하십시오. 이 명령은 분석 명령 (Windows) 또는 분석 명령 (Linux 및 macOS) 주제에 나열되어 있는 옵션과 함께 사용됩니다.

구성 파일 사용법

구성 파일은 IRX 파일 생성 시 포함하거나 제외할 대상을 나타내는 데 사용할 수 있습니다. 또한 빌드된 아티팩트를 포함하는 디렉토리를 지정할 수 있으며 IRX 파일 생성 중에 사용해야 하는 빌드 정보를 지정할 수 있습니다. 예를 들어, 이 설정을 사용하여 특정 JDK 또는 JSP 컴파일러를 지정할 수 있습니다.

appscan prepare (Windows) 또는 appscan.sh prepare (Linux 및 macOS) 명령을 실행하여 IRX 파일을 생성할 때마다 Static Analyzer Client Utility가 현재 디렉토리에서 appscan-config.xml 파일을 자동으로 검사합니다. 발견되는 경우 파일이 자동으로 사용됩니다.

다른 디렉토리에 구성 파일이 있거나 다른 이름이 지정된 경우, appscan prepare 또는 appscan.sh prepare 명령을 실행할 때 -c 옵션을 포함시켜서 파일을 사용할 수 있습니다. 이 경우 (Windows)에서는 appscan prepare -c <configuration_file>을 실행하고 (Linux 및 macOS)에서는 appscan.sh prepare -c <configuration_file>을 실행할 수 있습니다. 여기서 <configuration_file>은 구성 파일의 전체 경로 및 파일 이름입니다.

구성 파일 템플리트 및 설정

구성 파일을 준비할 때 다음 템플리트를 사용하십시오.

<Configuration>
  <Targets>
    <Target outputs-only="true/false" path="scan_target_path">
      <CustomBuildInfo build_info="info"/>
      <Include>string_pattern</Include>
      <Exclude>string_pattern</Exclude>
    </Target>
  </Targets>
</Configuration>

여기서는 Targets 요소를 사용하여 IRX 파일 생성 시에 스캔할 대상을 표시할 수 있습니다. 대상(Target) 속성 및 하위 요소는 다음과 같이 사용됩니다.

  • Target 요소 outputs-only 속성은 디렉토리를 스캔하고 Client Utility에서 빌드 출력 파일(예: .jar, .war 및 .class 파일)을 강제로 찾도록 하려는 경우에만 사용됩니다. 기본적으로 이 속성은 false로 설정됩니다. 이는 Client Utility에서 디렉토리를 검색하여 디렉토리가 대상인지(예: 애플리케이션 서버 또는 Eclipse 작업공간) 또는 디렉토리에 빌드 스크립트, Maven .pom 파일 및 make 파일 등의 항목이 포함되어 있는지 여부를 판별한다는 것을 의미합니다. 스캔 대상을 단순 디렉토리로 처리하고 출력 파일만 찾도록 하려는 경우, Target 요소에서 outputs-only="true"를 지정하십시오.

    예를 들어, <Target outputs-only="false" path="C:\Tomcat">을 지정할 경우 Client Utility가 대상이 Tomcat 애플리케이션 서버라고 가정하고 배치된 해당 .war 파일을 검색합니다. 그러나 <Target outputs-only="true" path="C:\Tomcat">을 지정할 경우 Client Utility가 해당 위치를 디렉토리로 처리하며 내부의 모든 빌드 출력 파일을 찾습니다.

  • Target 요소 path 속성을 사용하여 스캔 대상의 디렉토리 또는 스캔 대상의 경로(<scan_target_path>)를 지정합니다. 디렉토리가 지정되면 모든 해당 하위 디렉토리가 스캔 동안 포함됩니다.

    예를 들어, <Target outputs-only="false" path="C:\WebSphere\AppServer85\profiles\AppSrv01">을 지정하는 경우 Client Utility가 AppSrv01 프로파일에서 배치된 모든 .ear 파일을 찾습니다. <Target outputs-only="false" path="C:\WebSphere\AppServer85">를 지정하는 경우 Client Utility가 모든 프로파일에서 배치된 모든 .ear 파일을 찾습니다.

  • 다음과 같은 선택적 Target 하위 요소를 지정할 수도 있습니다.
    • CustomBuildInfo 요소를 사용하여 속성을 설정할 수 있습니다. 템플리트의 다음 파트를 사용하십시오. 
      <CustomBuildInfo build_info="info"/>
      그러면 대상 언어에 따라 빌드 정보를 지정할 수 있습니다. 일부 언어의 경우에는 둘 이상의 속성을 설정할 수 있습니다. 예를 들어, 대상 언어에 따라 <CustomBuildInfo build_info_1="info_1" build_info_2="info_2" build_info_3="info_3"/>을 설정할 수 있습니다.
      • Java의 경우에는 다음 속성을 설정할 수 있습니다. 
        <CustomBuildInfo additional_classpath="dependency_path"
  jdk_path="JDK_path" jsp_compiler="JSP_compiler_path"/>
        • additional_classpath 속성을 사용하여 추가 클래스 경로를 지정합니다. Windows에서는 여러 클래스 경로를 세미콜론으로 구분하십시오. Linux 및 macOS에서는 여러 클래스 경로를 콜론으로 구분하십시오.
        • jdk_path 속성을 사용하여 JDK 설치의 경로를 지정합니다.
        • jsp_compiler 속성을 사용하여 JSP 컴파일러의 경로(예: jsp_compiler="C:\Tomcat")를 지정합니다.
      • .NET(Windows에만 해당)의 경우에는 다음 속성을 설정할 수 있습니다. 
        <CustomBuildInfo reference="assembly_reference"
  configuration="build_configuration"/>
        • reference 속성을 사용하여 어셈블리 참조를 추가합니다. 참조가 여럿이면 세미콜론으로 구분하십시오.
        • configuration 속성을 사용하여 Visual Studio 솔루션 검색을 위한 빌드 구성을 포함합니다.
      • C/C++(Windows에만 해당)의 경우에는 다음 속성을 설정할 수 있습니다. 
        <CustomBuildInfo configuration="build_configuration"
  include_paths="include_directories" macros="macros" compiler_opts=/>
        • configuration 속성을 사용하여 빌드 구성을 포함합니다.
        • include_paths 속성을 사용하여 포함 경로를 지정합니다. 포함 경로가 여럿이면 세미콜론으로 구분하십시오.
        • macros 속성을 사용하여 매크로를 포함합니다. 매크로가 여럿이면 세미콜론으로 구분하십시오.
        • compiler_opts 속성을 사용하여 컴파일러 옵션을 지정합니다. 옵션이 여럿이면 세미콜론으로 구분하십시오.
    참고: 하위 대상은 이러한 속성으로 설정된 값을 상속합니다. 예를 들어, 대상이 WAR 및 JAR 하위 대상을 포함하는 EAR 파일인 경우, WAR 및 JAR이 이러한 속성을 사용하여 EAR 파일에 설정된 동일한 값을 가지고 있다고 가정합니다.
  • IRX 파일을 생성할 경우 포함할 파일 패턴(<string_pattern>)을 지정하려면 Include 하위 요소를 사용하십시오. 대상 포함 및 제외 동작 섹션에 설명된 대로 Include 동작은 대상 유형에 따라 달라집니다. 여러 include 패턴을 지정하려면 각 패턴을 고유의 <Include></Include> 태그에 추가하십시오. 예를 들면 다음과 같습니다. 
    <Include>string_pattern_1</Include>
<Include>string_pattern_2</Include>
    참고: 지정한 include 및 exclude 패턴이 충돌되는 경우 exclude 패턴이 우선적으로 적용됩니다.
  • IRX 파일을 생성할 경우 제외할 파일 패턴을 지정하려면 Exclude 하위 요소를 사용하십시오. 대상 포함 및 제외 동작 섹션에 설명된 대로 Exclude 동작은 대상 유형에 따라 달라집니다. 여러 exclude 패턴을 지정하려면 각 패턴을 고유의 <Exclude></Exclude> 태그에 추가하십시오.
    참고: 지정한 include 및 exclude 패턴이 충돌되는 경우 exclude 패턴이 우선적으로 적용됩니다.

대상 포함 및 제외 동작

표 2. 대상 포함 및 제외 동작
대상 유형 동작 예
.jar, .dll 대상이 .jar 또는 .dll 파일이면 대상의 포함 또는 제외가 스캔에 영향을 주지 않습니다. 이러한 파일에 대상이 포함될 수 없기 때문입니다.  
.war 대상이 .war 파일인 경우, WEB-INF/lib의 .jar 파일을 포함 또는 제외시킬 수 있습니다.
참고: 기본적으로 .jar 파일은 써드파티 코드로 처리되고 .war 파일의 클래스 경로에 추가됩니다(또한 기본적으로 제외됨).
com.ibm.*.jar
.ear 대상이 .ear 파일인 경우, .jar 및 .war 파일을 포함 또는 제외시킬 수 있습니다.
  • com.ibm.*.jar
  • JSPWiki.war
디렉토리 대상이 디렉토리인 경우, 모든 대상 또는 대상 유형의 파일 이름이나 경로를 포함 또는 제외시킬 수 있습니다.
  • com.ibm.*.jar
  • test/: test/ 디렉토리와 해당 하위 디렉토리의 모든 항목을 포함 또는 제외시킵니다.
  • test/*.jar: test/ 디렉토리의 모든 .jar 파일을 포함 또는 제외시키지만 하위 디렉토리는 해당되지 않습니다.
웹 서버(지원되는 웹 서버에 대해 알아보려면 시스템 요구사항 참조) 대상이 웹 서버인 경우, 서버에 배치된 모든 애플리케이션의 파일 이름을 포함 또는 제외시킬 수 있습니다.
  • JSPWiki.war
  • MyApp.ear
  • *.war(대상이 Tomcat 서버인 경우)
.sln(.NET) 대상이 .NET 프로젝트가 포함된 Visual Studio 솔루션 파일이면 솔루션의 프로젝트에서 작성한 .NET 어셈블리의 파일 이름을 포함 또는 제외시킬 수 있습니다.
  • *\MyLib.dll
  • *\MyApp.exe
.sln(C/C++) 대상이 비관리 C/C++ 프로젝트가 포함된 Visual Studio 솔루션 파일이면 프로젝트의 파일 이름을 포함 또는 제외시킬 수 있습니다.
  • *\myProject.vcxproj

구성 파일 예

<Configuration>
  <Targets>
    <Target path="C:\test\MyApplication.ear">
      <CustomBuildInfo additional_classpath="C:\java\lib;C:\java2\lib"
        jdk_path="C:\Program Files\Java\jdk1.7" jsp_compiler="C:\Tomcat"/>
      <Include>ShoppingCart.war</Include>
      <Include>*Account.jar</Include>
      <Exclude>thirdPartyJars/</Exclude>
      <Exclude>*test*</Exclude>
    </Target>
  </Targets>
</Configuration>
이 구성 파일에는 다음이 적용됩니다.
  • C:\test\MyApplication.ear이 대상입니다.
  • C:\java\lib 및 C:\java2\lib에서 컴파일에 필요한 모든 클래스 파일 종속 항목이 검색됩니다.
  • C:\Program Files\Java\jdk1.7은 종속 항목을 찾는 데 사용됩니다.
  • C:\Tomcat 컴파일러는 Static Analyzer Client Utility에 포함된 Apache Tomcat 버전 7 애플리케이션 대신 JSP 컴파일에 사용됩니다.
  • IRX 파일이 생성되면 *Account.jar의 끝에 있는 모든 파일과 같이 ShoppingCart.war 파일이 포함됩니다. EAR 파일의 thirdPartyJars/에 있는 모든 항목은 파일 이름에 test 문자열이 있는 모든 파일처럼 제외됩니다. 충돌이 있는 경우, exclude 패턴이 우선순위를 가집니다. 예를 들어, EAR 파일에 testCustomerAccount.jar 파일이 포함된 경우, test exclude 패턴이 *Account.jar include 패턴보다 우선권을 가지므로 이 파일은 제외됩니다.

통합 개발 환경에서 스캔

다음 지시사항은 정적 분석 플러그인이 Eclipse, IntelliJ IDEA 또는 Visual Studio IDE(Integrated Development Environment)에 설치된 후 이를 사용하여 소스 코드를 스캔하는 방법에 대해 설명합니다. Eclipse 및 IntelliJ IDEA에서는 Java 프로젝트를 스캔할 수 있습니다. 그리고 Visual Studio에서는 .NET (C#, ASP.NET, VB.NET)를 스캔할 수 있습니다.

시작하기 전에

기본적으로, Java and .NET을 스캔할 때는 써드파티 코드가 포함되지 않습니다. 써드파티 Java 및 .NET 제외 관리의 지시사항을 따라 써드파티 코드 제외 설정을 수정할 수 있습니다.
  • Eclipse, IntelliJ IDEA 또는 Visual Studio에서 스캔 중일 때 써드파티 코드를 포함하려면 다음 방법 중 하나를 사용하십시오.
    • ID를 시작하기 전에 이 글로벌 또는 시스템 환경 변수를 설정하십시오. 
      APPSCAN_OPTS=-DthirdParty
    • IDE를 사용할 때마다 IDE 시작 전에 다음 명령을 실행할 수 있습니다. 
      set APPSCAN_OPTS=-DthirdParty
  • 또는 Eclipse에서 스캔할 때는 -vmargs 섹션에 -DthirdParty가 포함되도록 Eclipse를 시작하기 전에 eclipse.ini 파일을 수정할 수 있습니다.
  • 또는 IntelliJ IDEA에서 스캔할 때는 IntelliJ IDEA 설치의 bin 디렉토리에서 IntelliJ IDEA를 시작하기 전에 idea.exe.vmoptions 파일을 수정할 수 있습니다. 파일에 -DthirdParty를 추가하고 이를 저장하십시오.

정상적으로 스캔에 포함되는 써드파티 코드의 개발자인 경우 설정을 사용하여 써드파티 코드를 포함시켜야 합니다.

프로시저

보안 코드를 스캔하고 평가서나 보고서를 열려면 다음을 수행하십시오.

  1. 플러그인이 IDE에 설치되었는지 확인하십시오. 설치 중에 IDE가 열려 있었다면 이를 다시 시작하십시오.
  2. 스캔할 항목을 선택하십시오.
    • Eclipse에서 스캔할 프로젝트를 선택하십시오(복수 선택 가능). 전체 Eclipse 작업공간을 스캔하려면 모든 프로젝트를 선택하십시오.
    • IntelliJ IDEA에서 스캔할 프로젝트 모듈을 선택하십시오.
    • Visual Studio에서 스캔할 솔루션, 프로젝트 또는 웹 사이트를 선택하십시오.
  3. 선택사항을 마우스 오른쪽 단추로 클릭하고 보안 분석 > 정적 분석 실행을 선택하십시오.
    참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.
  4. 스캔이 준비된 후 아직 서비스에 로그인하지 않은 경우 로그인 대화 상자가 열립니다. 열리는 로그인 대화 상자는 install-plugins.bat(Windows) 또는 install-plugins.sh(Linux 및 macOS) 설치 프로그램을 실행하기 전에 구축된 연결 설정에 의해 결정됩니다. 대화 상자에서 서비스 신임 정보를 입력하십시오.
    • Cloud Marketplace에 연결된 경우: Application Security on Cloud 서비스에서 API 키 생성 시 키 ID 및 키 시크릿을 수신합니다. ID 및 시크릿 필드에 해당 값을 입력하십시오. API 키가 아직 생성되지 않은 경우 작성 관련 대화 상자의 링크를 따르십시오.
    • Bluemix에 연결된 경우: 서비스 신임 정보에 표시된 바인딩 ID와 비밀번호를 입력하십시오.

      Bluemix 서비스 신임 정보를 판별하려면 서비스 대시보드의 왼쪽 탐색 분할창에서 서비스 신임 정보를 선택하십시오. 외부 앱이 Bluemix 서비스를 사용하도록 설정을 참조하십시오.

    서비스에 로그인하면 암호화된 키 파일이 작성됩니다. 그리고 이 토큰 파일은 ASoC 서비스와 상호작용할 때 기타 조치에 의해 참조됩니다.

    Bluemix에 로그인한 경우 이 토큰 파일은 작성한 후 2시간이 지나면 만료됩니다. 이 경우 이 파일이 만료된 이후에 추가 조치를 계속하려면 서비스에 다시 로그인해야 합니다. 또는 로그인할 때 신임 정보 저장 선택란을 선택할 수 있습니다. 이 선택란을 선택하는 경우, Client Utility에서는 토큰이 만료될 때 자동으로 서비스에 대한 재인증을 시도합니다.

  5. IBM Cloud Marketplace 전용: IBM Cloud Marketplace에서 Application Security on Cloud 서비스에 연결하는 경우 IDE의 정적 분석 스캔이 기존 Application Security on Cloud 애플리케이션과 연관되어야 합니다. 스캔이 실행된 후에 연관시킬 애플리케이션을 선택하도록 대화 상자에 프롬프트가 표시됩니다.

    스캔을 연관시킬 애플리케이션을 선택한 후에 연관 정보가 다음과 같이 저장됩니다.

    • Eclipse에서 애플리케이션이 전체 작업공간과 연관됩니다. 연관에 대한 정보는 <workspace>/.metadata/.plugins/com.ibm.appscan.eclipse.plugin/.scans/.app에 저장됩니다. 여기서 <workspace>는 Eclipse 작업공간의 위치입니다.
    • IntelliJ IDEA에서 애플리케이션은 개별 프로젝트 모듈과 연관됩니다. 연관에 대한 정보는 <project>/.scans/.app에 저장됩니다. 여기서 <project>는 프로젝트 모듈의 위치입니다.
    • Visual Studio에서 애플리케이션은 솔루션과 연관됩니다. 연관에 대한 정보는 <solution>/.scans/.app에 저장됩니다. 여기서 <solution>은 .sln 파일이 포함된 디렉토리입니다.

    .app 파일이 해당 위치에서 제거되면 다음에 스캔할 때 연관된 애플리케이션을 선택하도록 프롬프트가 표시됩니다. 애플리케이션 연관을 선택하고 다른 애플리케이션으로 변경하려는 경우 .app 파일을 삭제한 후 다시 스캔하십시오.

    참고: IBM Bluemix에서 Application Security on Cloud 서비스에 연결하는 경우 애플리케이션 연관이 적용되지 않습니다.
  6. 스캔이 제출된 후 정적 분석 스캔 보기가 열립니다.
  7. 스캔이 완료되면 결과 또는 보고서를 열 수 있는 링크가 포함된 알림이 열립니다. 그리고 스캔을 포함하도록 정적 분석 스캔 보기가 업데이트됩니다. 보기에는 스캔 이름, 상태, 시작 시간 및 종료 시간, 발견된 취약점 수와 심각도가 나열됩니다.
    1. 결과: 알림의 링크를 선택하여 결과를 열거나 보기의 결과 열에서 아이콘을 두 번 클릭합니다(Visual Studio에서는 시스템 트레이를 사용하여 결과를 열 수도 있습니다). 그러면 수정 그룹에서 찾은 결과를 나열하는 대화식 평가서가 열립니다. 수정 그룹은 그룹화된 찾은 결과가 플로우되는 가장 공통적인 노드를 나타냅니다. 일반적으로 수정 그룹에 대한 수정이 구현되는 경우 작업 대비 효과가 가장 높습니다. 또한 수정 그룹은 관련된 찾은 결과를 동시에 검토할 수 있다는 점에서 논리 그룹화 관점으로 고려될 수 있습니다. 수정 그룹이 수정이 반드시 있어야 하는 위치가 아닐 수도 있음을 주의하십시오. 수정의 수정 그룹 위치를 사용하여 이후의 리팩토링, 코드 예시 및 기타 요인이 미리 포함될 수 있습니다.

      평가서에서:

      • 각 수정 그룹은 권장되는 수정 위치, 수정 위치의 소스가 열리는 링크, 취약성, 소스 코드가 정정되면 수정되는 취약성 발생 수를 표시합니다.
      • 취약성을 선택하는 경우 해당 설명이 열립니다(일반적으로 예제와 권장사항이 포함됨).
      • 세부사항 단추를 선택하면 소스 코드가 정정된 경우 수정되는 모든 찾은 결과에 대해 수정 그룹이 열립니다. 세부사항 보기에서 소스 위치를 선택하면 소스 편집기에서 열립니다. 추적 아이콘을 선택하면 애플리케이션을 통한 데이터 플로우가 표시되는 추적 페이지가 열립니다.

    2. 보고서: 알림의 링크를 선택하여 보고서를 열거나 보기의 보고서 열에서 아이콘을 두 번 클릭합니다(Visual Studio에서는 시스템 트레이를 사용하여 보고서를 열 수도 있습니다). 그러면 찾은 결과의 보고서가 열립니다. 또한 이 보고서는 연결하는 Cloud 서비스에도 있습니다.

정적 분석 IRX 파일 생성 시 오류 또는 경고가 발생함

증상

IRX 파일을 생성할 때 오류 또는 경고를 수신하거나 IRX 파일 생성에 실패할 수 있습니다. IRX 파일 생성에 실패하면 결과 파일의 이름에 _failed가 추가됩니다(예: my_irx_file_failed.irx). 이와 같은 상황이 발생하는 경우 IRX 파일 생성 문제점 해결을 위한 다양한 옵션이 있습니다.

문제점 해결

  • 구성 파일을 사용하여 IRX 파일 생성 문제점 해결

    구성 파일을 사용하지 않고 IRX 파일을 생성했지만 파일 생성이 완료되지 않은 경우(또는 오류와 함께 완료된 경우) Static Analyzer Client Utility는 후속 스캔을 위해 임시 구성 파일을 생성합니다. 이 파일은 구성 파일 템플리트와 유사하게 보이며 그 안에는 대상에 대해 알려진 정보와 함께 Targets 요소가 포함됩니다. 그러나 이 파일에는 Client Utility에서 전체 IRX 파일을 생성할 수 있도록 사용자가 정보를 추가하여 수정할 수 있는 <TargetSettings> 요소도 포함되어 있습니다. 이 파일은 appscan-config.xml이라는 이름이 지정되고 현재 디렉토리에 저장됩니다. 즉, 이 파일을 다음 스캔에 자동으로 사용합니다.

    예를 들어, 찾을 수 없는 외부 종속성이 포함된 대상에 대한 IRX 파일 생성이 제공된 경우 다음과 같은 메시지가 수신될 수 있습니다.

    C:\myTargetDirectory>appscan prepare
An IRX file was created, but it may be incomplete.
C:\myTargetDirectory\myTarget.jar requires the following packages:
      org.apache.jasper.*
An appscan-config.xml file was created in: C:\myTargetDirectory\appscan-config.xml
Please update the file and rerun the command.

    생성되는 appscan-config.xml 파일은 다음과 유사하게 보입니다.

    <Configuration>
  <Targets>
    <Target outputs-only="true" path="C:\myTargetDirectory"/>
  </Targets>
  <Settings/>
  <TargetSettings>
    <Target path="C:\myTargetDirectory\myTarget.jar"/>
      <!--Please add the path of the following dependencies to the attribute "additional_classpath" below, separated by semi-colon.
			  org.apache.jasper.*-->
      <CustomBuildInfo additional_classpath=""/>
    </Target>
  </TargetSettings>
</Configuration>

    문제점을 해결하려면 클래스 경로에서 org.apache.jasper.* 클래스를 포함하도록 <CustomBuildInfo additional_classpath=""/>를 업데이트하고 appscan-config.xml 파일을 저장한 후에 appscan prepare (Windows) 또는 appscan.sh prepare (Linux 및 macOS) 명령을 다시 실행하십시오. JAR 파일에 대한 경로를 지정하는 경우, 표준 Java 클래스 경로 구문이 지원됩니다(예를 들면, 와일드카드가 지원됨).

    구성 파일을 사용해도 유사한 문제점이 발생하는 경우 문제점을 나타내는 메시지가 표시됩니다. 이 메시지를 사용하여 기존 구성 파일을 업데이트한 후 명령을 다시 실행하십시오.

    구성 파일은 CLI로 IRX 파일 생성 구성에 설명되어 있습니다.

  • 디버그 모드에서 IRX 파일 생성

    CLI에서 IRX 파일을 생성 중인 경우 디버그 옵션을 사용하여 appscan prepare (Windows) 또는 appscan.sh prepare (Linux 및 macOS) 명령을 실행할 수 있습니다. 그러면 더 많은 로그 파일이 생성됩니다.

  • IRX 파일 생성과 연관된 로그 파일

    오류 또는 경고를 확인하려면 함께 생성된 로그 파일을 참조하십시오. 로그 파일은 .zip 아카이브 파일에 저장됩니다. 기본적으로 이 파일은 IRX가 저장되는 위치와 동일한 위치에 저장됩니다. 그렇지 않으면 appscan prepare (Windows) 또는 appscan.sh prepare (Linux 및 macOS) 명령의 -l 옵션을 사용하여 다른 위치를 지정할 수 있습니다. .zip 파일의 이름은 생성된 IRX 파일의 이름을 기반으로 합니다.

문제점 해결

  10. 프롬프트되면 Eclipse를 다시 시작하십시오.

보안 스캔이 ID가 x인 애플리케이션이 존재하지 않음 오류로 실패함

증상

Jenkins나 IDE 플러그인 또는 명령 인터페이스(CLI)를 사용하는 스캔 실행 중 다음 오류로 실패합니다.

ID가 x인 애플리케이션이 존재하지 않음
보안 스캔 실행 실패: 분석 서비스에 IRX 파일을 제출하는 중에 문제점이 발생했습니다.

원인

이전에는 Application Security on Cloud가 app_id에 기반하여 애플리케이션을 연관시켰습니다. 최신 업데이트는 이 연관을 app_id에서 guid로 변경했습니다. 대개 ASoC 서비스는 이 업데이트에 기반하여 애플리케이션을 다시 연관시키도록 프롬프트를 표시합니다. Jenkins 및 IDE 플러그인과 CLI의 경우 추가 단계를 수행해야 합니다.

문제점 해결

Jenkins 플러그인의 경우 Jenkins 자동 서버에 보안 분석 추가의 지시사항에 따라 Jenkins 플러그인 구성을 다시 저장한 후 스캔을 다시 시도하십시오.

IDE 플러그인의 경우, 프롬프트가 표시되면 동일한 애플리케이션을 다시 선택하여 스캔한 후 확인을 클릭하십시오.

CLI의 경우, 새 guid가 app_id를 대체합니다. guid는 32자의 고유 ID입니다. 예를 들어, 33c3281a-6d7c-e711-b334-002590731623입니다. appscan list_apps 명령을 사용하여 각 앱에 대한 guid를 판별하고 명령행 또는 스크립트의 queue_analysis 명령에서 app_id를 이 guid로 대체하십시오.

지원되는 Java 소스 코드 어노테이션

정적 분석을 사용하여 Java를 스캔하는 경우 @ValidatorMethod, @CallbackMethod 및 @SuppressSecurityTrace 메소드 레벨 어노테이션이 지원됩니다.

  • 어노테이션 사용
  • @ValidatorMethod
  • @SuppressSecurityTrace
  • @CallbackMethod

어노테이션 사용

다음 단계를 따라 어노테이션을 사용할 수 있습니다.

  1. 기본적으로 어노테이션 지원이 사용으로 설정됩니다. 어노테이션 .jar 파일은 추출된 SAClientUtil_<version>_<os>.zip 파일의 \annotation 디렉토리에 있습니다.
  2. 사전 컴파일된 클래스 .war 파일 또는 .jar 파일을 스캔하는 경우 com.ibm.security.annotation.jar를 어노테이션된 소스가 있는 Java 프로젝트에 추가하고 프로젝트를 다시 빌드하십시오.
  3. 정적 분석 플러그인이 설치된 IDE(Integrated Development Environment)에서 스캔하는 경우 com.ibm.security.annotation.jar를 Java 클래스 경로에 추가하십시오.
  4. 정적 분석 CLI를 사용하는 경우 구성 파일을 작성하고 구성 파일에서 dependency_path에 com.ibm.security.annotation.jar 파일의 위치가 포함되도록 CustomBuildInfo additional_classpath="dependency_path"를 설정하십시오.

어노테이션은 스캔하기 전에 소스 코드에 추가될 수 있습니다. 또는 스캔한 후와 허위 양성(false-positive)을 식별하고 제거하기 위해 분류하는 중에 추가될 수도 있습니다. 정적 분석 플러그인이 설치된 IDE에서 스캔하는 경우 스캔 보고서에서 수정하는 데 도움 주기 단추를 사용할 수 있습니다. 이 단추를 선택하면 어노테이션이 추가될 수 있는 지점이 표시됩니다.

어노테이션을 사용하면 사용자의 지식을 소스 코드에 보안 어노테이션 양식으로 직접 삽입할 수 있습니다. 어노테이션은 코드 안전 부분을 선언하는 데 사용되므로 매우 주의하여 사용해야 합니다. 보안 취약점을 스캔해야 하는 코드에는 어노테이션을 사용하면 안됩니다. 어노테이션을 사용하는 경우 보안 분석가는 추출된 SAClientUtil_<version>_<os>.zip 파일의 \config 디렉토리에 있는 irgen-engine.apsettings 파일에서 기능을 사용 안함으로 설정하여 어노테이션을 무시하도록 선택할 수 있습니다. 이 파일에서 다음 설정을 찾아보십시오. 

<Setting
  name="process_security_annotations"
  value="true"
  default_value="true"
  description="When turned on, method annotations will be processed - used by DotNet and Java LSPs"
  display_name="Process Security Annotations"
  type="bool"
/>

기능을 사용 안함으로 설정하려면 value="true"를 value="false"로 변경하십시오.

참고: 설정이 value="false"로 설정된 경우, 어노테이션을 사용하려면 이를 value="true"로 설정하십시오.

@ValidatorMethod

유효성 검증기는 입력 데이터에서 검사를 수행하는 메소드이며, 보통 입력이 유효한지 여부를 표시하는 부울 값을 리턴합니다. 유효성 검증기를 사용하여 입력을 승인하거나 거부하기 보다 사용자 입력을 승인 가능한 형식으로 변경할 수 있습니다. 이러한 메소드를 무결 처리기라고 합니다.

@ValidatorMethod 어노테이션을 사용하여 애플리케이션 소스 코드에서 모든 유효성 검증기와 무결 처리기 메소드를 식별할 수 있습니다. 정적 분석 스캔 중에 데이터가 안전하다고 판단되면 이러한 메소드를 통해 전달되는 데이터 플로우를 제거하는 데 이 정보가 사용됩니다.

참고: 현재 유효성 검증되었다고 인식되어야 하는 어노테이션된 메소드의 매개변수를 지정하는 프로비저닝이 없습니다. 정적 분석 스캔 중에 모든 입력 매개변수가 유효성 검증된다고 가정합니다.

@SuppressSecurityTrace

이 어노테이션으로 표시된 메소드를 통해 플로우되는 추적이 제거됩니다. 이는 특정 추적 그룹이 허위 양성(false-positive)으로 식별되거나 다른 그룹에 비해 덜 중요하거나 관심 그룹이 아닌 경우에 유용합니다. 이 어노테이션을 사용하여 이러한 추적을 필터링하거나 채우기를 줄일 목적으로 해당 추적을 숨길 수 있습니다.

@CallbackMethod

이 어노테이션을 사용하여 애플리케이션에 대한 콜백이나 시작점을 식별합니다. 모든 인수는 감염을 전달할 것으로 생각됩니다.

지원되는 .NET 소스 코드 속성

정적 분석을 사용하여 .NET를 스캔하는 경우, [ValidatorMethod], [CallbackMethod] 및 [SuppressSecurityTrace] 메소드 레벨 속성이 지원됩니다. 이 속성이 사용되면 [ValidatorMethod()], [CallbackMethod()] 및 [SuppressSecurityTrace()]도 허용됩니다.

  • 속성 사용
  • [ValidatorMethod] 또는 [ValidatorMethod()]
  • [SuppressSecurityTrace] 또는 [SuppressSecurityTrace()]
  • [CallbackMethod] 또는 [CallbackMethod()]

속성 사용

다음 단계에서 속성을 사용할 수 있습니다.

  1. 기본적으로 속성에 대한 지원은 사용 가능으로 설정됩니다. SecurityAttributes.dll 속성 파일은 추출된 SAClientUtil_<version>_<os>.zip 파일의 \annotation 디렉토리에 있습니다.
  2. 사전 컴파일된 클래스 .exe 파일 또는 .dll 파일을 스캔하는 경우 어노테이션이 있는 소스가 포함된 .NET 프로젝트에 SecurityAttributes.dll을 추가한 다음 프로젝트를 다시 빌드하십시오.
  3. 정적 분석 플러그인이 설치된 IDE(Integrated Development Environment)에서 스캔하는 경우 SecurityAttributes.dll을 참조로 추가하십시오.
  4. 정적 분석 CLI를 사용하는 경우 구성 파일을 작성하고 구성 파일에서 dependency_path에 SecurityAttributes.dll 파일의 위치가 포함되도록 CustomBuildInfo additional_classpath="dependency_path"를 설정하십시오.

스캔하기 전에 소스 코드에 속성을 추가할 수 있습니다. 또는 스캔한 후와 허위 양성(false-positive)을 식별하고 제거하기 위해 선별하는 중에 추가할 수도 있습니다. 정적 분석 플러그인이 설치된 IDE에서 스캔하는 경우 스캔 보고서에서 수정하는 데 도움 주기 단추를 사용할 수 있습니다. 이 단추를 선택하면 속성을 추가할 수 있는 지점이 표시됩니다.

속성을 사용하면 사용자의 지식을 소스 코드에 보안 속성 양식으로 직접 삽입할 수 있습니다. 속성은 코드 안전 부분을 선언하는 데 사용되므로 매우 주의하여 사용해야 합니다. 보안 취약점을 스캔해야 하는 코드에는 어노테이션을 사용하면 안됩니다. 속성을 사용하는 경우 보안 분석가는 추출된 SAClientUtil_<version>_<os>.zip 파일의 \config 디렉토리에 있는 irgen-engine.apsettings 파일에서 기능을 사용 안함으로 설정하여 속성을 무시하도록 선택할 수 있습니다. 이 파일에서 다음 설정을 찾아보십시오. 

<Setting
  name="process_security_annotations"
  value="true"
  default_value="true"
  description="When turned on, method annotations will be processed - used by DotNet and Java LSPs"
  display_name="Process Security Annotations"
  type="bool"
/>

기능을 사용 안함으로 설정하려면 value="true"를 value="false"로 변경하십시오.

참고: 설정이 value="false"로 설정된 경우, 어노테이션을 사용하려면 이를 value="true"로 설정하십시오.

해결되지 않은 PHP include 표현식 및 알 수 없는 PHP 클래스 구성

PHP에 대한 IRX 파일 생성 이후, 그 다음에 파일을 스캔할 때 IRX 생성을 구성하도록 허용하는 구성 파일이 작성됩니다.

구성 파일 php-scan-config.xml이 prepare 명령을 실행한 위치에 생성됩니다. 파일에는 두 개의 섹션인 <UserInputs> 및 <Php-Unknown-Configuration>이 포함됩니다.

<UserInputs> 섹션에는 PHP 스캔에 대해 수정될 수 있는 구성 설정이 포함됩니다. 섹션은 샘플 값으로 사전에 채워집니다. 세미콜론(;)을 통해 이러한 값에 추가할 수 있습니다. 이 섹션의 설정은 다음과 같습니다.

  • <RegularExpression>: 이는 정규식이 검색 텍스트에 대해 지정되도록 허용하는 고급 기능입니다.
  • <ReplaceText>: 이는 단순 텍스트 검색 및 대체에 사용됩니다. 검색 텍스트는 현상태대로 사용되고, 포함된 텍스트의 임의의 위치에서 발견되면 대체 텍스트로 바뀝니다.
  • <Function>: 이 설정은 함수 호출이 대체될 때 사용됩니다. 검색 텍스트는 소괄호 없는 함수의 이름이어야 합니다. 검색 텍스트는 소괄호가 뒤따르는 지정된 이름으로 함수 호출을 검색하도록 기능 보강되고 소괄호 내에서 임의의 항목과 일치합니다.
  • <SimpleText>: 단순화된 대체 텍스트입니다. 대체 텍스트가 단일 문자열일 때 이를 사용하십시오.
  • <SimpleRegEx>: 단순화된 대체 정규식입니다. 대체 정규식이 단순 문자열일 때 이를 사용하십시오.
  • <AdditionalIncludePaths>: 이 설정을 사용하여 참조된 파일이 포함된 추가 include 경로를 추가하십시오.

<Php-Unknown-Configuration> 섹션에는 마지막 스캔에서 발견된 모든 해결되지 않은 include 표현식 및 알 수 없는 클래스가 포함됩니다. 이 정보를 사용하여 <UserInputs> 섹션을 완료하십시오.

  • <Summary>: 마지막 섹션의 알 수 없는 include 표현식(<NumberOfUnknownIncludeExprs>) 및 알 수 없는 클래스 (<NumberOfUnknownClasses>)의 수를 요약합니다.
  • <SourceFile>: 모든 알 수 없는 include 표현식 및 알 수 없는 클래스가 소스 파일 이름에 따라 알파벳순으로 나열됩니다. 각 소스 파일마다 파일 경로(<FilePath>)가 소스의 절대 경로, 알 수 없는 클래스 (<UnknownClass>) 및 알 수 없는 include 표현식 (<UnknownIncludes>)을 나열합니다.

알 수 없는 클래스(<UnknownClass>) 정보

태그 설명
<ClassName> 클래스가 사용되는 소스 파일(<FilePath>)의 위치와 함께 소스 코드에서 참조된 클래스 이름입니다. 알 수 없는 클래스를 해결하려면 편집기에서 위치를 봐야할 수 있습니다.
<Status> 클래스가 해결됨 또는 해결되지 않음인지 표시합니다.
<ResolvedBy> 알 수 없는 클래스가 해결된 방식을 표시합니다. 값은 다음과 같습니다.
  • AutoResolver: 파일을 찾기 위한 애플리케이션 사용 내부 체험입니다.
  • SearchReplace: 업데이트된 텍스트를 생성하는 데 포함된 텍스트에 하나 이상의 검색 및 대체 규칙이 적용되었습니다.

알 수 없는 include 표현식(<UnknownIncludes>) 정보

태그 설명
<IncludeText> 표현식이 사용된 소스 파일(<FilePath>)의 위치와 함께 소스 코드에 있는 표현식 텍스트입니다. 알 수 없는 include표현식을 해결하려면 편집기에서 위치를 봐야할 수 있습니다.
<UpdatedText> 마지막 스캔 중 사용된 업데이트된 텍스트입니다.
<Status> 표현식이 해결됨 또는 해결되지 않음인지 표시합니다.
<ResolvedBy> 업데이트된 텍스트가 생성된 방식을 표시합니다. 값은 다음과 같습니다.
  • AutoResolver: 파일을 찾기 위한 애플리케이션 사용 내부 체험입니다.
  • SearchReplace: 업데이트된 텍스트를 생성하는 데 포함된 텍스트에 하나 이상의 검색 및 대체 규칙이 적용되었습니다.
  • SearchReplace+AutoResolver: 업데이트된 텍스트를 생성하는 데 포함된 텍스트에 하나 이상의 검색 및 대체 규칙이 적용된 후, 파일을 찾는 데 내부 체험이 적용되었습니다.
  • SearchReplace+IncludePath: 업데이트된 텍스트를 생성하는 데 포함된 텍스트에 하나 이상의 검색 및 대체 규칙이 적용된 후, 파일을 찾는 데 include 경로의 디렉토리와 결합되었습니다.
  • SearchReplace+RelativeDir: 업데이트된 텍스트를 생성하는 데 포함된 텍스트에 하나 이상의 검색 및 대체 규칙이 적용된 후, include 표현식이 포함되는 파일의 소스 디렉토리에 상대적으로 발견되었습니다.

써드파티 Java 및 .NET 제외 관리

기본적으로 써드파티 Java 및 .NET 코드는 IRX 파일 생성 중에 스캔되지 않습니다. 이 주제의 지시사항에 따라 제외되는 써드파티 코드를 관리할 수 있습니다.

이 태스크 정보

사용자 코드에 대한 IRX 파일을 생성할 때 써드파티 코드가 제외되므로 사용자의 평가에는 사용자가 분석하지 않는 코드에 대한 정보와 찾은 결과가 포함되지 않습니다. 제외되는 내용을 수정하려면 이 주제의 단계에 따르십시오. 스캐닝 중에 써드파티 코드를 포함시키려면 다음 주제를 참조하십시오.

  • 명령행 인터페이스(CLI)를 사용하여 IRX 파일 생성
  • Maven 프로젝트를 위한 IRX 파일 생성
  • 통합 개발 환경에서 스캔

프로시저

  1. 추출된 SAClientUtil_<version>_<os>.zip 파일의 config 디렉토리를 찾으십시오(여기서 <version>은 Client Utility의 현재 버전입니다).
  2. 이 디렉토리에서 dot_net.exclusions(.NET) 또는 java.exclusions(Java) 파일을 찾으십시오.
    참고: 이 디렉토리의 다른 .exclusions 파일은 실험용으로, 이들을 수정해도 써드파티 제외에는 영향이 없습니다.
  3. 텍스트 편집기에서 파일을 열면 다음 형식의 XML을 확인할 수 있습니다. 
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<Exclusions>
  <packages>
    <package name="package_name"/>
  </packages>
  ...
</Exclusions>
  4. 제외시킬 패키지를 추가하려면 파일을 편집하십시오.
    참고:
    • 지정된 패키지 이름은 동일한 접두부의 모든 패키지와 일치합니다(와일드카드가 후속하는 것처럼 취급됩니다). 예를 들어, <package name="com.mycompany.common"/>을 제외로 추가하는 경우, <package name="com.mycompany.common.action1"/> 및 <package name="com.mycompany.common.action2"/>가 제외됩니다.
    • <packages> 컨테이너에 추가된 항목만 써드파티 제외에 영향을 줍니다. 클래스와 메소드를 추가해도 제외에는 영향을 주지 않습니다.
  5. 파일을 저장한 다음 사용자의 소스 코드에 대한 IRX 파일을 생성하십시오.

다음에 수행할 작업

제외 파일을 수정하는 경우 다음 우수 사례를 따라야 합니다.
  • 수정된 제외 파일의 백업을 보관하십시오. 특히, Static Analyzer Client Utility의 새 버전으로 업그레이드하는 경우 업데이트 없이 제외 파일의 버전이 다운로드됩니다. Client Utility의 업데이트 시, 수정된 제외 파일의 사본을 저장한 다음 이 파일을 새 Client Utility에 복사하십시오.
  • 필요 시 복사하여 붙여넣기 쉽도록 수정사항을 단일의 XML 항목 블록으로 보관하십시오.

CLI를 사용하여 사용 가능한 명령 목록을 검색하려면 appscan help 명령을 실행하십시오.

CLI를 사용하여 개별 명령에 대한 정보를 검색하려면 appscan help <command>를 실행하십시오. 여기서, <command>는 도움말을 표시하려는 명령입니다.

분석 명령 (Windows)

분석 명령은 스캔 요청을 클라우드에 제출하거나 클라우드에 이미 제출된 스캔 요청에 대한 작업을 수행하기 위해 사용합니다. 분석 명령을 사용하여 스캔에 대한 정보를 수신할 수도 있습니다. 이 정보는 자동화 스크립트에 유용하게 사용됩니다.

이러한 명령을 실행하기 전에 분석 서비스에 로그인했는지 확인하십시오(인증 명령 참조).

  • appscan status
  • appscan list
  • appscan list_apps
  • appscan cancel
  • appscan queue_analysis
  • appscan info

appscan status

구문:

appscan status -i <job_id>

설명:

분석 작업에 대해 다음 상태 코드 중 하나를 리턴합니다. 

0 = Pending
1 = Starting
2 = Running
3 = FinishedRunning
4 = FinishedRunningWithErrors
5 = PendingSupport
6 = Ready
7 = ReadyIncomplete
8 = FailedToScan
9 = ManuallyStopped
10 = None
11 = Initiating
12 = MissingConfiguration
13 = PossibleMissingConfiguration

필수 옵션:

  • -i: -i <job_id>를 지정합니다. 여기서, <job_id>는 분석 작업의 ID입니다.
팁: 모든 명령에서 옵션은 임의의 순서로 사용할 수 있습니다.

예제:

작업 ID 12345의 상태를 보려면 다음을 지정하십시오. 

appscan status -i 12345

리턴 코드가 0이면 작업이 보류 중 상태입니다. 리턴 코드가 1이면 작업이 시작 중입니다.

appscan list

설명:

appscan list_apps

구문:

appscan list_apps

설명:

IBM Cloud Marketplace에만 해당: IBM Cloud Marketplace에서 Application Security on Cloud 서비스에 연결된 경우, 클라우드에 제출하는 IRX 파일은 기존 Application Security on Cloud 애플리케이션과 연관되어야 합니다.

이 명령을 사용하면 액세스 권한이 있는 Application Security on Cloud 애플리케이션을 볼 수 있습니다. 명령을 사용하려면 IBM Cloud Marketplace에서 서비스에 대해 인증을 받아야 합니다. 명령을 실행한 후에 Application Security on Cloud 애플리케이션의 목록은 애플리케이션 이름 및 해당 ID(소괄호로 묶임)와 함께 표시됩니다. queue_analysis 명령의 -a 옵션을 사용할 때는 이 목록의 ID 값을 사용하십시오.

참고: Bluemix에서 ASoC 서비스에 연결하는 경우 애플리케이션 연관이 적용되지 않습니다.

appscan queue_analysis

구문:

appscan queue_analysis -a <app_id> -f <irx_file> -n <scan_name>

설명:

분석을 위해 IRX 파일을 클라우드에 제출합니다.스캔이 완료되면 이메일 알림을 받게 됩니다(분석 서비스에 로그인할 때 사용된 계정과 연관된 이메일 주소로). 이 이메일에는 스캔을 다운로드하기 위해 로그인할 수 있도록 클라우드에 대한 링크가 포함되어 있습니다.

참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.

필수 옵션:

  • -f: -f <irx_file>을 지정합니다. 여기서, <irx_file>은 스캔을 위해 제출하려는 IRX 파일입니다. IRX 파일이 현재 디렉토리에 없는 경우 이 옵션을 사용하여 IRX 파일 경로 및 파일 이름을 지정하십시오.
    참고: 이 옵션은 이러한 명령문 중 하나 또는 둘 다가 true인 경우에만 필요합니다.
    • 두 개 이상의 IRX 파일이 있는 디렉토리에서 명령을 실행합니다. 디렉토리에 하나의 IRX 파일만 있으면 -f 옵션이 사용되지 않는 경우 해당 파일이 제출됩니다.
    • IRX 파일이 없는 디렉토리에서 명령을 실행합니다. 이 경우에는 -f 옵션을 사용하여 제출할 IRX 파일의 경로 및 이름을 지정해야 합니다.
  • -n: -n <scan_name>을 지정하십시오. 여기서 <scan_name>은 클라우드에서 수행할 스캔의 이름입니다.
  • -a (IBM Cloud Marketplace에만 해당): IBM Cloud Marketplace에서 Application Security on Cloud 서비스에 연결된 경우, 클라우드에 제출하는 IRX 파일은 기존 Application Security on Cloud 애플리케이션과 연관되어야 합니다. 이 옵션에서 -a <app_id>를 지정하십시오. 여기서 <app_id>는 연관되는 애플리케이션의 ID입니다. ID를 판별하려면 list_apps 명령을 사용하십시오.
    참고: Bluemix에서 ASoC 서비스에 연결하는 경우 애플리케이션 연관이 적용되지 않습니다.
팁: 모든 명령에서 옵션은 임의의 순서로 사용할 수 있습니다.

예제:

Bluemix의 서비스에 연결될 때 이 명령을 사용하십시오. 

appscan queue_analysis -f my_irx.irx -n my_scan

스캔을 위해 현재 디렉토리에 있는 my_irx.irx을 클라우드에 제출하는 데 사용됩니다. 모든 현재 스캔을 확인하기 위해 클라우드 사용자 인터페이스 또는 list 명령을 사용할 경우, my_scan이 목록에 표시됩니다.

IBM Cloud Marketplace의 서비스에 연결될 때 위의 명령을 실행하되, -a 12345를 포함하십시오. 여기서 12345는 스캔과 연관되는 애플리케이션의 ID입니다(애플리케이션 ID는 list_apps 명령을 사용하여 판별될 수 있음).

appscan prepare

구문:

appscan prepare -c <configuration_file> -d <save_path> -l <log_path> -n <file_name> -t,--thirdParty -v,--verbose -X,--debug

설명:

IRX 파일을 생성합니다.

참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.

선택적 옵션:

  • -c: 이 옵션은 스캔을 구성하는 데 사용됩니다.
  • -d: -d <save_path>를 지정합니다. 여기서, <save_path>는 IRX 파일을 저장하려는 디렉토리입니다.
  • -l: -l <log_path>를 지정합니다. 여기서, <log_path>는 로그 파일을 저장하려는 디렉토리입니다.
  • -n: -n <file_name>을 지정합니다. 여기서, <file_name>은 IRX 파일 이름으로 사용됩니다. 파일 이름에는 .irx 파일 확장자를 지정하거나 지정하지 않을 수 있습니다. 이를 확장자 없이 지정할 경우, 파일이 생성될 때 자동으로 추가됩니다.
  • -t,--thirdParty: 기본적으로, 써드파티 Java 및 .NET 코드는 IRX 파일 생성 중에 스캔되지 않습니다. 써드파티 Java 및 .NET 제외 관리의 지시사항을 따라 써드파티 코드 제외 설정을 수정할 수 있습니다. 써드파티 코드를 포함하려면 prepare 명령을 실행할 때 -t 또는 --thirdParty 옵션을 지정하십시오.

    정상적으로 스캔에 포함되는 써드파티 코드의 개발자인 경우 설정을 사용하여 써드파티 코드를 포함시켜야 합니다.

  • -v,--verbose: IRX 파일 생성 중에 더 많은 정보 출력이 표시되도록 하려면 -v 또는 --verbose를 지정하십시오.
  • -X,--debug: -X 또는 --debug를 지정하면 전체 명령이 디버그 모드로 실행됩니다. 디버그 모드로 실행할 때 문제점 해결을 위해 추가 로그 파일이 생성됩니다.
팁: 모든 명령에서 옵션은 임의의 순서로 사용할 수 있습니다.

예제:

c:\my_config_files\my_config.xml 구성 파일을 사용하여 IRX 파일을 생성하고 IRX 파일을 c:\my_irx_files\my_scan.irx에 저장하려면 다음 명령을 실행하십시오. 

appscan prepare -c c:\my_config_files\my_config.xml -d c:\my_irx_files -n my_scan.irx

appscan package

구문:

appscan package -d <save_path> -f <assessment_file> -n <file_name>

설명:

IBM Security AppScan® Source 버전 9.0 이상 제품에서 작성된 평가(.ozasmt) 파일이 포함된 IRX 파일을 생성하십시오.

참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.

선택적 옵션:

  • -d: -d <save_path>를 지정합니다. 여기서, <save_path>는 IRX 파일을 저장하려는 디렉토리입니다.
  • -f: -f <assessment_file>을 지정합니다. 여기서, <assessment_file>은 스캔을 위해 패키지로 만드는 .ozasmt 파일입니다. <assessment_file> 파일이 현재 디렉토리에 없으면 이 옵션을 사용하여 평가 파일 경로 및 파일 이름을 지정하십시오.
    참고: 이 옵션은 이러한 명령문 중 하나 또는 둘 다가 true인 경우에만 필요합니다.
    • 두 개 이상의 평가 파일이 있는 디렉토리에서 명령을 실행합니다. 디렉토리에 평가 파일이 하나만 포함된 경우, -f 옵션이 사용되지 않으면 해당 파일이 패키지됩니다.
    • 평가 파일이 없는 디렉토리에서 명령을 실행합니다. 이 경우에는 -f 옵션을 사용하여 패키지로 만들 평가 파일의 경로와 이름을 지정해야 합니다.
  • -n: -n <file_name>을 지정합니다. 여기서, <file_name>은 IRX 파일 이름으로 사용됩니다. 파일 이름에는 .irx 파일 확장자를 지정하거나 지정하지 않을 수 있습니다. 이를 확장자 없이 지정할 경우, 파일이 생성될 때 자동으로 추가됩니다.
팁: 모든 명령에서 옵션은 임의의 순서로 사용할 수 있습니다.

예제:

c:\my_irx_files 디렉토리에서 다음 명령을 실행하여 c:\my_assessment_files\my_assessment.ozasmt 평가 파일이 들어 있는 IRX 파일을 생성할 수 있습니다. 

appscan package -f c:\my_assessment_files\my_assessment.ozasmt

결과 IRX 파일이 c:\my_irx_files에 저장됩니다.

appscan get_pubkey

구문:

appscan get_pubkey -d <save_path>

설명:

인터넷에 연결되지 않은 컴퓨터에 사용할 공용 암호화 키를 다운로드하십시오.

인터넷에 연결된 컴퓨터에서 IRX 파일을 생성하는 경우 prepare 명령을 실행할 때 암호화 키가 자동으로 다운로드되므로 이 명령이 필요하지 않습니다. 암호화 키가 컴퓨터에 이미 존재하는 경우, prepare 명령을 실행할 때 필요에 따라 업데이트됩니다.

하지만 인터넷에 연결되지 않은 컴퓨터에서 IRX 파일을 생성하는 경우 이 명령을 사용하여 암호화 키를 다운로드할 수 있습니다. 그런 다음 IRX 파일을 생성할 때 사용할 수 있도록 인터넷에 연결되지 않은 컴퓨터에 암호화 키를 복사할 수 있습니다. 해당 컴퓨터에서 암호화 키를 사용하려면 rsa.pub 파일 이름을 유지하고 해당 파일을 추출된 SAClientUtil_<version>_<os>.zip 파일의 config 디렉토리에 배치해야 합니다(여기서 <version>은 Client Utility의 현재 버전임).

참고: 사용된 암호화 키는 최신이어야 합니다. IRX 파일이 기한이 지난 암호화 키로 생성된 경우에는 업로드될 때 IRX 파일이 거부됩니다. 컴퓨터가 인터넷에 연결된 경우 prepare 명령을 다시 실행하여 암호화 키를 자동으로 업데이트하십시오. 컴퓨터가 인터넷에 연결되지 않은 경우 get_pubkey 명령을 사용해야 합니다.

선택적 옵션:

선택적 옵션:

  • -d: -d <file_path>를 지정합니다. 여기서, <file_path>는 대상 파일의 완전한 경로 및/또는 대상 파일의 파일 이름입니다. 파일 이름이 지정되지 않으면 파일 이름은 스캔 작업 이름을 기반으로 생성됩니다. 경로가 지정되지 않으면 파일이 현재 디렉토리에 저장됩니다. 이 옵션이 포함되지 않으면 파일이 스캔 작업 이름을 기반으로 생성된 파일 이름으로 현재 디렉토리에 저장됩니다.
  • -t: -t <type>을 지정합니다. 여기서, <type>은 html 또는 zip 중 하나입니다. 결과는 HTML 파일 또는 HTML 결과가 포함된 .zip 파일로 저장됩니다. 이 옵션이 포함되지 않으면 결과가 HTML 파일로 저장됩니다.

appscan.sh queue_analysis

구문:

appscan.sh queue_analysis -a <app_id> -f <irx_file> -n <scan_name>

설명:

분석을 위해 IRX 파일을 클라우드에 제출합니다.스캔이 완료되면 이메일 알림을 받게 됩니다(분석 서비스에 로그인할 때 사용된 계정과 연관된 이메일 주소로). 이 이메일에는 스캔을 다운로드하기 위해 로그인할 수 있도록 클라우드에 대한 링크가 포함되어 있습니다.

참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.

필수 옵션:

  • -f: -f <irx_file>을 지정합니다. 여기서, <irx_file>은 스캔을 위해 제출하려는 IRX 파일입니다. IRX 파일이 현재 디렉토리에 없는 경우 이 옵션을 사용하여 IRX 파일 경로 및 파일 이름을 지정하십시오.
    참고: 이 옵션은 이러한 명령문 중 하나 또는 둘 다가 true인 경우에만 필요합니다.
    • 두 개 이상의 IRX 파일이 있는 디렉토리에서 명령을 실행합니다. 디렉토리에 하나의 IRX 파일만 있으면 -f 옵션이 사용되지 않는 경우 해당 파일이 제출됩니다.
    • IRX 파일이 없는 디렉토리에서 명령을 실행합니다. 이 경우에는 -f 옵션을 사용하여 제출할 IRX 파일의 경로 및 이름을 지정해야 합니다.
  • -n: -n <scan_name>을 지정하십시오. 여기서 <scan_name>은 클라우드에서 수행할 스캔의 이름입니다.
  • -a (IBM Cloud Marketplace에만 해당): IBM Cloud Marketplace에서 Application Security on Cloud 서비스에 연결된 경우, 클라우드에 제출하는 IRX 파일은 기존 Application Security on Cloud 애플리케이션과 연관되어야 합니다. 이 옵션에서 -a <app_id>를 지정하십시오. 여기서 <app_id>는 연관되는 애플리케이션의 ID입니다. ID를 판별하려면 list_apps 명령을 사용하십시오.
    참고: Bluemix에서 ASoC 서비스에 연결하는 경우 애플리케이션 연관이 적용되지 않습니다.
팁: 모든 명령에서 옵션은 임의의 순서로 사용할 수 있습니다.

예제:

Bluemix의 서비스에 연결될 때 이 명령을 사용하십시오.

appscan.sh prepare

구문:

appscan.sh prepare -c <configuration_file> -d <save_path> -l <log_path> -n <file_name> -t,--thirdParty -v,--verbose -X,--debug

설명:

IRX 파일을 생성합니다.

참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.

선택적 옵션:

  • -c: 이 옵션은 스캔을 구성하는 데 사용됩니다.
  • -d: -d <save_path>를 지정합니다. 여기서, <save_path>는 IRX 파일을 저장하려는 디렉토리입니다.
  • -l: -l <log_path>를 지정합니다. 여기서, <log_path>는 로그 파일을 저장하려는 디렉토리입니다.
  • -n: -n <file_name>을 지정합니다. 여기서, <file_name>은 IRX 파일 이름으로 사용됩니다. 파일 이름에는 .irx 파일 확장자를 지정하거나 지정하지 않을 수 있습니다. 이를 확장자 없이 지정할 경우, 파일이 생성될 때 자동으로 추가됩니다.
  • -t,--thirdParty: 기본적으로, 써드파티 Java 및 .NET 코드는 IRX 파일 생성 중에 스캔되지 않습니다. 써드파티 Java 및 .NET 제외 관리의 지시사항을 따라 써드파티 코드 제외 설정을 수정할 수 있습니다. 써드파티 코드를 포함하려면 prepare 명령을 실행할 때 -t 또는 --thirdParty 옵션을 지정하십시오.

    정상적으로 스캔에 포함되는 써드파티 코드의 개발자인 경우 설정을 사용하여 써드파티 코드를 포함시켜야 합니다.

  • -v,--verbose: IRX 파일 생성 중에 더 많은 정보 출력이 표시되도록 하려면 -v 또는 --verbose를 지정하십시오.
  • -X,--debug: -X 또는 --debug를 지정하면 전체 명령이 디버그 모드로 실행됩니다. 디버그 모드로 실행할 때 문제점 해결을 위해 추가 로그 파일이 생성됩니다.
팁: 모든 명령에서 옵션은 임의의 순서로 사용할 수 있습니다.

appscan.sh package -d <save_path> -f <assessment_file> -n <file_name>

설명:

참고: 소스 코드를 스캔하거나 IRX 파일을 생성할 때 최신 Client Utility로 업데이트하는 방법에 대한 메시지를 수신할 수 있습니다. Client Utility 및 클라우드 서비스 버전 호환성의 내용을 참조하십시오.

appscan.sh get_pubkey

구문:

appscan.sh get_pubkey -d <save_path>

설명:

인터넷에 연결되지 않은 컴퓨터에 사용할 공용 암호화 키를 다운로드하십시오.

인터넷에 연결된 컴퓨터에서 IRX 파일을 생성하는 경우 prepare 명령을 실행할 때 암호화 키가 자동으로 다운로드되므로 이 명령이 필요하지 않습니다. 암호화 키가 컴퓨터에 이미 존재하는 경우, prepare 명령을 실행할 때 필요에 따라 업데이트됩니다.

하지만 인터넷에 연결되지 않은 컴퓨터에서 IRX 파일을 생성하는 경우 이 명령을 사용하여 암호화 키를 다운로드할 수 있습니다. 그런 다음 IRX 파일을 생성할 때 사용할 수 있도록 인터넷에 연결되지 않은 컴퓨터에 암호화 키를 복사할 수 있습니다. 해당 컴퓨터에서 암호화 키를 사용하려면 rsa.pub 파일 이름을 유지하고 해당 파일을 추출된 SAClientUtil_<version>_<os>.zip 파일의 config 디렉토리에 배치해야 합니다(여기서 <version>은 Client Utility의 현재 버전임).

참고: 사용된 암호화 키는 최신이어야 합니다. IRX 파일이 기한이 지난 암호화 키로 생성된 경우에는 업로드될 때 IRX 파일이 거부됩니다. 컴퓨터가 인터넷에 연결된 경우 prepare 명령을 다시 실행하여 암호화 키를 자동으로 업데이트하십시오. 컴퓨터가 인터넷에 연결되지 않은 경우 get_pubkey 명령을 사용해야 합니다.

선택적 옵션:

  • -d: -d <save_path>를 지정합니다. 여기서, <save_path>는 암호화 키를 저장하려는 디렉토리입니다. 이 옵션이 지정되지 않으면 추출된 SAClientUtil_<version>_<os>.zip 파일의 config 디렉토리에 키가 저장됩니다.

바인딩 ID 123-mybindingid-456과 비밀번호 123-mypassword-456으로 클라우드에 로그인하려면 다음 명령을 실행하십시오. 

appscan.sh login -P 123-mypassword-456 -u 123-mybindingid-456

결과 명령 (Linux 및 macOS)

결과 명령을 사용하여 분석 서비스에서 스캔 결과를 검색할 수 있습니다.

appscan.sh get_result

구문:

appscan.sh get_result -d <file_path> -i <job_id> -t <type>

설명:

스캔 결과를 검색합니다.

이러한 명령을 사용하기 전에 분석 서비스에 로그인했는지 확인하십시오.

필수 옵션:

  • -i: -i <job_id>를 지정합니다. 여기서, <job_id>는 분석 작업의 ID입니다.

선택적 옵션:

  • -d: -d <file_path>를 지정합니다. 여기서, <file_path>는 대상 파일의 완전한 경로 및/또는 대상 파일의 파일 이름입니다. 파일 이름이 지정되지 않으면 파일 이름은 스캔 작업 이름을 기반으로 생성됩니다. 경로가 지정되지 않으면 파일이 현재 디렉토리에 저장됩니다. 이 옵션이 포함되지 않으면 파일이 스캔 작업 이름을 기반으로 생성된 파일 이름으로 현재 디렉토리에 저장됩니다.
  • -t: -t <type>을 지정합니다. 여기서, <type>은 html 또는 zip 중 하나입니다. 결과는 HTML 파일 또는 HTML 결과가 포함된 .zip 파일로 저장됩니다. 이 옵션이 포함되지 않으면 결과가 HTML 파일로 저장됩니다.

    스캔 결과가 package 명령을 통해 생성된 IRX 파일에 대한 내용인 경우, -t zip을 지정하면 AppScan Source 버전 9.0 이상 제품으로 로드할 수 있는 새로운 .ozasmt 파일이 포함된 결과가 저장됩니다.

팁: 모든 명령에서 옵션은 임의의 순서로 사용할 수 있습니다.
