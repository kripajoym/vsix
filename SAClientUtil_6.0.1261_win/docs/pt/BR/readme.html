Introdução ao Static Analyzer

Use a análise estática para varrer o código-fonte para vulnerabilidades de segurança. Para realizar essa ação, um pequeno utilitário deve ser transferido por download. Se você extrair o utilitário para o disco local, será possível usar sua interface da linha de comandos (CLI) para executar a análise de segurança em todos os idiomas suportados. Ou, é possível executar um instalador integrado que inclua os plug-ins de análise estática para Eclipse, Maven, IntelliJ IDEA ou Visual Studio. No Eclipse e no IntelliJ IDEA, é possível varrer projetos Java e, no Visual Studio, é possível varrer .NET (C#, ASP.NET, VB.NET).

Procedimento

  • Para aprender como usar o serviço Application Security on Cloud em IBM Cloud Marketplace, consulte http://www.ibm.com/support/knowledgecenter/SSYJJF_1.0.0/ApplicationSecurityonCloud/Welcome.html.
  • Para aprender como usar o Application Security on Cloud for Bluemix, consulte https://console.ng.bluemix.net/docs/services/ApplicationSecurityonCloud/index.html.
  • O restante deste arquivo leia-me descreve a instalação e o uso do Static Analyzer Client Utility.

Configurando o Static Analyzer Client Utility

Para usar a análise estática, faça o download de um pequeno Client Utility. Se você extrair o utilitário para o disco local, será possível usar sua interface da linha de comandos (CLI) para executar análises de segurança. Ou, é possível executar um instalador integrado que inclua o plug-in de análise estática em seu ambiente de construção Maven ou em ambientes de desenvolvimento integrado (IDE) do Eclipse, IntelliJ IDEA ou Visual Studio.

Sobre Esta Tarefa

Ao usar a análise estática, será possível varrer o código-fonte para gerar um arquivo transferido por upload para a nuvem. O arquivo, um arquivo IRX criptografado (.irx), é varrido usando análise de rastreio para encontrar vulnerabilidades de segurança. Quando o arquivo IRX for gerado para linguagens compiladas, os arquivos de bytecode do aplicativo são convertidos em um Representação intermediária do código. Quando um arquivo IRX for gerado para linguagens de script, os arquivos de origem serão incluídos no arquivo IRX criptografado.

Ao usar os plug-ins do Eclipse, IntelliJ IDEA ou Visual Studio, acontecerá a geração e o upload de arquivo do IRX automaticamente a partir do IDE ao usar as ações de análise estática.

Dica: Ao executar o instalador, se você não estiver usando a análise estática em um determinado IDE que esteja instalado e pretende usá-la no futuro, você poderá desejar instalar o plug-in naquele IDE agora para que não seja necessário executar de novo o instalador posteriormente.
Atalho: Em vez de inserir sim ou não no instalador, é possível inserir y ou n como abreviações. Se você inserir abreviações em um instalador que esteja executando em uma máquina com idioma diferente de inglês, as abreviações inseridas deverão ser y ou n.
Importante: O Client Utility não deve ser colocado em um diretório que contém caracteres não ASCII.

Procedimento

Para configurar o do Client Utility:

  1. Selecione a plataforma na qual planeja executar o Client Utility e clique em Fazer download. Isso faz download de um arquivo SAClientUtil_<version>_<os>.zip (em que <version> é a versão atual do Client Utility e <os> é o sistema operacional ao qual o Client Utility se destina).
    Nota: É possível reabrir o formulário de boas-vindas selecionando Como crio um arquivo IRX? no formulário para selecionar um arquivo IRX.
  2. Extraia o arquivo em uma unidade local.
  3. Se você planeja usar a CLI para gerar o arquivo IRX, fazer upload dos arquivos IRX ou para gerenciar varreduras: inclua o local do diretório \bin do arquivo SAClientUtil_<version>_<os>.zip extraído na variável de ambiente PATH. Se você não fizer isto, todos os comandos precisarão ser qualificados usando o diretório \bin do arquivo SAClientUtil_<version>_<os>.zip extraído cada vez que o comando for emitido.
    Importante: Se você planeja varrer .NET com a CLI ou com o Microsoft Visual Studio, esses três arquivos .bat (no arquivo SAClientUtil_<version>_<os>.zip extraído) devem ser executados como administrador:
    • bin\register.bat
    • walalib\dotnet.libs40\RegisterInGAC.bat
    • walalib\dotnet.libs35\RegisterInGAC.bat

    Esses arquivos podem ser executados manualmente ou é possível executar os arquivos install-plugins.bat como um administrador (se você usar apenas a CLI, execute install-plugins.bat, mas insira Não quando for solicitado para instalar o plug-in de análise estática).

    Se você não executar esses arquivos como um administrador, uma mensagem de aviso aparecerá no console toda vez que a CLI for usada. Essa mensagem poderá ser ignorada se você estiver varrendo destinos Java.

    Dica: Após mudar o PATH, emita appscan version (Windows) ou appscan.sh version (Linux e macOS) em um prompt de comandos. Se a versão, início e outras informações do Static Analyzer Client Utility forem retornadas, o PATH está configurado corretamente.
  4. Se você planeja incluir o plug-in de análise estática no Eclipse, Microsoft Visual Studio (apenas no Windows), IntelliJ IDEA ou Maven: Localize e execute install-plugins.bat (Windows) ou install-plugins.sh (Linux e macOS) no Client Utility extraído. No Windows, execute o arquivo como um administrador. Se você planeja incluir o plug-in no Eclipse no Linux ou macOS, assegure-se de ter efetuado o login em uma conta que tenha permissões para modificar a instância do Eclipse na qual você está instalando.
    1. Por padrão, o Client Utility será conectado ao ASoC no IBM Cloud Marketplace. Caso pretenda se conectar ao serviço no IBM Bluemix, insira yes quando solicitado.
    2. Se você planeja varrer o código-fonte no Eclipse, siga os prompts no instalador. Se você optar por instalar o plug-in do Eclipse, será solicitado para inserir o local de sua instalação do Eclipse. Insira o caminho completo do diretório que contém o arquivo eclipse.exe (Windows) ou eclipse (Linux e macOS).
    3. Se você planeja usar o Maven para a geração do arquivo IRX, siga os prompts no instalador.
    4. Se você planeja varrer o código-fonte no IntelliJ IDEA, siga os prompts no instalador. Se você escolher instalar o plug-ing do IntelliJ IDEA, será solicitado que você insira o local do diretório de configuração do IntelliJ IDEA. Insira o caminho completo desse diretório.
    5. Somente Windows: se você planeja varrer o código-fonte no Visual Studio, siga os prompts no instalador. Se você optar por instalar o plug-in do Visual Studio, será solicitado para escolher em que versão do Visual Studio o plug-in deve ser instalado. Insira a versão correspondente à versão do Visual Studio na qual deseja instalar o plug-in.
    6. Feche o prompt de comandos pressionando qualquer tecla.
    7. Para qualquer IDE em que o plug-in tenha sido instalado, reinicie o IDE.
    8. Somente Linux: em um prompt de comandos, procure o diretório que contém o utilitário do cliente e inclua estas linhas no arquivo de recursos do seu shell (por exemplo, .bash_profile ou .shrc): 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export LD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${LD_LIBRARY_PATH}

      em que <client_utility> é o caminho para seu Client Utility. Após incluir essas linhas, recarregue seu shell antes de usar o plug-in de análise estática.

    9. apenas macOS: em um prompt de comandos, navegue até o diretório que contém o utilitário do cliente e inclua estas linhas no arquivo de recurso do shell (por exemplo, .bash_profile ou .shrc): 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export DYLD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${DYLD_LIBRARY_PATH}

      em que <client_utility> é o caminho para seu Client Utility. Após incluir essas linhas, recarregue seu shell antes de usar o plug-in de análise estática.

      Importante: Configurando a variável de ambiente no arquivo de recursos shell apenas afeta aplicativos ativados no terminal. Deve-se, portanto, usar a janela do terminal para ativar os ambientes de desenvolvimento integrado (IDE) no qual você deseja executar a análise estática.

O que Fazer Depois

Se estiver executando o Client Utility em um computador que esteja atrás de um proxy – use um destes métodos para especificar o proxy de modo que o Client Utility possa se conectar à nuvem:

  • Client Utility CLI e ambientes de desenvolvimento integrado (IDE) suportados: configure essa variável de ambiente do sistema ou global para que o proxy seja automaticamente reconhecido:
    • Windows: APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>
    • Linux e macOS: APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"

    Em que <proxy> é o nome do host do servidor proxy e <port> é o número da porta que o servidor proxy está usando.

    Ou, cada vez que você usar a CLI ou IDE (ativado a partir de um prompt de comandos ou terminal), será possível configurar o Client Utility para usar o proxy emitindo um comando:

    • Windows: set "APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
    • Linux e macOS: export APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
  • Maven: se você incluiu o plug-in de análise estática no Maven, será possível incluir a propriedade na variável de ambiente globalmente MAVEN_OPTS ou incluir a propriedade em cada comando.

Removendo o Static Analyzer Client Utility

Esta tarefa descreve como remover o plug-in do Static Analyzer Client Utility do ambiente de desenvolvimento integrado (IDE). Além disso, são fornecidas instruções para remover o Client Utility. Essas instruções precisarão ser seguidas somente se você planejar remover os componentes do Static Analyzer Client Utility do sistema. Se estiver fazendo upgrade desses componentes para uma nova versão do Client Utility, a instalação do plug-in removerá os componentes antigos automaticamente antes de instalar a nova versão.

Antes de Iniciar

Importante: Se você instalou o plug-in do Client Utility em um IDE, precisará executar um ou mais arquivos ou comandos no local do SAClientUtil_<version>_<os>.zip extraído para remover completamente o plug-in do IDE. Esses arquivos são:
  • Static Analyzer Client Utility Versão 6.0.865 e anterior:
    • walalib\dotnet.libs35\UnRegisterInGAC.bat
    • walalib\dotnet.libs40\UnRegisterInGAC.bat
  • Todas as versões do Static Analyzer Client Utility, incluindo a Versão 6.0.865 e anterior: abra um prompt de comandos como administrador e mude o diretório para o local no qual você extraiu o SAClientUtil_<version>_<os>.zip. Mude o diretório para a pasta bin\ e, em seguida, emita este comando:
    regsvr32 /u msdia80.dll

Para poder executar esses arquivos e comandos, não exclua o conteúdo do SAClientUtil_<version>_<os>.zip extraído até que tenha removido todos os plug-ins do IDE.

É possível executar esses arquivos e comandos antes ou após seguir as instruções abaixo para remover os plug-ins do IDE.

Procedimento

  • Eclipse:
    1. No Eclipse, selecione Ajuda > Instalar novo software na barra de menus principal do ambiente de trabalho.
    2. Na seleção Detalhes da caixa de diálogo Instalar, clique no link O que já está instalado?.
    3. Na lista de software instalado, localize e selecione Static Analyzer e, em seguida, selecione Desinstalar.
    4. Na caixa de diálogo Desinstalar, clique em Concluir.
    5. Quando solicitado em um prompt, reinicie o Eclipse.
  • IntelliJ IDEA:
    1. No IntelliJ IDEA, selecione Arquivo > Configurações na barra de menu principal.
    2. Na caixa de diálogo Configurações, selecione Plug-ins.
    3. Na lista de plug-ins, localize e selecione Plug-in do Static Analyzer.
    4. Clique em Desinstalar.
    5. Clique em Reiniciar IntelliJ IDEA.
  • Visual Studio:
    1. No Visual Studio, selecione Ferramentas > Extensões e atualizações no menu principal.
    2. Na lista de extensões e atualizações instaladas, localize e selecione IBM Static Analyzer.
    3. Clique em Desinstalar e, em seguida, quando solicitado a desinstalar o IBM Static Analyzer, clique em Sim.
    4. Selecione Reiniciar agora para reiniciar o Visual Studio - ou selecione Fechar se for reiniciar o Visual Studio mais tarde.
  • Client Utility: para remover o Client Utility, somente é preciso excluir o conteúdo extraído do SAClientUtil_<version>_<os>.zip do sistema. Se você incluiu o local do diretório \bin do Client Utility na variável de ambiente PATH, remova-o.

Suporte ao sistema operacional

O Static Analyzer Client Utility é suportado no Windows, macOS e Linux.

Windows: o Client Utility é suportado em sistemas de 64 bits e é executado no modo de 64 bits.

macOS: a CLI de análise estática, o plug-in do Eclipse e do Maven são suportados em versões 10.11 e superiores do macOS. O Client Utility é suportado em sistemas de 64 bits e executado no modo de 64 bits.

Linux: o Client Utility é suportado em sistemas de 32 bits e de 64 bits:
  • Em sistemas Linux de 64 bits, o Static Analyzer Client Utility é executado no modo de 32 bits.
  • O Linux de 64 bits requer a instalação destas bibliotecas antes da instalação do Static Analyzer Client Utility:
    • glibc.i686
    • libgcc.i686
    • libstdc++.i686

Suporte ao projeto Maven

Depois de fazer o download do Static Analyzer Client Utility, será possível optar por instalar um plug-in no Maven para que seja possível varrer projetos do Maven. O Maven Versão 3.x é suportado.

Gerando um arquivo do IRX usando a interface da linha de comandos (CLI)

Para poder iniciar uma análise de seus arquivos, deve-se gerar um arquivo do IRX que você envia para a nuvem. Se desejar usar a CLI, siga estas instruções para criar esse arquivo.

Procedimento

Para gerar o arquivo IRX:

  1. Assegure-se de abrir uma nova janela de prompt de comandos (não use a janela de prompt de comandos que foi usada para instalar o Client Utility).
  2. No prompt de comandos, emita o comando para mudar o diretório para o local que contém seus arquivos. Consulte Suporte ao idioma para aprender sobre os tipos de arquivo suportados. Se você estiver varrendo projetos Java no Eclipse, mude o diretório para sua área de trabalho do Eclipse.
    Nota:
    • Esta etapa não será necessária se estiver usando um arquivo de configuração para gerar o arquivo do IRX. Consulte o Configurando a geração de arquivo do IRX com a CLI.
    • Usando a CLI, não é possível varrer projetos individuais em uma área de trabalho do Eclipse; é possível varrer apenas a área de trabalho toda. Se desejar varrer projetos individuais, você deverá gerar o arquivo IRX no Eclipse.
    • Os arquivos .dll e .exe devem ser conjuntos .NET válidos.
  3. Emita o comando appscan prepare (Windows) ou appscan.sh prepare (Linux e macOS) . Esse comando é usado com as opções listadas nos tópicos Comandos de configuração (Windows) e Comandos de configuração (Linux e macOS). Por exemplo, é possível usar as opções -d e -n para especificar um diretório específico no qual salvará o arquivo IRX e para salvar o arquivo com um determinado nome de arquivo. Além disso, é possível executar o comando com um arquivo de configuração para especificar informações adicionais que ajudariam a gerar um arquivo completo do IRX (consulte Configurando a geração de arquivo do IRX com a CLI).
    Dica: Por padrão, o código Java e .NET de terceiros não é varrido durante a geração de arquivo do IRX. É possível modificar as configurações de exclusão de código de terceiro seguindo as instruções em Gerenciando exclusões de Java e .NET de terceiro. Para incluir o código de terceiros, especifique a opção -t ou --thirdParty ao emitir o comando prepare.

    Se você for o desenvolvedor de um código de terceiro que normalmente seria incluído em uma varredura, será necessário usar a configuração para incluir o código de terceiro.

    Nota: Ao varrer um código-fonte ou gerar um arquivo IRX, pode ser exibida uma mensagem sobre a atualização para o Client Utility mais recente. Consulte o Compatibilidade de versão do Client Utility e de serviço de nuvem.
  4. Após o comando ser emitido, o Client Utility reunirá informações para quaisquer arquivos suportados no diretório a partir do qual o comando foi emitido e todos os seus subdiretórios. Um arquivo do IRX é criado no diretório. O nome desse arquivo é baseado no diretório do qual o comando foi emitido e inclui um registro de data e hora (presumindo que você não usou uma opção de configuração para especificar um nome do arquivo diferente).
  5. Envie o arquivo IRX para a nuvem para análise. Para fazer isso, emita o comando appscan queue_analysis (Windows) ou appscan.sh queue_analysis (Linux e macOS). Esse comando é usado com as opções listadas nos tópicos Comandos de análises (Windows) ou Comandos de análises (Linux e macOS).

Gerando um arquivo IRX para um projeto Maven

Para poder iniciar uma análise de seus arquivos, deve-se gerar um arquivo do IRX que você envia para a nuvem. Se desejar varrer um projeto Maven, siga estas instruções para criar esse arquivo.

Procedimento

Para gerar o arquivo IRX:

  1. Assegure-se de abrir uma nova janela de prompt de comandos (não use a janela de prompt de comandos que foi usada para instalar o Client Utility).
  2. No prompt de comandos, emita um comando para mudar o diretório para o local que contém seu projeto Maven (a pasta que contém o arquivo pom.xml).
  3. Emita um destes comandos:
    • Se seu projeto Maven ainda não estiver construído, emita mvn package com.ibm.appscan:appscan-maven-plugin:prepare. Após o comando ser emitido, um arquivo do IRX será criado no diretório target do projeto Maven. O nome desse arquivo é baseado no projeto Maven de nível superior (o nome do projeto no arquivo pom.xml), presumindo que você não usou uma opção de configuração para especificar um nome de arquivo diferente.
    • Se seu projeto Maven já estiver construído, emita mvn com.ibm.appscan:appscan-maven-plugin:prepare. Após o comando ser emitido, um arquivo do IRX será criado no diretório target do projeto Maven. O nome desse arquivo é baseado no projeto Maven de nível superior (o nome do projeto no arquivo pom.xml), presumindo que você não usou uma opção de configuração para especificar um nome de arquivo diferente.
      Nota: Se seu projeto Maven tiver dependências de tempo de compilação em outro projeto na mesma compilação do reator, um erro do Maven existente (https://issues.apache.org/jira/browse/MNG-3283) fará com que o comando mvn com.ibm.appscan:appscan-maven-plugin:prepare falhe. Para uma solução alternativa para esse problema, faça uma das etapas a seguir:
      • Execute o comando mvn install em vez de mvn package antes de executar o comando prepare do Client Utility.
      • Emita o comando mvn package e o prepare do Client Utility ao mesmo tempo (emita mvn package com.ibm.appscan:appscan-maven-plugin:prepare).
    • Para que o arquivo IRX seja criado em um diretório diferente - ou tenha um nome de arquivo diferente - emita os comandos com a opção -Doutput:

      No Windows, emita este comando:

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<output_location>\<filename>.irx

      No Linux e macOS, emita este comando:

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<output_location>/<filename>.irx

      em que <local_de_saída> é o diretório em que você deseja salvar o arquivo e filename é o nome do arquivo.

    • O Client Utility inclui um compilador JSP do Tomcat. Esse compilador JSP é usado, por padrão, durante a geração de arquivo IRX. Se você estiver gerando um arquivo IRX para um arquivo WAR ou EAR e deseja que um compilador JSP diferente seja usado, emita o comando com uma opção -DjspCompiler. Por Exemplo:
      mvn com.ibm.appscan:appscan-maven-plugin:prepare -DjspCompiler=<compilador_jsp>

      em que <compilador_jsp> é o caminho para o compilador JSP que você deseja usar (especifique o diretório-raiz do servidor).

    Dica: Por padrão, o código de terceiro não é varrido durante a geração de arquivo do IRX. É possível modificar as configurações de exclusão de código de terceiro seguindo as instruções em Gerenciando exclusões de Java e .NET de terceiro. Para incluir código de terceiros, emita o comando com uma opção -DthirdParty.

    Se você for o desenvolvedor de um código de terceiro que normalmente seria incluído em uma varredura, será necessário usar a configuração para incluir o código de terceiro.

    Nota: Ao varrer um código-fonte ou gerar um arquivo IRX, pode ser exibida uma mensagem sobre a atualização para o Client Utility mais recente. Consulte o Compatibilidade de versão do Client Utility e de serviço de nuvem.
  4. Envie o arquivo IRX para a nuvem para análise. Para fazer isso, emita o comando appscan queue_analysis (Windows) ou appscan.sh queue_analysis (Linux e macOS). Esse comando é usado com as opções listadas nos tópicos Comandos de análises (Windows) ou Comandos de análises (Linux e macOS).

Configurando a geração de arquivo do IRX com a CLI

É possível usar um arquivo de configuração para a geração de arquivo IRX, em que é possível especificar destinos individuais, ou incluir ou excluir destinos. Além disso, é possível usar o arquivo de configuração para especificar informações adicionais que ajudariam a gerar um arquivo completo do IRX.

Uso do arquivo de configuração

Um arquivo de configuração pode ser usado para indicar quais destinos incluir ou excluir ao gerar um arquivo do IRX. Também é possível especificar diretórios que contêm artefatos construídos e especificar informações de construção que devem ser usadas durante a geração de arquivo IRX (por exemplo, é possível usar essa configuração para especificar um determinado compilador JDK ou JSP).

Sempre que emitir o comando appscan prepare (Windows) ou appscan.sh prepare (Linux e macOS) para gerar um arquivo IRX, o Static Analyzer Client Utility verifica automaticamente um arquivo appscan-config.xml no diretório atual. Se o arquivo for localizado, ele será usado automaticamente.

Se você tiver um arquivo de configuração em um diretório diferente ou com um nome de arquivo diferente, será possível usar o arquivo incluindo a opção -c ao emitir o comando appscan prepare ou appscan.sh prepare. Nesse caso, você emitiria appscan prepare -c <configuration_file> em (Windows) ou appscan.sh prepare -c <configuration_file> em (Linux e macOS), em que <configuration_file> é o caminho completo e o nome do arquivo de configuração.

Modelo de arquivo de configuração e configurações

Ao preparar o arquivo de configuração, use este modelo:

<Configuration>
  <Targets>
    <Target outputs-only="true/false" path="scan_target_path">
      <CustomBuildInfo build_info="info"/>
      <Include>string_pattern</Include>
      <Exclude>string_pattern</Exclude>
    </Target>
  </Targets>
</Configuration>

Em que é possível usar o elemento Targets para indicar quais destinos varrer ao gerar o arquivo do IRX. Os atributos de destino (Target) e os atributos filho são usados conforme a seguir:

  • O atributo outputs-only do elemento Target só é usado quando você deseja varrer um diretório e forçar o Client Utility para localizar apenas os arquivos de saída de compilação (como arquivos .jar, .war e .class). Por padrão, esse atributo é configurado como false. Isso significa que o Client Utility procura no diretório para determinar se ele é um destino (como um servidor de aplicativos ou uma área de trabalho do Eclipse) ou se o diretório contém itens como scripts de construção, arquivos Maven .pom e arquivos make. Se desejar varrer o destino a ser tratado como um diretório simples e tiver localizado apenas os arquivos de saída, especifique outputs-only="true" no elemento Target.

    Por exemplo, se especificar <Target outputs-only="false" path="C:\Tomcat">, o Client Utility presumirá que o destino é seu servidor de aplicativos Tomcat e procurará seus arquivos .war implementados. Entretanto, se especificar <Target outputs-only="true" path="C:\Tomcat">, o Client Utility tratará esse local como um diretório e localizará todos os arquivos de saída de construção dentro dele.

  • Use o atributo path do elemento Target para especificar o caminho para um destino de varredura ou um diretório de destinos de varredura (<caminho_do_destino_de_varredura>). Quando um diretório é especificado, todos os seus subdiretórios são incluídos durante a varredura.

    Por exemplo, se você especificar <Target outputs-only="false" path="C:\WebSphere\AppServer85\profiles\AppSrv01">, o Client Utility localizará todos os arquivos .ear implementados no perfil AppSrv01. Se você especificar <Target outputs-only="false" path="C:\WebSphere\AppServer85">, o Client Utility localizará todos arquivos .ear implementados em todos os perfis.

  • Também é possível especificar esses subelementos Target opcionais:
    • É possível usar o elemento CustomBuildInfo para configurar atributos. Usando essa parte do modelo,
      <CustomBuildInfo build_info="info"/>
      é possível especificar as informações de construção dependendo do idioma de destino. Para alguns idiomas, mais de um atributo pode ser configurado, por exemplo, é possível configurar <CustomBuildInfo build_info_1="info_1" build_info_2="info_2" build_info_3="info_3"/>, dependendo do idioma de destino.
      • Para Java, é possível configurar estes atributos:
        <CustomBuildInfo additional_classpath="dependency_path"
  jdk_path="JDK_path" jsp_compiler="JSP_compiler_path"/>
        • Especifique mais caminhos de classe usando o atributo additional_classpath. No Windows, separe vários caminhos de classe com um ponto e vírgula. No Linux e macOS, separe os vários caminhos de classe com dois pontos.
        • Use o atributo jdk_path para especificar o caminho para a sua instalação do JDK.
        • Use o atributo jsp_compiler para especificar o caminho para seu compilador JSP - por exemplo, jsp_compiler="C:\Tomcat".
      • Para o .NET (somente Windows), é possível configurar estes atributos:
        <CustomBuildInfo reference="assembly_reference"
  configuration="build_configuration"/>
        • Inclua as referências usando o atributo reference. Separe as diversas referências com um ponto-e-vírgula.
        • Use o atributo configuration para incluir uma configuração de construção para a descoberta da solução do Visual Studio.
      • Para o C/C++ (somente Windows), é possível configurar estes atributos:
        <CustomBuildInfo configuration="build_configuration"
  include_paths="include_directories" macros="macros" compiler_opts=/>
        • Use o atributo configuration para incluir uma configuração de construção.
        • Especifique os caminhos de inclusão usando o atributo include_paths. Separe os diversos caminhos de inclusão com um ponto-e vírgula.
        • Inclua as macros usando o atributo macros. Separe as diversas macros com um ponto-e-vírgula.
        • Especifique as opções do compilador com o atributo compiler_opts. Separe as diversas opções com um ponto-e-vírgula.
    Nota: Os valores que são configurados por esses atributos são herdados por subdestinos. Por exemplo, se seu destino for um arquivo EAR que inclui subdestinos WAR e JAR, será presumido que o WAR e o JAR tenham os mesmos valores configurados para o arquivo EAR que usa esses atributos.
  • Use o elemento-filho Include para especificar padrões de arquivo (<string_pattern>) para incluir ao gerar o arquivo do IRX. O comportamento Include depende do tipo de destino, conforme descrito na seção Comportamento include e exclude de destino. Para especificar múltiplos padrões include, inclua cada padrão em sua própria tag <Include></Include>. Por exemplo,
    <Include>string_pattern_1</Include>
<Include>string_pattern_2</Include>
    Nota: Se especificar padrões include e exclude que entram em conflito, os padrões exclude terão precedência.
  • Use o elemento-filho Exclude para especificar padrões de arquivo para excluir ao gerar o arquivo do IRX. O comportamento Exclude depende do tipo de destino, conforme descrito na seção Comportamento include e exclude de destino. Para especificar múltiplos padrões exclude, inclua cada padrão em sua própria tag <Exclude></Exclude>.
    Nota: Se especificar padrões include e exclude que entram em conflito, os padrões exclude terão precedência.

Exemplo de arquivo de configuração

<Configuration>
  <Targets>
    <Target path="C:\test\MyApplication.ear">
      <CustomBuildInfo additional_classpath="C:\java\lib;C:\java2\lib"
        jdk_path="C:\Program Files\Java\jdk1.7" jsp_compiler="C:\Tomcat"/>
      <Include>ShoppingCart.war</Include>
      <Include>*Account.jar</Include>
      <Exclude>thirdPartyJars/</Exclude>
      <Exclude>*test*</Exclude>
    </Target>
  </Targets>
</Configuration>
Neste arquivo de configuração:
  • C:\test\MyApplication.ear é o destino.
  • C:\java\lib e C:\java2\lib são procurados em busca de quaisquer dependências de arquivo de classe que sejam necessárias para compilação.
  • C:\Program Files\Java\jdk1.7 é usado para localizar dependências.
  • O compilador C:\Tomcat é usado para compilação JSP em vez do aplicativo Apache Tomcat Versão 7 que está incluído com o Static Analyzer Client Utility.
  • Quando o arquivo do IRX é gerado, o arquivo ShoppingCart.war é incluído, assim como todos os arquivos que terminam em *Account.jar. Tudo no thirdPartyJars/ do arquivo EAR é excluído, assim como quaisquer arquivos que tenham a sequência test em seu nome do arquivo. Se houver conflitos, os padrões exclude terão precedência. Por exemplo, se o arquivo EAR contiver um arquivo testCustomerAccount.jar, ele será excluído, porque o padrão test exclude tem precedência sobre o padrão *Account.jar include.

Erros ou avisos que ocorrerão ao gerar um arquivo de análise estática IRX

Sintomas

Ao gerar um arquivo do IRX, é possível receber erros ou avisos ou a geração do arquivo do IRX pode falhar. Quando a geração do arquivo do IRX falha, o nome do arquivo resultante é anexo ao _failed (por exemplo, my_irx_file_failed.irx). Quando isso acontece, você tem várias opções para resolução de problemas de geração de arquivos do IRX.

Resolvendo o problema

  • Usar um arquivo de configuração para resolução de problemas na geração do arquivo IRX

    Se você gerar um arquivo IRX sem usar um arquivo de configuração e a geração do arquivo não for concluída (ou for concluída com erro), o Static Analyzer Client Utility gerará um arquivo de configuração temporário para ser usado em varreduras subsequentes. Este arquivo será semelhante ao modelo de arquivo de configuração, pelo fato de que conterá um elemento Targets com informações que são conhecidas sobre seus destinos. No entanto, o arquivo também conterá um elemento <TargetSettings> para que você modifique incluindo informações que permitirão que o Client Utility gere um arquivo IRX completo. Esse arquivo é chamado de appscan-config.xml e é salvo no diretório atual. Isto significa que o arquivo será usado automaticamente para a próxima varredura.

    Por exemplo, dada a geração de arquivo do IRX para um destino que contém dependências externas que não pode ser localizado, é possível que receba uma mensagem como esta:

    C:\myTargetDirectory>appscan prepare
Um arquivo IRX foi criado, mas pode estar incompleto.
C:\myTargetDirectory\myTarget.jar requer os pacotes a seguir:
      org.apache.jasper.*
Um arquivo appscan-config.xml foi criado em: C:\myTargetDirectory\appscan-config.xml
Atualize o arquivo e execute novamente o comando.

    O arquivo appscan-config.xml que é gerado se parece com algo como:

    <Configuration>
  <Targets>
    <Target outputs-only="true" path="C:\myTargetDirectory"/>
  </Targets>
  <Settings/>
  <TargetSettings>
    <Target path="C:\myTargetDirectory\myTarget.jar"/>
      <!--Inclua o caminho das dependências a seguir no atributo "additional_classpath" abaixo, separado por ponto e vírgula.
			  org.apache.jasper.*-->
      <CustomBuildInfo additional_classpath=""/>
    </Target>
  </TargetSettings>
</Configuration>

    Para resolver o problema, atualize <CustomBuildInfo additional_classpath=""/> para incluir as classes org.apache.jasper.* no caminho da classe, salve o arquivo appscan-config.xml e, em seguida, execute novamente o comando appscan prepare (Windows) ou appscan.sh prepare (Linux e macOS). Ao especificar o caminho para o arquivo ou arquivos JAR, a sintaxe de caminho de classe Java padrão é suportada (por exemplo, curingas são suportados).

    Se você encontrar problemas semelhantes enquanto já estiver usando um arquivo de configuração, receberá a mensagem que indica os problemas. Use essa mensagem para atualizar seu arquivo de configuração existente e, em seguida, execute novamente o comando.

    O arquivo de configuração é descrito em Configurando a geração de arquivo do IRX com a CLI.

  • Gerar arquivos IRX no modo de depuração

    Se estiver gerando um arquivo IRX a partir da CLI, será possível emitir o comando appscan prepare (Windows) ou appscan.sh prepare (Linux e macOS) com uma opção de depuração. Isto faz com que mais arquivos de log sejam gerados.

  • Arquivos de log associados à geração do arquivo IRX

    Para ver os erros ou avisos, é possível consultar os arquivos de log que também são gerados. Os arquivos de log são salvos em um archive .zip. Por padrão, o arquivo é salvo no mesmo local em que seu IRX é salvo ou é possível especificar outro local usando a opção -l do comando appscan prepare (Windows) ou appscan.sh prepare (Linux e macOS). O nome do arquivo .zip é baseado no nome do arquivo do IRX que é gerado.

Utilizando anotações

As anotações podem ser usadas seguindo estas etapas:

  1. O suporte para anotações é ativado por padrão. O arquivo de anotação .jar localiza-se no diretório \annotation do arquivo SAClientUtil_<version>_<os>.zip extraído.

  4. Se você estiver usando a CLI de análise estática, crie um arquivo de configuração e nele configure CustomBuildInfo additional_classpath="dependency_path", então dependency_path incluirá o local do arquivo com.ibm.security.annotation.jar.

Como as anotações podem ser usadas para declarar partes de código seguro, elas devem ser usadas com muito cuidado. Elas não devem ser usadas para códigos que precisem ser varridos em busca de vulnerabilidades de segurança.   type="bool"
/>

Para desativar o recurso, altere value="true" para value="false".

@ValidatorMethod

Validadores são métodos que executam verificações em dados de entrada e, geralmente, retornam um valor booleano que indica se a entrada é válida ou não. Em vez de aceitar ou rejeitar uma entrada usando validadores, é possível mudar a entrada do usuário para um formato aceitável. Esses métodos são chamados de limpadores.

Usando a anotação @ValidatorMethod, é possível identificar todos os métodos validadores e limpadores no código-fonte do aplicativo. Durante as varreduras de análise estática, essas informações serão usadas para remover os fluxos de dados que passarem por esses métodos desde que os dados sejam considerados seguros agora.

Todos os rastreios que fluem por um método marcado com essa anotação são removidos. Isso é útil quando um determinado grupo de rastreios é identificado como falso positivo ou menos importante ou interessante do que os outros. Essa anotação pode ser usada para filtrar esses rastreios ou para ocultá-los com o propósito de reduzir a confusão.

Essa anotação é usada para identificar retornos de chamada ou pontos de entrada para um aplicativo. Todos os argumentos são considerados como carregadores de contaminação.

Configurando expressões include PHP não resolvidas e classes PHP desconhecidas

Depois de gerar um arquivo IRX para PHP, um arquivo de configuração é criado que permite configurar a geração de IRX na próxima vez em que você varrer os arquivos.

O arquivo de configuração, php-scan-config.xml, é gerado no local a partir do qual você emitiu o comando prepare. O arquivo contém duas seções, <UserInputs> e <Php-Unknown-Configuration>.

A seção <UserInputs> contém definições de configuração que podem ser modificadas para sua varredura de PHP. A seção é pré-preenchida com valores de amostra. É possível anexar a esses valores usando um ponto e vírgula (;). As configurações nesta seção são:

  • <RegularExpression>: esse é um recurso avançado que permite que uma expressão regular seja especificada para texto da procura.
  • <ReplaceText>: essa configuração é usada para procura e substituição de texto simples. O texto da procura é usado no estado em que se encontra e, se for localizado em qualquer lugar no texto incluído, será substituído pelo texto de substituição.
  • <Function>: essa configuração é usada quando uma chamada de função tiver de ser substituída. O texto da procura deverá ser o nome da função sem parênteses. O texto da procura será aumentado para procurar por uma chamada de função com o nome especificado seguido pelos parênteses e corresponderá a tudo nos parênteses.
  • <SimpleText>: texto de substituição simplificado. Use isso quando o texto de substituição for uma sequência única.
  • <SimpleRegEx>: expressão regular de substituição simplificada. Use isso quando a expressão regular de substituição for uma sequência única.
  • <AdditionalIncludePaths>: use essa configuração para incluir caminhos include adicionais que contenham arquivos referenciados.

A seção <Php-Unknown-Configuration> contém todas as expressões include não resolvidas e as classes desconhecidas que foram localizadas na última varredura. Use essas informações para concluir a seção <UserInputs>.

  • <Summary>: resume o número de expressões include desconhecidas (<NumberOfUnknownIncludeExprs>) e as classes desconhecidas (<NumberOfUnknownClasses>) na última varredura.
  • <SourceFile>: todas as expressões include desconhecidas e as classes desconhecidas são listadas alfabeticamente pelo nome do arquivo de origem. Para cada arquivo de origem, o caminho do arquivo (<FilePath>) lista o caminho absoluto da origem, a classe desconhecida (<UnknownClass>) e a expressão include desconhecida (<UnknownIncludes>).

Informações sobre a classe desconhecida (<UnknownClass>)

Marca Descrição
<ClassName> O nome da classe que foi referenciada no código-fonte, junto com a localização no arquivo de origem (<FilePath>) em que a classe é usada. Talvez você queira ver o local em um editor para ver como a classe desconhecida deve ser resolvida.
<Status> Indica se a classe é Resolved ou Not Resolved.
<ResolvedBy> Indica como a classe desconhecida foi resolvida. Os valores incluem:
  • AutoResolver: O aplicativo usou heurísticas internas para localizar o arquivo.
  • SearchReplace: Foram aplicadas uma ou mais regras de procura e substituição no texto incluído para gerar o texto atualizado.

Informações sobre a expressão include desconhecida (<UnknownIncludes>)

Marca Descrição
<IncludeText> O texto da expressão como ela existe no código-fonte, junto com a localização no arquivo de origem (<FilePath>) em que a expressão é usada. Talvez você deseja olhar o local em um editor para ver como a expressão include desconhecida deve ser resolvida.
<UpdatedText> O texto atualizado que foi usado durante a última varredura.
<Status> Indica se a expressão é Resolved ou Not Resolved.
<ResolvedBy> Indica como o texto atualizado foi gerado. Os valores incluem:
  • AutoResolver: O aplicativo usou heurísticas internas para localizar o arquivo.
  • SearchReplace: Foram aplicadas uma ou mais regras de procura e substituição no texto incluído para gerar o texto atualizado.
  • SearchReplace+AutoResolver: Foram aplicadas uma ou mais regras de procura e substituição no texto incluído para gerar o texto atualizado e, em seguida, as heurísticas internas foram aplicadas para localizar o arquivo.
  • SearchReplace+IncludePath: Foram aplicadas uma ou mais regras de procura e substituição no texto incluído para gerar o texto atualizado e, em seguida, foram combinados com um diretório no caminho include para localizar o arquivo.
  • SearchReplace+RelativeDir: Foram aplicadas uma ou mais regras de procura e substituição no texto incluído para gerar o texto atualizado e, em seguida, foram localizadas relativas ao diretório de origem do arquivo que contém a expressão include.

Gerenciando exclusões de Java e .NET de terceiro

Por padrão, o código Java e .NET de terceiro não é varrido durante a geração de arquivo do IRX. É possível gerenciar o código de terceiro que é excluído seguindo as instruções neste tópico.

Sobre Esta Tarefa

Ao gerar um arquivo do IRX para seu código, o código de terceiro é excluído para que sua avaliação não inclua informações e provas sobre o código que você não está interessado em analisar. Para modificar o que é excluído, siga as etapas neste tópico. Se, em vez disso, você desejar incluir o código de terceiro ao fazer a varredura, consulte estes tópicos:

  • Gerando um arquivo do IRX usando a interface da linha de comandos (CLI)
  • Gerando um arquivo IRX para um projeto Maven
  • Varrendo em ambientes de desenvolvimento integrado

Procedimento

  1. Localize o diretório config do arquivo do SAClientUtil_<version>_<os>.zip extraído (em que <version> é a versão atual do Client Utility).
  2. Nesse diretório, localize o arquivo dot_net.exclusions (.NET) ou java.exclusions (Java).
    Nota: Outros arquivos .exclusions nesse diretório são experimentais e modificá-los não afetará as exclusões de terceiro.
  3. Abra o arquivo em um editor de texto, no qual você verá que ele é XML neste formato: 
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<Exclusions>
  <packages>
    <package name="package_name"/>
  </packages>
  ...
</Exclusions>
  4. Edite o arquivo para incluir pacotes a serem excluídos.
    Nota:
    • Os nomes dos pacotes que são especificados corresponderão a todos os pacotes com o mesmo prefixo (eles serão tratados como se fossem seguidos por um curinga). Por exemplo, se você incluir <package name="com.mycompany.common"/> como uma exclusão, <package name="com.mycompany.common.action1"/> e <package name="com.mycompany.common.action2"/> serão excluídos.
    • Somente entradas que são incluídas no contêiner <packages> afetam as exclusões de terceiro. Incluir classes e métodos não afetará as exclusões.
  5. Salve o arquivo e, em seguida, gere um arquivo do IRX para seu código-fonte.

O que Fazer Depois

Se você modificar os arquivos de exclusão, será necessário seguir estas melhores práticas:
  • Mantenha um backup dos arquivos de exclusão modificados. Em particular, lembre-se que, se fizer upgrade para uma nova versão do Static Analyzer Client Utility, você estará fazendo download das versões dos arquivos de exclusão sem suas atualizações. Ao atualizar o Client Utility, mantenha uma cópia de seus arquivos de exclusão modificados e, em seguida, copie-os para o novo Client Utility.
  • Mantenha as modificações em um bloco único de entradas XML para que seja possível copiar e colá-las facilmente caso seja necessário.

appscan get_pubkey

Sintaxe:

appscan get_pubkey -d <save_path>

Descrição:

Faça download da chave de criptografia pública para uso em um computador que não está conectado à Internet.

appscan.sh prepare -c <configuration_file> -d <save_path> -l <log_path> -n <file_name> -t,--thirdParty -v,--verbose -X,--debug

Descrição:

  • -d: especifique -d <save_path>, em que <save_path> é o diretório no qual deseja salvar o arquivo IRX.
  • -l: Especifique -l <caminho_de_log>, em que <caminho_de_log> é o diretório no qual você deseja salvar os arquivos de log.

  • -v,--verbose: especifique -v ou --verbose para exibir uma saída mais informativa durante a geração do arquivo do IRX.
  • -X,--debug: Especifique -X ou --debug para executar o comando inteiro no modo de depuração. Ao executar no modo de depuração, mais arquivos de log são gerados para resolução de problemas.
Dica: Para todos os comandos, as opções podem ser usadas em qualquer ordem.

Exemplos:

    • Você está emitindo o comando a partir de um diretório que não contém arquivos de avaliação. Nesse caso, a opção -f deve ser usada para especificar o caminho e o nome do arquivo de avaliação a ser empacotado.

appscan.sh get_pubkey

Sintaxe:

appscan.sh get_pubkey -d <save_path>

Descrição:

Faça download da chave de criptografia pública para uso em um computador que não está conectado à Internet.

Se estiver gerando um arquivo do IRX a partir de um computador conectado à Internet, esse comando não será necessário uma vez que uma chave de criptografia é transferida por download automaticamente ao emitir o comando prepare. Se uma chave de criptografia já estiver presente no computador, ela será atualizada, se necessário, quando você emitir o comando prepare.

Entretanto, se estiver gerando um arquivo do IRX a partir de um computador que não está conectado à Internet, é possível fazer download da chave de criptografia usando esse comando. É possível copiar a chave de criptografia no computador que não está conectado à Internet para uso ao gerar o arquivo do IRX. Se seu computador estiver conectado à Internet, emita o comando prepare novamente para atualizar automaticamente a chave de criptografia.
