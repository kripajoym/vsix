Начало работы с Static Analyzer

Статический анализ позволяет проверить исходный код на наличие уязвимостей. Для этого необходимо загрузить маленькую утилиту. В случае сохранения на локальном диске эта утилита предлагает интерфейс командной строки (CLI), позволяющий выполнять анализ защиты на всех поддерживаемых языках. Кроме того, можно запустить программу установки, добавляющую модули статического анализа в Eclipse, Maven, IntelliJ IDEA или Visual Studio. В Eclipse и IntelliJ IDEA поддерживается сканирование проектов Java, а в Visual Studio - сканирование .NET (C#, ASP.NET, VB.NET).

Процедура

  • Информация по применению службы Application Security on Cloud в IBM Cloud Marketplace приведена на веб-странице http://www.ibm.com/support/knowledgecenter/SSYJJF_1.0.0/ApplicationSecurityonCloud/Welcome.html.
  • Информация о применении Application Security on Cloud for Bluemix приведена в https://console.ng.bluemix.net/docs/services/ApplicationSecurityonCloud/index.html.
  • Кроме того, этот файл readme содержит инструкции по установке продукта Static Analyzer Client Utility и работе с ним.

Настройка Static Analyzer Client Utility

Для статического анализа загрузите небольшую утилиту Client Utility. В случае сохранения на локальном диске эта утилита предлагает интерфейс командной строки (CLI), позволяющий выполнять анализ защиты. Кроме того, можно запустить программу установки, добавляющую модуль статического анализа в среду компоновки Maven или в интегрированные среды разработки (IDE) Eclipse, IntelliJ IDEA и Visual Studio.

Об этой задаче

В случае применения статического анализа можно выполнить сканирование исходного кода для создания файла, который передается в облако. Файл (зашифрованный файл IRX (.irx)) сканируется с помощью анализа трассировки для поиска уязвимостей защиты. Если файл IRX создан для компилируемых языков, то файлы байтового кода приложения преобразуются в Intermediate Representation. Если файл IRX создан для языков сценариев, то исходные файлы добавляются в зашифрованный файл IRX.

В случае применения модулей Eclipse, IntelliJ IDEA и Visual Studio файл IRX создается и передается из IDE в автоматическом режиме при выполнении действий статического анализа.

Совет: В случае применения программы установки (если статический анализ не применяется в конкретной среде IDE, но эта среда установлена и может использоваться в будущем) рекомендуется установить модуль в IDE, чтобы избежать повторного запуска программы установки впоследствии.
Быстрый путь: Вместо значений yes и no в программе установки можно вводить сокращения y и n. Если в системе настроен язык, отличный от английского, то в программе установки следует вводить сокращения y и n.
Важное замечание: Client Utility не следует размещать в каталоге, содержащем символы, отличные от ASCII.

Процедура

Для настройки Client Utility выполните следующие действия:

  1. Выберите платформу, на которой планируется использовать Client Utility, и щелкните на Загрузить. Будет загружен файл SAClientUtil_<версия>_<ОС>.zip (где <версия> - текущая версия Client Utility, а <ОС> - операционная система, для которой предназначен Client Utility).
    Прим.: Форму приветствия можно снова открыть, щелкнув на Как создать файл IRX? в форме для выбора файла IRX.
  2. Извлеките и сохраните файл на локальном диске.
  3. Если вы планируете использовать CLI для создания файла IRX, передачи файлов IRX или управления операциями сканирования: добавьте расположение каталога \bin полученного файла SAClientUtil_<версия>_<ОС>.zip в переменную среды PATH. Без этого действия все команды придется уточнять, добавляя каталог \bin полученного файла SAClientUtil_<версия>_<ОС>.zip, всякий раз при вводе команды.
    Важное замечание: Если планируется сканировать .NET с помощью интерфейса командной строки или Microsoft Visual Studio, то следующие три файла .bat (извлеченные из файла SAClientUtil_<версия>_<ОС>.zip) следует запускать от имени администратора:
    • bin\register.bat
    • walalib\dotnet.libs40\RegisterInGAC.bat
    • walalib\dotnet.libs35\RegisterInGAC.bat

    Эти файлы можно выполнить вручную. Кроме того, можно выполнить файл install-plugins.bat от имени администратора (в случае применения только интерфейса командной строки выполните файл install-plugins.bat и введите no в ответ на предложение установить модуль статического анализа).

    Если эти файлы выполняются не от имени администратора, то при каждом обращении к интерфейсу командной строки в консоль будет выводиться предупреждение. На это сообщение можно не обращать внимания в случае сканирования целевых объектов Java.

    Совет: После изменения переменной PATH выполните appscan version (Windows) или appscan.sh version (Linux и macOS) в командной строке. Если будет возвращена версия Static Analyzer Client Utility, а также исходное расположение и другая информация, значит переменная PATH задана правильно.
  4. Если планируется добавить модуль статического анализа в Eclipse, Microsoft Visual Studio (только в Windows) или Maven: найдите и выполните файл install-plugins.bat (Windows) или install-plugins.sh (Linux и macOS) в извлеченной утилите Client Utility. В Windows файл должен выполняться от имени администратора. Если планируется добавить модуль в Eclipse в Linux или macOS; убедитесь, что есть права доступа на изменение экземпляра Eclipse, куда устанавливается модуль.
    1. По умолчанию Client Utility подключается к службе ASoC в IBM Cloud Marketplace. Для подключения к службе в IBM Bluemix введите yes в ответ на запрос.
    2. Если планируется сканировать исходный код в Eclipse, следуйте приглашениям программы установки. Если вы решили установить модуль Eclipse, появится приглашение указать установочный каталог экземпляра Eclipse. Введите полный путь к каталогу, содержащему файл eclipse.exe (Windows) или eclipse (Linux и macOS).
    3. Если планируется использовать Maven для создания файлов IRX, следуйте приглашениям программы установки.
    4. Если планируется сканировать исходный код в IntelliJ IDEA, следуйте приглашениям программы установки. Если вы решили установить модуль IntelliJ IDEA, появится приглашение указать каталог конфигурации IntelliJ IDEA. Введите полный путь к этому каталогу.
    5. Только Windows: если планируется сканировать исходный код в Visual Studio, следуйте приглашениям программы установки. Если вы решили установить модуль Visual Studio, появится приглашение выбрать версию Visual Studio, для которой должен быть установлен модуль. Введите версию, соответствующую версии Visual Studio, в которую требуется установить модуль.
    6. Нажмите любую клавишу, чтобы закрыть командную строку.
    7. После установки модуля перезапустите IDE.
    8. Только для Linux: в командной строке перейдите в каталог, содержащий клиентскую утилиту, и добавьте в свой файл ресурсов оболочки (например, .bash_profile или .shrc) следующие строки: 
      export APPSCAN_INSTALL_DIR=<клиентская-утилита>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export LD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${LD_LIBRARY_PATH}

      где <клиентская-утилита> - путь к Client Utility. После добавления этих строк перезагрузите оболочку, перед тем как использовать модуль статического анализа.

    9. Только для macOS: в командной строке перейдите в каталог, содержащий клиентскую утилиту, и добавьте в свой файл ресурсов оболочки (например, .bash_profile или .shrc) следующие строки: 
      export APPSCAN_INSTALL_DIR=<клиентская-утилита>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export DYLD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${DYLD_LIBRARY_PATH}

      где <клиентская-утилита> - путь к Client Utility. После добавления этих строк перезагрузите оболочку, перед тем как использовать модуль статического анализа.

      Важное замечание: Настройка переменной среды в файле ресурсов оболочки влияет только на приложения, запускаемые из этого терминала. Поэтому запускать IDE надо именно в том окне терминала, где планируется запускать статический анализ.

Дальнейшие действия

Если Client Utility выполняется в системе, которая находится за прокси-сервером, используйте один из следующих способов для указания прокси-сервера, чтобы утилита Client Utility могла подключиться к облаку:

  • CLI и поддерживаемые IDE Client Utility: настройте эту глобальную или системную переменную среды для автоматического распознавания прокси-сервера:
    • Windows: APPSCAN_OPTS=-Dhttps.proxyHost=<прокси-сервер> -Dhttps.proxyPort=<порт>
    • Linux и macOS: APPSCAN_OPTS="-Dhttps.proxyHost=<прокси-сервер> -Dhttps.proxyPort=<порт>"

    Где <прокси-сервер> - это имя хоста прокси-сервера, а <порт> - это номер порта, используемый прокси-сервером.

    Или при каждом открытии CLI или IDE (запущенной из командной строки или терминала) можно настраивать Client Utility для использования прокси-сервера следующей командой:

    • Windows: set "APPSCAN_OPTS=-Dhttps.proxyHost=<прокси-сервер> -Dhttps.proxyPort=<порт>"
    • Linux и macOS: export APPSCAN_OPTS="-Dhttps.proxyHost=<прокси-сервер> -Dhttps.proxyPort=<порт>"
  • Maven: если в Maven добавлен модуль статического анализа, можно добавить этот параметр в переменную среды MAVEN_OPTS глобально или добавлять его в каждую команду.

Удаление Static Analyzer Client Utility

В этом разделе приведены инструкции по удалению модулей Static Analyzer Client Utility из интегрированной среды разработки (IDE). Кроме того, приведены инструкции по удалению Client Utility. Эти инструкции следует выполнять только в том случае, если необходимо удалить компоненты Static Analyzer Client Utility из системы. В случае обновления этих компонентов до новой версии Client Utility старые компоненты будут автоматически удалены перед установкой новой версии.

Прежде чем начать

Важное замечание: Если модуль Client Utility установлен в IDE, то для его полного удаления потребуется запустить один или несколько файлов или команд в излеченном расположении SAClientUtil_<версия>_<ОС>.zip. Эти файлы перечислены ниже:
  • Static Analyzer Client Utility версии 6.0.865 и ниже:
    • walalib\dotnet.libs35\UnRegisterInGAC.bat
    • walalib\dotnet.libs40\UnRegisterInGAC.bat
  • Все версии Static Analyzer Client Utility, включая версию 6.0.865 и ниже: Откройте командную строку от имени администратора и перейдите в каталог, в котором был распакован SAClientUtil_<версия>_<ОС>.zip. Перейдите в каталог bin\ и выполните следующую команду:
    regsvr32 /u msdia80.dll

Для того чтобы иметь возможность выполнения этих файлов и команд, не удаляйте распакованное содержимое SAClientUtil_<версия>_<ОС>.zip до тех пор, пока не будут удалены все модули IDE.

Поддержка операционной системы

Static Analyzer Client Utility поддерживается в Windows, macOS и Linux.

Windows: Client Utility поддерживается в 64-разрядных системах и работает в 64-разрядном режиме.

macOS: CLI статического анализа, модуль Eclipse и модуль Maven поддерживаются в macOS версии 10.11 и более поздних версиях. Client Utility поддерживается в 64-разрядных системах и работает в 64-разрядном режиме.

Linux: Client Utility поддерживается в 32-разрядных и 64-разрядных системах:
  • В 64-разрядных системах Linux Static Analyzer Client Utility работает в 32-разрядном режиме.
  • Перед установкой Static Analyzer Client Utility в 64-разрядную систему Linux требуется установка следующих библиотек:
    • glibc.i686
    • libgcc.i686
    • libstdc++.i686

Совместимость версий облачной службы и Client Utility

Версия Static Analyzer Client Utility автоматически проверяется в следующих случаях:

  • Выполните команду appscan prepare в Windows или команду appscan.sh prepare в Linux и macOS.
  • Выполняется действие Выполнить статический анализ в интегрированной среде разработки, в которой установлен модуль статического анализа.
  • Использует параметр prepare для создания файла IRX в проекте Maven.
  • Передайте файл IRX командой appscan queue_analysis в Windows или командой appscan.sh queue_analysis в Linux и macOS.
  • Передает файл IRX в облако.

При выполнении любого из действий подготовки или Выполнить статический анализ может быть выдано сообщение о том, чтоб доступна новая версия Client Utility. В этом случае можно продолжить работу без обновления Client Utility либо можно обновить Client Utility для поддержки новых функций и возможностей.

При выполнении указанных выше действий в неподдерживаемой версии Client Utility выдается сообщение, указывающее на необходимость обновления Client Utility. В этом случае загрузите и установите последнюю версию Client Utility.

Создание файла IRX с помощью интерфейса командной строки (CLI)

Для того чтобы запустить анализ файлов, необходимо создать файл IRX для передачи в облако. В этом разделе приведены инструкции по созданию этого файла с помощью CLI.

Процедура

Для создания файла IRX выполните следующие действия:

  1. Должно быть открыто новое окно командной строки (не используйте окно командной строки, которое применялось для установки Client Utility).
  2. В командной строке введите команду для перехода в каталог, содержащий файлы. Список поддерживаемых типов файлов приведен в разделе Поддержка языков. В случае сканирования проектов Java в Eclipse перейдите в каталог рабочей области Eclipse.
    Прим.:
    • Это действие не требуется при создании файла IRX с помощью файла конфигурации. См. раздел Настройка создания файла IRX с помощью CLI.
    • С помощью интерфейса командной строки невозможно сканировать отдельные проекты в рабочей области Eclipse, можно сканировать только всю рабочую область. Для сканирования отдельных проектов необходимо создать файл IRX в Eclipse.
    • Файлы .dll и .exe должны быть верными сборками .NET.
  3. Выполните команду appscan prepare (Windows) или appscan.sh prepare (Linux и macOS). Параметры этой команды описаны в разделах Команды настройки (Windows) и Команды настройки (Linux macOS). Например, с помощью параметров -d и -n можно указать конкретный каталог и имя сохраняемого файла IRX. Кроме того, в команде можно указать файл конфигурации с дополнительной информацией, помогающей создать полный файл IRX (см. раздел Настройка создания файла IRX с помощью CLI).
    Совет: по умолчанию сторонний код Java и .NET не сканируется во время создания файла IRX. Параметры исключения стороннего кода можно изменить в соответствии с инструкциями из раздела Управление внешними исключениями Java и .NET. Для включения стороннего кода укажите параметр -t или --thirdParty при вводе команды prepare.

    Этот параметр должен указывать разработчик стороннего кода, включаемого в сканирование.

    Прим.: При сканировании или создании файла IRX может выдаваться сообщение об обновлении Client Utility до новой версии. См. раздел Совместимость версий облачной службы и Client Utility.
  4. После ввода команды Client Utility собирает информацию для всех поддерживаемых файлов в каталоге, из которого команда была введена, включая все подкаталоги. В каталоге создается файл IRX. Файлу присваивается имя на основе имени каталога, из которого была введена команда, с добавлением даты и системного времени (если не было указано другое имя файла с помощью параметра конфигурации).
  5. Откройте файл IRX в облако для анализа. Для этого выполните команду appscan queue_analysis (Windows) или appscan.sh queue_analysis (Linux и macOS). Параметры этой команды описаны в разделах Команды анализа (Windows) и Команды анализа (Linux и macOS).

Создание файла IRX для проекта Maven

Для того чтобы запустить анализ файлов, необходимо создать файл IRX для передачи в облако. В этом разделе приведены инструкции по созданию этого файла для сканирования проектов Maven.

Процедура

Для создания файла IRX выполните следующие действия:

  1. Должно быть открыто новое окно командной строки (не используйте окно командной строки, которое применялось для установки Client Utility).
  2. В командной строке введите команду для перехода в каталог, содержащий ваш проект Maven (папка с файлом pom.xml).
  3. Введите одну из следующих команд:
    • Если проект Maven еще не скомпонован, введите mvn package com.ibm.appscan:appscan-maven-plugin:prepare. После выполнения этой команды будет создан файл IRX в каталоге target проекта Maven. Файлу присваивается имя на основе имени проекта Maven верхнего уровня (имя проекта в файле pom.xml), если не было указано другое имя файла в параметре конфигурации.
    • Если проект Maven уже скомпонован, введите mvn com.ibm.appscan:appscan-maven-plugin:prepare. После выполнения этой команды будет создан файл IRX в каталоге target проекта Maven. Файлу присваивается имя на основе имени проекта Maven верхнего уровня (имя проекта в файле pom.xml), если не было указано другое имя файла в параметре конфигурации.
      Прим.: Если в проекте Maven есть зависимости (на этапе компиляции) от другого проекта в той же компоновке реактора, то из-за известной ошибки Maven (https://issues.apache.org/jira/browse/MNG-3283) произойдет сбой команды mvn com.ibm.appscan:appscan-maven-plugin:prepare. Для обхода этой неполадки выполните одно из следующих действий:
      • Введите команду mvn install вместо mvn package перед выполнением команды Client Utility prepare.
      • Запустите команду mvn package и команду Client Utility prepare одновременно (введите mvn package com.ibm.appscan:appscan-maven-plugin:prepare).
    • Если требуется создать файл IRX в другом каталоге или присвоить файлу другое имя, введите команду с параметром -Doutput:

      В Windows введите следующую команду:

      mvn
package com.ibm.appscan:appscan-maven-plugin:prepare
-Doutput=<расположение-вывода>\<имя-файла>.irx

      В Linux и macOS выполните следующую команду:

      mvn
package com.ibm.appscan:appscan-maven-plugin:prepare
-Doutput=<расположение-вывода>/<имя-файла>.irx

      где <расположение-вывода> - каталог для сохранения файла, а файл - имя файла.

    • В состав Client Utility входит компилятор JSP Tomcat. Этот компилятор JSP применяется по умолчанию во время создания файла IRX. Если создается файл IRX для файла WAR или EAR и планируется использовать другой компилятор JSP, введите команду с параметром -DjspCompiler. Пример:
      mvn com.ibm.appscan:appscan-maven-plugin:prepare -DjspCompiler=<компилятор-jsp>

      где <компилятор-jsp> - путь к компилятору JSP, который требуется использовать (укажите корневой каталог сервера).

    Совет: По умолчанию сторонний код не сканируется во время создания файла IRX. Параметры исключения стороннего кода можно изменить в соответствии с инструкциями из раздела Управление внешними исключениями Java и .NET. Для включения стороннего кода введите команду с параметром -DthirdParty.

    Этот параметр должен указывать разработчик стороннего кода, включаемого в сканирование.

    Прим.: При сканировании или создании файла IRX может выдаваться сообщение об обновлении Client Utility до новой версии. См. раздел Совместимость версий облачной службы и Client Utility.
  4. Откройте файл IRX в облако для анализа. Для этого выполните команду appscan queue_analysis (Windows) или appscan.sh queue_analysis (Linux и macOS). Параметры этой команды описаны в разделах Команды анализа (Windows) и Команды анализа (Linux и macOS).

Использование файла конфигурации

С помощью файла конфигурации можно указать, какие целевые объекты следует включить или исключить при создании файла IRX. Также можно указать каталоги, содержащие артефакты компоновки, и ввести информацию о компоновке, которую необходимо использовать при создании файла IRX (например, с помощью этого значения можно задать определенный компилятор JSP или JDK).

При каждом выполнении команды appscan prepare (Windows) или appscan.sh prepare (Linux и macOS) для создания файла IRX Static Analyzer Client Utility автоматически проверяет наличие файла appscan-config.xml в текущем каталоге. Если этот файл обнаружен, то он автоматически используется.

Если нужный файл конфигурации находится в другом каталоге или имеет другое имя, то для использования этого файла следует добавить параметр -c при вводе команды appscan prepare или appscan.sh prepare. В этом случае можно ввести команду appscan prepare -c <файл-конфигурации> в (Windows) или appscan.sh prepare -c <файл-конфигурации> в (Linux и macOS), где <файл-конфигурации> - полное имя файла конфигурации (включая путь).

Шаблон файла конфигурации и значения параметров

При подготовке файла конфигурации используйте следующий шаблон:

<Configuration>
  <Targets>
    <Target outputs-only="true/false" path="путь-к-целевому-объекту-сканирования">
      <CustomBuildInfo build_info="info"/>
      <Include>шаблон-строки</Include>
      <Exclude>шаблон-строки</Exclude>
    </Target>
  </Targets>
</Configuration>

С помощью элемента Targets можно указать, какие целевые объекты требуется сканировать при создании файла IRX. Атрибуты элемента Target и дочерние элементы используются следующим образом:

  • Атрибут outputs-only элемента Target используется, только когда требуется сканировать каталог и настроить Client Utility для поиска только выходных файлов компоновки, таких как .jar, .war и .class. По умолчанию этому атрибуту присвоено значение false. Это означает, что Client Utility будет вести поиск в каталоге, чтобы определить, является ли он целевым объектом, таким как сервер приложений или рабочая область Eclipse, содержит ли каталог такие объекты, как сценарии компоновки, файлы Maven .pom и файлы make. Если нужно, чтобы целевой объект сканирования воспринимался как простой каталог и чтобы искались только файлы вывода; укажите outputs-only="true" в элементе Target.

    Например, если указать <Target outputs-only="false" path="C:\Tomcat">, то Client Utility будет считать целевым объектом сервер приложений Tomcat и выполнит поиск его развернутых файлов .war. Но если указать <Target outputs-only="true" path="C:\Tomcat">, то Client Utility будет считать расположение каталогом и будет искать в нем все файлы вывода компоновки.

  • Относящийся к элементу Target атрибут path применяется для указания пути к целевому объекту сканирования или к каталогу целевых объектов сканирования (<путь-к-целевому-объекту-сканирования>). Если указан каталог, то при сканировании обрабатываются все его вложенные каталоги.

    Например, если указать <Target outputs-only="false" path="C:\WebSphere\AppServer85\profiles\AppSrv01">, то Client Utility найдет все развернутые файлы .ear в профайле AppSrv01. Если указать <Target outputs-only="false" path="C:\WebSphere\AppServer85">, то Client Utility найдет все развернутые файлы .ear во всех профайлах.

  • Также можно указать следующие необязательные дочерние элементы Target:
    • Элемент CustomBuildInfo можно использовать для указания атрибутов. С помощью части шаблона
      <CustomBuildInfo build_info="info"/>
      можно указать сведения о компоновке в зависимости от целевого языка. Для некоторых языков можно указать больше одного атрибута, например можно указать <CustomBuildInfo build_info_1="info_1" build_info_2="info_2" build_info_3="info_3"/> в зависимости от целевого языка.
      • Для Java можно задать следующие атрибуты:
        <CustomBuildInfo additional_classpath="путь-к-зависимости"
  jdk_path="путь-к-JDK" jsp_compiler="путь-к-компилятору-JSP"/>
        • Атрибут additional_classpath для указания дополнительных путей к классам. В Windows для разделения нескольких путей к классам применяется символ точка с запятой. В Linux и macOS для разделения нескольких путей к классам применяется двоеточие.
        • Атрибут jdk_path для указания пути к установленному экземпляру JDK.
        • Атрибут jsp_compiler для указания пути к своему компилятору JSP, например jsp_compiler="C:\Tomcat".
      • Для .NET (только для Windows) можно задать следующие атрибуты:
        <CustomBuildInfo reference="путь-к-зависимости"
  configuration="конфигурация-компоновки"/>
        • Добавьте пути к зависимостям с помощью атрибута reference. Несколько путей разделяются точкой с запятой.
        • С помощью атрибута configuration добавьте конфигурацию компоновки для поиска решения Visual Studio.
      • Для C/C++ (только для Windows) можно добавить следующие атрибуты:
        <CustomBuildInfo configuration="конфигурация-компоновки"
  include_paths="каталоги-include" macros="macros" compiler_opts=/>
        • С помощью атрибута configuration добавьте конфигурацию компоновки.
        • С помощью атрибута include_paths укажите пути в файлам include. Несколько путей разделяются точкой с запятой.
        • Включите макрос с помощью атрибута macros. Несколько макросов разделяются точкой с запятой.
        • Укажите опции компилятора с помощью атрибута compiler_opts. Несколько опций разделяются точкой с запятой.
    Прим.: Значения, заданные этими атрибутами, наследуются дочерними целевыми объектами. Например, если целевым объектом является файл EAR, включающий дочерние целевые объекты WAR и JAR, то считается, что WAR и JAR содержат те же значения, которые были заданы для файла EAR с помощью этих атрибутов.
  • С помощью дочернего элемента Include можно указать шаблоны файлов (<шаблон-строки>), включаемых при создании файла IRX. Поведение элемента Include зависит от типа целевого объекта (см. раздел Поведение при включении и исключении целевого объекта). Для указания нескольких шаблонов include каждый шаблон следует добавлять в отдельном теге <Include></Include>. Пример:
    <Include>шаблон-строки-1</Include>
<Include>шаблон-строки-2</Include>
    Прим.: В случае указания конфликтующих шаблонов include и exclude приоритет имеют шаблоны exclude.
  • С помощью дочернего элемента Exclude можно указать шаблоны файлов, исключаемых при создании файла IRX. Поведение элемента Exclude зависит от типа целевого объекта (см. раздел Поведение при включении и исключении целевого объекта). Для указания нескольких шаблонов exclude каждый шаблон следует добавлять в отдельном теге <Exclude></Exclude>.
    Прим.: В случае указания конфликтующих шаблонов include и exclude приоритет имеют шаблоны exclude.

Поведение при включении и исключении целевого объекта

Табл. 2. Поведение при включении и исключении целевого объекта
Тип целевого объекта Поведение Примеры
.jar, .dll Если целевым объектом является файл .jar или .dll, включение или исключение целевых объектов не влияют на сканирование, так как эти файлы не могут содержать целевых объектов.  
.war Если целевым объектом является файл .war, то можно включать файлы .jar в WEB-INF/lib или исключать их оттуда.
Прим.: По умолчанию файлы .jar воспринимаются как коды третьей стороны и добавляются в путь к классам файла .war (и исключаются по умолчанию).
 com.ibm.*.jar
.ear Если целевым объектом является .ear, то можно включать или исключать файлы .jar и .war.
  • com.ibm.*.jar
  • JSPWiki.war
каталог Если целевым объектом является каталог, то можно включать или исключать путь или имя файла любого целевого объекта или типа целевого объекта.
  • com.ibm.*.jar
  • test/: включает или исключает любой объект в каталоге test/ или в его подкаталогах.
  • test/*.jar: включает или исключает все файлы .jar в каталоге test/, но не в подкаталогах.
веб-сервер (поддерживаемые веб-серверы описаны в разделе Требования к системе) Если целевым объектом является веб-сервер, то можно включать или исключать имя файла любого приложения, развернутого на сервере.
  • JSPWiki.war
  • MyApp.ear
  • *.war (если целевым объектом является сервер Tomcat)
.sln (.NET) Если целевым объектом является файл решения Visual Studio, содержащий проекты .NET, то можно включать или исключать имя файла любой сборки .NET, созданной проектом в решении.
  • *\MyLib.dll
  • *\MyApp.exe
.sln (C/C++) Если целевым объектом является файл решения Visual Studio, содержащий неуправляемые проекты C/C++, то можно включать или исключать имя файла любого проекта.
  • *\myProject.vcxproj

Пример файла конфигурации

<Configuration>
  <Targets>
    <Target path="C:\test\MyApplication.ear">
      <CustomBuildInfo additional_classpath="C:\java\lib;C:\java2\lib"
        jdk_path="C:\Program Files\Java\jdk1.7" jsp_compiler="C:\Tomcat"/>
      <Include>ShoppingCart.war</Include>
      <Include>*Account.jar</Include>
      <Exclude>thirdPartyJars/</Exclude>
      <Exclude>*test*</Exclude>
    </Target>
  </Targets>
</Configuration>
В этом файле конфигурации:
  • C:\test\MyApplication.ear - это целевой объект.
  • В расположениях C:\java\lib и C:\java2\lib выполняется поиск всех зависимостей файлов классов, необходимых для компиляции.
  • C:\Program Files\Java\jdk1.7 используется для поиска зависимостей.
  • Для компиляции JSP используется компилятор C:\Tomcat, а не приложение Apache Tomcat версии 7, входящее в состав Static Analyzer Client Utility.
  • При создании файла IRX включается файл ShoppingCart.war, как и все файлы с окончанием *Account.jar. Все объекты thirdPartyJars/ из файла EAR исключаются, как и любые файлы, содержащие строку test в имени файла. В случае конфликтов приоритет имеют шаблоны exclude. Например, если в файле EAR содержится файл testCustomerAccount.jar, он исключается, так как шаблон test exclude имеет более высокий приоритет, чем шаблон *Account.jar include.

Сканирование в интегрированных средах разработки

Инструкции по сканированию исходного кода с помощью модуля статического анализа, установленного в интегрированной среде разработки (IDE) Eclipse, IntelliJ IDEA или Visual Studio. В Eclipse и IntelliJ IDEA поддерживается сканирование проектов Java, а в Visual Studio - сканирование .NET (C#, ASP.NET, VB.NET).

Прежде чем начать

По умолчанию внешний код исключается из сканирования Java и .NET. Параметры исключения стороннего кода можно изменить в соответствии с инструкциями из раздела Управление внешними исключениями Java и .NET.
  • Сканирование внешнего кода в Eclipse, IntelliJ IDEA и Visual Studio можно разрешить следующими способами:
    • Установите эту глобальную или системную переменную среды перед запуском IDE: 
      APPSCAN_OPTS=-DthirdParty
    • Каждый раз перед запуском IDE можно выполнять следующую команду:
      set APPSCAN_OPTS=-DthirdParty
  • В случае сканирования в Eclipse можно выполнить следующее действие: измените файл eclipse.ini перед запуском Eclipse, чтобы в раздел -vmargs был включен параметр -DthirdParty.
  • В случае сканирования в IntelliJ IDEA можно выполнить следующее действие: в каталоге bin экземпляра IntelliJ IDEA внесите изменения в файл idea.exe.vmoptions перед запуском IntelliJ IDEA. Добавьте -DthirdParty в файл и сохраните его.

Этот параметр должен указывать разработчик стороннего кода, включаемого в сканирование.

Процедура

Для сканирования исходного кода и открытия оценок или отчетов выполните следующие действия:

  1. Убедитесь, что модуль установлен в IDE. Если во время установки среда IDE была открыта, перезапустите ее.
  2. Выберите элемент для сканирования:
    • В Eclipse выберите один или несколько проектов для сканирования. Для сканирования всей рабочей области Eclipse выберите все проекты.
    • В IntelliJ IDEA выберите модули проектов для сканирования.
    • В Visual Studio выберите решения, проекты или веб-сайты для сканирования.
  3. Щелкните правой кнопкой мыши на выбранном элементе и выберите Анализ защиты > Выполнить статический анализ.
    Прим.: При сканировании или создании файла IRX может выдаваться сообщение об обновлении Client Utility до новой версии. См. раздел Совместимость версий облачной службы и Client Utility.
  4. После подготовки сканирования откроется окно входа в систему, если вход в службу еще не выполнен. Открываемое окно зависит от параметра соединения, установленного после запуска программы установки install-plugins.bat (Windows) или install-plugins.sh (Linux и macOS). В окне диалога введите идентификационные данные службы:
    • В случае подключения к Cloud Marketplace: если ключ API создается в службе Application Security on Cloud, будут получены ИД ключа и Пароль ключа. Введите эти значения в поля ИД и Пароль соответственно. Если ключ API еще не создан, щелкните на ссылке в окне для его создания.
    • В случае подключения к Bluemix: введите ИД привязки и Пароль, указанные в идентификационных данных службы.

      Для того чтобы узнать идентификационные данные службы Bluemix, выберите Идентификационные данные службы на левой панели навигации сводной панели службы. См. раздел Включение внешних приложений для использования служб Bluemix.

    При входе в службу создается файл ключа шифрования. В дальнейшем на этот файл будут ссылаться другие действия при взаимодействии со службой ASoC.

    При входе в Bluemix срок действия этого файла ключа истекает через 2 часа после его создания. В этом случае после истечения срока действия потребуется снова войти в службу, чтобы продолжить работу. Кроме того, при входе в систему можно включить переключатель Сохранить идентификационные данные. Если этот переключатель включен, то Client Utility автоматически выполняет повторную идентификацию после истечения срока действия ключа.

  5. Только IBM Cloud Marketplace: при наличии подключения к службе Application Security on Cloud в IBM Cloud Marketplace сканирования статического анализа в IDE должны быть связаны с существующим приложением Application Security on Cloud. После запуска сканирования предлагается выбрать приложение для связывания.

    После выбора приложения для связывания со сканированиями сохраняется следующая информация:

    • В Eclipse приложение связывается со всей рабочей областью. Информация о связывании сохраняется в файле <рабочая-область>/.metadata/.plugins/com.ibm.appscan.eclipse.plugin/.scans/.app, где <рабочая-область> - это расположение рабочей области Eclipse.
    • В IntelliJ IDEA приложения связываются с отдельными модулями проекта. Информация о связывании сохраняется в файле <проект>/.scans/.app, где <проект> - это расположение модуля проекта.
    • В Visual Studio приложения связываются с решениями. Информация о связывании сохраняется в файле <решение>/.scans/.app, где <решение> - это каталог, содержащий файл .sln.

    Если удалить файл .app из любого из этих расположений, то вам будет предложено выбрать связанное приложение при следующем запуске сканирования. При необходимости для изменения связанного приложения удалите файл .app и снова запустите сканирование.

    Прим.: Связывание приложения игнорируется в случае подключения к службе Application Security on Cloud в IBM Bluemix.
  6. После передачи сканирования на выполнение открывается представление Сканирования Static Analyzer.
  7. После завершения сканирования открывается уведомление с ссылками на результаты или отчет. Кроме того, сканирование добавляется в представление Сканирования Static Analyzer. Отображается следующая информация: имя сканирования, состояние, время запуска и завершения, а также уровень серьезности обнаруженных уязвимостей.
    1. Результат: щелкните на ссылке в уведомлении, чтобы открыть результаты, либо дважды щелкните на значке в столбце Результат (в Visual Studio результат также можно открыть из системного лотка). Откроется интерактивная оценка со списком результатов, упорядоченных по группе исправления. Группа исправления представляет наиболее общий узел, через который передаются сгруппированные находки. Обычно, если реализовано исправление для группы исправления, можно получить максимальный результат при минимальных трудозатратах. Группа исправлений также может считаться логической точкой объединения, где связанные находки могут проверяться одновременно. Примечание: группа исправления может не совпадать с местом размещения исправления. Будущий рефакторинг, применяемые правила разработки кода и другие факторы могут не позволить использовать расположение группы исправления в качестве исправления.

      В оценке:

      • В каждой группе исправления отображается следующая информация: расположение рекомендуемого исправления, ссылка на исходный код расположения исправления, уязвимость и число вхождений уязвимости, которые будут исправлены в исходном коде.

      Важное замечание: Интерактивные оценки (результаты) недоступны для IntelliJ IDEA и отдельных уровней подписки.

Ошибки и предупреждения при создании файла IRX статического анализа

Симптомы

Во время создания файла IRX могут выдаваться ошибки или предупреждения. Кроме того, может произойти сбой создания файла IRX. В случае сбоя создания файла IRX к имени файла результата добавляется добавлена строка _failed (пример: my_irx_file_failed.irx). Когда такое происходит, существует ряд возможностей по устранению неполадок создания файла IRX.

Разрешение проблемы

  • Использование файла конфигурации для устранения неполадок создания файла IRX

    Если файл IRX создается без использования файла конфигурации и создание файла не выполняется (или выполняется с ошибкой), Static Analyzer Client Utility создаст временный файл конфигурации для использования в последующих операциях сканирования. Этот файл похож на шаблон файла конфигурации тем, что содержит элемент Targets с известной информацией о ваших целевых объектах. Однако файл будет также содержать элемент <TargetSettings>, который можно изменять, добавляя информацию, с помощью которой Client Utility сможет создать полноценный файл IRX. Этот файл называется appscan-config.xml и сохраняется в текущем каталоге. Это означает, что файл будет автоматически использоваться в следующей операции сканирования.

    Например, если создается файл IRX для целевого объекта, содержащего внешние зависимости, которые не удается найти, то может быть выдано сообщение следующего вида:

    C:\myTargetDirectory>appscan prepare
Создан файл IRX, но,
возможно, он сформирован не полностью.
Для C:\myTargetDirectory\myTarget.jar требуются следующие пакеты:
      org.apache.jasper.*
Файл appscan-config.xml создан в расположении: C:\myTargetDirectory\appscan-config.xml
Обновите файл и повторите команду.

    Созданный файл appscan-config.xml выглядит следующим образом:

    <Configuration>
  <Targets>
    <Target outputs-only="true" path="C:\myTargetDirectory"/>
  </Targets>
  <Settings/>
  <TargetSettings>
    <Target path="C:\myTargetDirectory\myTarget.jar"/>
      <!--Добавьте путь к следующим зависимостям в атрибут "additional_classpath" ниже, используя в качестве разделителя символ точка с запятой.
			  org.apache.jasper.*-->
      <CustomBuildInfo additional_classpath=""/>
    </Target>
  </TargetSettings>
</Configuration>

    Для устранения неполадки обновите строку <CustomBuildInfo additional_classpath=""/> для включения org.apache.jasper.* в путь к классам, сохраните файл appscan-config.xml, затем еще раз выполните команду appscan prepare (Windows) или appscan.sh prepare (Linux и macOS). При указании пути к файлам JAR поддерживается стандартный синтаксис пути к классам Java (например, поддерживаются символы подстановки).

    Если похожие неполадки возникнут уже при использовании файла конфигурации, то будет выдано сообщение о неполадках. С помощью этого сообщения можно обновить свой файл конфигурации, а затем повторить команду.

    Файл конфигурации описан в разделе Настройка создания файла IRX с помощью CLI.

  • Создание файлов IRX в режиме отладки

    В случае создания файла IRX через интерфейс командной строки можно выполнить команду appscan prepare (Windows) и appscan.sh prepare (Linux и macOS) с параметром debug. Это приведет к созданию дополнительных файлов протокола.

  • Файлы протокола, связанные с созданием файла IRX

    Для просмотра сообщений об ошибках и предупреждений можно открыть файлы протокола, которые также создаются. Файлы протокола сохраняются в файле архива .zip. По умолчанию файл сохраняется в тот же каталог, куда сохраняется IRX. Можно указать другой каталог с помощью параметра -l команды appscan prepare (Windows) или appscan.sh prepare (Linux и macOS). Имя файла .zip формируется из имени созданного файла IRX.

Сбой сканирования с выдачей сообщения об ошибке Приложение с ИД x не существует

Симптомы

При запуске сканирования на основе модуля Jenkins или IDE либо интерфейса командной строки (CLI) выдается следующее сообщение об ошибке:

Приложение с ИД x не существует
Не удалось запустить сканирование защиты: возникли неполадки при
передаче файла IRX в службу анализа.

Причины

В предыдущих версиях для связывания приложений в Application Security on Cloud применялся параметр app_id. В последнем обновлении вместо app_id применяется guid. Служба ASoC предложит заново связать приложения с учетом этого обновления. Для модулей Jenkins и IDE, а также CLI необходимо выполнить дополнительные действия.

Разрешение проблемы

Для модуля Jenkins заново сохраните конфигурацию модуля Jenkins в соответствии с инструкциями из раздела Добавление анализа защиты на сервер автоматизации Jenkins и снова запустите сканирование.

Для модуля IDE заново выберите приложения для сканирования и нажмите кнопку OK.

Для CLI новый guid (32-символьный идентификатор) заменяет app_id. Пример: 33c3281a-6d7c-e711-b334-002590731623. С помощью команды appscan list_apps определите guid для каждого приложения и замените app_id в команде queue_analysis в сценариях или командной строке.

Поддерживаемые аннотации исходного кода Java

В процессе сканирования исходного кода Java с помощью статического анализа поддерживаются следующие аннотации уровня методов: @ValidatorMethod, @CallbackMethod и @SuppressSecurityTrace.

  • Использование аннотаций
  • @ValidatorMethod
  • @SuppressSecurityTrace
  • @CallbackMethod

Использование аннотаций

Ниже приведены инструкции по работе с аннотациями:

  1. Аннотации поддерживаются по умолчанию. Файл .jar аннотаций расположен в каталоге \annotation архива SAClientUtil_<версия>_<ОС>.zip.
  2. В случае сканирования предварительно скомпилированных классов, файлов .war или файлов .jar добавьте com.ibm.security.annotation.jar в проект Java, содержащий исходный код с аннотациями, затем заново скомпонуйте проект.
  3. В случае сканирования интегрированной среды разработки (IDE), в которой установлен модуль статического анализа, добавьте com.ibm.security.annotation.jar в путь к классам Java.
  4. В случае применения CLI статического анализа создайте файл конфигурации и укажите в нем параметр CustomBuildInfo additional_classpath="dependency_path", где dependency_path содержит расположение файла com.ibm.security.annotation.jar.

Аннотации можно добавить в исходный код перед сканированием или после сканирования в ходе сортировки с целью поиска и устранения ложных срабатываний. В случае сканирования в IDE, где установлен модуль статического анализа, в отчете о сканировании доступна кнопка Помогите мне исправить неполадку. При нажатии этой кнопки предлагаются точки для вставки аннотаций.

Аннотации позволяют вставить информацию непосредственно в исходный код в виде аннотаций защиты. Поскольку с помощью аннотаций можно объявить безопасные фрагменты исходного кода, их следует использовать с осторожностью. Их не следует использовать для кода, сканируемого на предмет уязвимостей безопасности. Аналитик в области безопасности может отключить аннотации в файле irgen-engine.apsettings, который расположен в каталоге \config распакованного файла SAClientUtil_<версия>_<ОС>.zip. В этом файле найдите следующий параметр:

<Setting 
  name="process_security_annotations"
  value="true"
  default_value="true"
  description="When turned on, method annotations will be processed - used by
DotNet and Java LSPs"
  display_name="Process Security Annotations"
  type="bool"
/>

Для выключения функции измените значение с value="true" на value="false".

Прим.: Если указано значение value="false", то для применения аннотаций укажите value="true".

@ValidatorMethod

Методы проверки анализируют входные данные и часто возвращают булевское значение, которое указывает, допустимы ли входные данные. Вместо принятия или отклонения входных данных с помощью методов проверки можно преобразовать пользовательские входные данные в приемлемый формат. Для этой цели применяются методы очистки.

Аннотация @ValidatorMethod позволяет указать все методы проверки и очистки в исходном коде приложения. В ходе сканирований статического анализа с учетом этой информации удаляются потоки данных, передаваемые через эти методы, поскольку такие данные считаются безопасными.

Прим.: В настоящее время отсутствует способ указания проверенных параметров метода с аннотациями. В ходе сканирований статического анализа все входные параметры считаются проверенными.

@SuppressSecurityTrace

Все данные трассировки, передаваемые через метод с этой аннотацией, удаляются. Это удобно. если отдельные данные трассировки рассматриваются как ложные срабатывания или могут быть проигнорированы. С помощью этой аннотации можно отфильтровать такие данные трассировки или скрыть их.

@CallbackMethod

Указывает обратные вызовы и точки входа в приложение. Все аргументы считаются искаженными.

Настройка необработанных операторов PHP include и неизвестных классов PHP

После создания файла IRX для PHP создается файл конфигурации, позволяющий настроить создание IRX в ходе следующего сканирования файлов.

Файл конфигурации php-scan-config.xml создается в каталоге, из которого была выполнена команда prepare. Файл содержит два раздела: <UserInputs> и <Php-Unknown-Configuration>.

Раздел <UserInputs> содержит параметры конфигурации, которые можно изменить для сканирования PHP. В этом разделе по умолчанию указаны примеры значений. Дополнительные значения можно добавить с помощью точки с запятой (;). Этот раздел содержит следующие параметры:

  • <RegularExpression>: расширенная функция, позволяющая указать регулярное выражение для поиска текста.
  • <ReplaceText>: применяется для простого поиска и замены. Строка поиска используется непосредственно и, если обнаруживается во включаемом тексте, заменяется целевым текстом.
  • <Function>: применяется для замены вызова функции. В качестве строки поиска должно быть указано имя функции без скобок. Искомый текст будет дополнен для поиска вызова функции с указанным именем, за которым идут скобки с любым текстом внутри.
  • <SimpleText>: упрощенный заменяющий текст. Этот параметр следует использовать в случае замены отдельной строки.
  • <SimpleRegEx>: упрощенное заменяющее регулярное выражение. Этот параметр следует использовать в случае замены отдельной строки с помощью регулярного выражения.
  • <AdditionalIncludePaths>: позволяет добавить дополнительные включаемые каталоги со связанными файлами.

Раздел <Php-Unknown-Configuration> содержит все необработанные выражения include и неизвестные классы, обнаруженные во время последнего сканирования. Эта информация применяется для заполнения раздела <UserInputs>.

  • <Summary>: общее число неизвестных выражений include (<NumberOfUnknownIncludeExprs>) и неизвестных классов (<NumberOfUnknownClasses>) для последнего сканирования.
  • <SourceFile>: список всех выражений include и неизвестных классов в алфавитном порядке по имени исходного файла. Для каждого исходного файла указаны полный путь к файлу (<FilePath>), неизвестные классы (<UnknownClass>) и неизвестные выражения include (<UnknownIncludes>).

Информация о неизвестном классе (<UnknownClass>)

Тег Описание
<ClassName> Имя класса, указанное в исходном коде, вместе с расположением в исходном файле (<FilePath>). Иногда может потребоваться взглянуть на это место в редакторе, чтобы понять требуемый способ обработки неизвестного класса.
<Status> Состояние класса: Обработан или Не обработан.
<ResolvedBy> Задает способ обработки неизвестного класса. Допустимые значения:
  • AutoResolver: приложение использовало внутренние эвристики для поиска файла.
  • SearchReplace: применено одно или несколько правил поиска и замены к включенному тексту для создания обновленного текста.

Информация о неизвестном выражении include (<UnknownIncludes>)

Тег Описание
<IncludeText> Текст выражения, указанный в исходном коде, вместе с расположением выражения в исходном файле (<FilePath>). Иногда может потребоваться взглянуть на это место в редакторе, чтобы понять требуемый способ обработки неизвестного выражения include.
<UpdatedText> Обновленный текст, который использовался во время последнего сканирования.
<Status> Состояние выражения: Обработано или Не обработано.
<ResolvedBy> Указан способ создания обновленного текста. Допустимые значения:
  • AutoResolver: приложение использовало внутренние эвристики для поиска файла.
  • SearchReplace: применено одно или несколько правил поиска и замены к включенному тексту для создания обновленного текста.
  • SearchReplace+AutoResolver: применено одно или несколько правил поиска и замены к включенному тексту для создания обновленного текста, а затем применены внутренние эвристики для поиска файла.
  • SearchReplace+IncludePath: применено одно или несколько правил поиска и замены к включенному тексту для создания обновленного текста, который затем был объединен с каталогом в пути оператора include для поиска файла.
  • SearchReplace+RelativeDir: применено одно или несколько правил поиска и замены к включенному тексту для создания обновленного текста, а затем выполнен поиск относительно каталога файла, содержащего оператор include.

Управление внешними исключениями Java и .NET

По умолчанию сторонний код Java и .NET не сканируется во время создания файла IRX. Исключениям стороннего кода можно управлять в соответствии с инструкциями, приведенными в этом разделе.

Об этой задаче

При создании файла IRX для собственного кода сторонний код исключается, чтобы в оценке не было лишней информации и результатов. Для того чтобы настроить исключения, выполните инструкции из данного раздела. Для включения стороннего кода в сканирование обратитесь к следующим разделам:

  • Создание файла IRX с помощью интерфейса командной строки (CLI)
  • Создание файла IRX для проекта Maven
  • Сканирование в интегрированных средах разработки

Процедура

  1. Найдите каталог config извлеченного файла SAClientUtil_<версия>_<ОС>.zip (где <версия> - текущая версия Client Utility).
  2. В этом каталоге найдите файл dot_net.exclusions (.NET) или java.exclusions (Java).
    Прим.: Остальные файлы .exclusions в этом каталоге являются экспериментальными и не влияют на исключение стороннего кода.
  3. Откройте файл в текстовом редакторе. Его содержимое выглядит следующим образом: 
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<Exclusions>
  <packages>
    <package name="package_name"/>
  </packages>
  ...
</Exclusions>
  4. Добавьте пакеты, которые требуется исключить.
    Прим.:
    • Указанные имена пакетов соответствуют всем пакетам с тем же префиксом (они обрабатываются, как если бы за ними следовали символы подстановки). Например, если добавить <package name="com.mycompany.common"/> в качестве исключения, то <package name="com.mycompany.common.action1"/> и <package name="com.mycompany.common.action2"/> также исключаются.
    • На внешние исключения влияют только записи, добавленные в контейнер <packages>. Добавление классов и методов не влияет на исключения.
  5. Сохраните файл и создайте файл IRX для своего исходного кода.

Дальнейшие действия

Обратите внимание на следующие рекомендации по изменению файлов исключений.
  • Создавайте резервные копии измененных файлов исключений. В частности, при обновлении до новой версии Static Analyzer Client Utility загружаются версий файлов исключений без внесенных вами изменений. При обновлении Client Utility загрузите копию измененных файлов исключений и скопируйте их в новый экземпляр Client Utility.
  • Группируйте изменения в одном блоке записей XML, чтобы при необходимости их можно было быстро скопировать и вставить.

Формат:

Описание:

Отправка файла IRX в облако для анализа. По завершении сканирования отправляется уведомление по электронной почте (по адресу электронной почты, связанному с учетной записью, от имени которой был выполнен вход в систему службы анализа). Прим.: При сканировании или создании файла IRX может выдаваться сообщение об обновлении Client Utility до новой версии. См. раздел Совместимость версий облачной службы и Client Utility.

    Прим.: Данный параметр обязателен только при выполнении одного из следующих условий:

    • Ввод команды осуществляется из каталога, не содержащего файлов IRX. В этом случае необходимо использовать параметр -f для указания пути и имени отправляемого файла IRX.
  • -n: введите -n <операция-сканирования>, где <операция-сканирования> - имя операции сканирования, выполняемой в облаке.

    Прим.: Связывание приложения игнорируется в случае подключения к службе ASoC в Bluemix.
Совет: Для всех команд параметры можно указывать в любом порядке.

Примеры:

appscan queue_analysis -f
my_irx.irx
-n my_scan

appscan get_pubkey

Формат:

appscan
get_pubkey -d <путь-сохранения>

Описание:

Загрузка открытого ключа шифрования для использования в системе, не подключенной к Интернету.

Если файл IRX создается в системе, подключенной к Интернету, эту команду можно не использовать, поскольку ключ шифрования автоматически загружается при выполнении команды prepare. Если в системе уже есть ключ шифрования, он при необходимости обновляется во время выполнения команды prepare.

Для того чтобы использовать ключ шифрования в такой системе, необходимо оставить имя файла rsa.pub без изменений и поместить этот файл в каталог config извлеченного файла SAClientUtil_<версия>_<ОС>.zip (где <версия> - текущая версия Client Utility).

Создание файла IRX, содержащего файл оценки (.ozasmt), созданного в продукте IBM Security AppScan Source версии 9.0 или выше.

Прим.: При сканировании или создании файла IRX может выдаваться сообщение об обновлении Client Utility до новой версии. См. раздел Совместимость версий облачной службы и Client Utility.

Необязательные параметры:

  • -d: введите -d <путь-сохранения>, где <путь-сохранения> соответствует каталогу, в котором нужно сохранить файл IRX.
  • -f: введите -f <файл-оценки>, где <файл-оценки> - файл .ozasmt, который требуется упаковать для сканирования. Если файл <файл-оценки> находится в каталоге, отличном от текущего, то данный параметр позволит указать путь к файлу оценки вместе с именем файла.
    Прим.: Данный параметр обязателен только при выполнении одного из следующих условий:
    • Ввод команды осуществляется из каталога, содержащего больше одного файла оценки. Когда в каталоге только один файл оценки, то упаковывается этот файл, если не указан параметр -f.
    • Ввод команды осуществляется из каталога, не содержащего файлов оценки. В этом случае необходимо использовать параметр -f для указания пути и имени файла оценки, который требуется упаковать.
  • -n: введите -n <файл>, где <файл> - имя файла IRX. Имя файла можно указать с расширением .irx или без него. Если расширение не указано, оно автоматически добавляется при создании файла.
Совет: Для всех команд параметры можно указывать в любом порядке.

Примеры:

Из каталога /root/Desktop/my_irx_files можно создать файл IRX, содержащий файл оценки /root/Desktop/my_assessment_files/my_assessment.ozasmt, с помощью следующей команды:

appscan.sh package -f
/root/Desktop/my_assessment_files/my_assessment.ozasmt

Полученный в результате файл IRX будет сохранен в каталоге /root/Desktop/my_irx_files

appscan.sh get_pubkey

Формат:

appscan.sh get_pubkey -d <путь-сохранения>

Описание:

Загрузка открытого ключа шифрования для использования в системе, не подключенной к Интернету.

Если файл IRX создается в системе, подключенной к Интернету, эту команду можно не использовать, поскольку ключ шифрования автоматически загружается при выполнении команды prepare. Если в системе уже есть ключ шифрования, он при необходимости обновляется во время выполнения команды prepare.

Но если файл IRX создается в системе, не подключенной к Интернету, то ключ шифрования можно загрузить с помощью этой команды. После этого ключ шифрования можно скопировать в систему, не подключенную к Интернету, и использовать при создании файла IRX. Для того чтобы использовать ключ шифрования в такой системе, необходимо оставить имя файла rsa.pub без изменений и поместить этот файл в каталог config извлеченного файла SAClientUtil_<версия>_<ОС>.zip (где <версия> - текущая версия Client Utility).

Прим.: Используемый ключ шифрования должен быть действующим. Если создать файл IRX с использованием просроченного ключа шифрования, файл IRX будет отклонен при передаче. Если система подключена к Интернету, еще раз введите команду prepare для автоматического обновления ключа шифрования. Если система не подключена к Интернету, необходимо повторно выполнить команду get_pubkey.

Необязательные параметры:

  • -d: введите -d <путь-сохранения>, где <путь-сохранения> соответствует каталогу, в котором нужно сохранить ключ шифрования. Если не указать этот параметр, ключ сохраняется в каталоге config полученного файла SAClientUtil_<версия>_<ОС>.zip.

appscan.sh api_login

Формат:

appscan.sh api_login -P <пароль-ключа> -u <ИД-ключа> -persist

Описание:

Войдите в службу ASoC в IBM Cloud Marketplace с помощью созданного ключа API.

Важное замечание: Команда api_login применяется только для входа из CLI в службу ASoC в IBM Cloud Marketplace. Для входа в службу ASoC в Bluemix следует использовать команду login.

При входе в службу файл ключа шифрования с именем cli.token сохраняется в папке .appscan вашего каталога home. После истечения срока действия потребуется снова войти в службу, чтобы продолжить работу. Кроме того, при входе в систему можно указать опцию -persist. Если эта опция указана, то Client Utility автоматически выполняет повторную идентификацию после истечения срока действия ключа.

Обязательные параметры:

  • -P: укажите -P <пароль-ключа>, где <пароль-ключа> - это Пароль ключа, созданный при создании ключа API.
  • -u: укажите -u <ИД-ключа>, где <ИД-ключа> - это ИД ключа, созданный при создании ключа API.

Необязательные параметры:

  • -persist: автоматически выполнять повторную идентификацию после истечения срока действия маркера сеанса.

Команды результатов (Linux и macOS)

Применение команд результатов для получения результатов сканирования из службы анализа.

appscan.sh get_result

Формат:

appscan.sh get_result -d <путь-к-файлу> -i <ид-задания> -t <тип>

Описание:

Получение результатов сканирования.

Перед тем как использовать эти команды, необходимо войти в службу анализа.

Обязательные параметры:

  • -i: введите -i <ид-задания>, где <ид-задания> - ИД задания анализа.

Необязательные параметры:

  • -d: введите -d <путь-к-файлу>, где <путь-к-файлу> - полный путь к целевому файлу или имя целевого файла. Если имя файла не указано, присваивается имя на основе имени задания сканирования. Если не указан путь, файл сохраняется в текущем каталоге. Если данный параметр не указан, файл сохраняется в текущем каталоге, а имя файла формируется из имени задания сканирования.
  • -t: введите -t <тип>, где значением <тип> может быть html или zip. Результаты сохраняются как файл HTML или файл .zip, содержащий результаты в формате HTML. Если данный параметр не указан, результаты сохраняются как файл HTML.

    Если результаты сканирования относятся к файлу IRX, созданному с помощью команды package, то добавление параметра -t zip приведет к сохранению результатов с новым файлом .ozasmt, который можно загрузить в продукт AppScan Source версии 9.0 или выше.

Совет: Для всех команд параметры можно указывать в любом порядке.
