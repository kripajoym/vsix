Mise en route de Static Analyzer

Utilisez l'analyse statique pour examiner le code source à la recherche de vulnérabilités de sécurité. Pour ce faire, vous devez télécharger un petit utilitaire. Si vous décompressez l'utilitaire sur votre disque local, vous pouvez utiliser son interface de ligne de commande pour effectuer l'analyse de sécurité dans tous les langages pris en charge. Sinon, vous pouvez exécuter un programme d'installation inclus qui ajoute les plug-in d'analyse statique à Eclipse, Maven, IntelliJ IDEA ou Visual Studio. Vous pouvez examiner les projets Java dans Eclipse et IntelliJ IDEA, et les projets .NET (C#, ASP.NET et VB.NET) dans Visual Studio.

Procédure

  • Pour apprendre à utiliser le service Application Security on Cloud sur IBM Cloud Marketplace, voir http://www.ibm.com/support/knowledgecenter/SSYJJF_1.0.0/ApplicationSecurityonCloud/Welcome.html.
  • Pour apprendre à utiliser Application Security on Cloud for Bluemix, voir https://console.ng.bluemix.net/docs/services/ApplicationSecurityonCloud/index.html.
  • La suite de ce fichier Readme décrit l'installation et l'utilisation de l'utilitaire client Static Analyzer.

Configuration de l'utilitaire client Static Analyzer

Pour une analyse statique, vous devez télécharger un petit utilitaire client. Si vous décompressez l'utilitaire sur votre disque local, vous pouvez utiliser son interface de ligne de commande pour effectuer l'analyse de sécurité. Vous pouvez également exécuter le programme d'installation livré avec lui qui ajoute le plug-in d'analyse statique à votre environnement de génération Maven ou aux environnements de développement intégrés Eclipse, IntelliJ IDEA ou Visual Studio.

Pourquoi et quand exécuter cette tâche

Lorsque vous utilisez l'analyse statique, vous pouvez examiner le code source pour générer un fichier chargé sur le cloud. Le fichier, un fichier IRX (.irx), est examiné à l'aide de l'analyse de trace en vue de détecter les vulnérabilités en matière de sécurité. Lorsque le fichier IRX est généré pour les langages compilés, les fichiers de code d'octet d'application sont convertis en représentation intermédiaire du code. Lorsqu'un fichier IRX est généré pour les langages de script, les fichiers source sont inclus dans le fichier IRX chiffré.

Lorsque vous utilisez les plug-in Eclipse, IntelliJ IDEA ou Visual Studio, la génération et le téléchargement des fichiers IRX sont effectués automatiquement dans l'environnement de développement intégré lorsque vous exécutez des actions d'analyse statique.

Conseil : Lors de l'exécution du programme d'installation, si vous ne pensez pas utiliser immédiatement l'analyse statique dans un environnement de développement intégré qui est installé et que vous prévoyez d'utiliser dans le futur, il peut être judicieux d'y installer le plug-in maintenant pour ne pas à avoir à réexécuter le programme d'installation ultérieurement.
Raccourci : Lorsque vous êtes invité à entrer oui ou non dans le programme d'installation, vous pouvez aussi entrer les abréviations y ou n. Si vous voulez utilisez des abréviations dans un programme d'installation exécuté sur un système dont la langue n'est pas l'anglais, vous devez entrer y ou n.
Important : L'utilitaire client ne doit pas être placé dans un répertoire contenant des caractères non ASCII.

Procédure

Pour configurer l'utilitaire client :

  1. Sélectionnez la plateforme sur laquelle vous prévoyez d'exécuter utilitaire client, puis cliquez sur Télécharger. Cette action télécharge un fichier SAClientUtil_<version>_<os>.zip (où <version> est la version en cours de l'utilitaire client et <os> est le système d'exploitation auquel il est destiné).
    Remarque : Vous pouvez rouvrir le formulaire d'accueil en sélectionnant How do I create an IRX file? dans le formulaire pour sélectionner un fichier IRX.
  2. Décompressez le fichier sur une unité locale.
  3. Si vous prévoyez d'utiliser l'interface de ligne de commande pour la génération ou le téléchargement des fichiers IRX ou la gestion des examens : Ajoutez l'emplacement du répertoire \bin du fichier SAClientUtil_<version>_<os>.zip décompressé à votre variable d'environnement PATH. Si vous n'effectuez pas cette opération, toutes les commandes devront être qualifiées avec le répertoire \bin du fichier SAClientUtil_<version>_<os>.zip extrait à chaque exécution de la commande.
    Important : Si vous prévoyez d'examiner .NET à l'aide de l'interface de ligne de commande ou de Microsoft Visual Studio, vous devez exécuter ces trois fichiers .bat (qui sont dans le fichier SAClientUtil_<version>_<os>.zip décompressé) en tant qu'administrateur :
    • bin\register.bat
    • walalib\dotnet.libs40\RegisterInGAC.bat
    • walalib\dotnet.libs35\RegisterInGAC.bat

    Ces fichiers peuvent être exécutés manuellement. Vous pouvez également exécuter le fichier install-plugins.bat en tant qu'administrateur (si vous n'utilisez que l'interface de ligne de commande, exécutez install-plugins.bat mais entrez non lorsque vous êtes invité à installer le plug-in d'analyse statique).

    Si vous n'exécutez pas ces fichiers en tant qu'administrateur, un message d'avertissement s'affiche dans la console à chaque utilisation de l'interface de ligne de commande. Ce message peut être ignoré si vous examinez des cibles Java.

    Conseil : Après avoir modifié la variable PATH, exécutez appscan version (Windows) ou appscan.sh version (Linux et macOS) dans une invite de commande. Si la version de l'utilitaire client Static Analyzer, le répertoire principal et d'autres informations sont renvoyés, cela signifie que la variable d'environnement PATH est définie correctement.
  4. Si vous prévoyez d'ajouter le plug-in d'analyse statique à Eclipse, Microsoft Visual Studio (sous Windows uniquement), IntelliJ IDEA ou Maven : Recherchez et exécutez install-plugins.bat (Windows) ou install-plugins.sh (Linux) dans l'utilitaire client décompressé. Sous Windows, exécutez le fichier en tant qu'administrateur. Si vous prévoyez d'ajouter le plug-in à Eclipse sous Linux ou macOS, vous devez être connecté à un compte disposant des droits nécessaires pour modifier l'instance Eclipse dans laquelle vous effectuez l'installation.
    1. Par défaut, l'utilitaire client se connecte au service ASoC sur IBM Cloud Marketplace. Si vous préférez vous connecter au service sur IBM Bluemix, entrez oui lorsque vous y êtes invité.
    2. Si vous prévoyez d'examiner du code source dans Eclipse, suivez les invites du programme d'installation. Si vous choisissez d'installer le plug-in Eclipse, vous êtes invité à entrer l'emplacement de votre installation Eclipse. Entrez le chemin complet du répertoire contenant le fichier eclipse.exe (Windows) ou eclipse (Linux et macOS).
    3. Si vous prévoyez d'utiliser Maven pour la génération des fichiers IRX, suivez les invites du programme d'installation.
    4. Si vous prévoyez d'examiner du code source dans IntelliJ IDEA, suivez les invites du programme d'installation. Si vous choisissez d'installer le plug-in IntelliJ IDA, vous êtes invité à entrer l'emplacement de votre répertoire de configuration IntelliJ IDEA. Entrez le chemin complet de ce répertoire.
    5. Windows uniquement : Si vous prévoyez d'examiner du code source dans Visual Studio, suivez les invites du programme d'installation. Si vous choisissez d'installer le plug-in Visual Studio, vous êtes invité à choisir la version de Visual Studio dans laquelle le plug-in doit être installé. Entrez la version correspondant à la version de Visual Studio dans laquelle vous voulez installer le plug-in.
    6. Fermez l'invite de commande en appuyant sur une touche.
    7. Redémarrez chaque environnement de développement intégré dans lequel vous avez installé le plug-in.
    8. Linux uniquement : Dans une invite de commande, accédez au répertoire contenant l'utilitaire client et ajoutez les lignes suivantes à votre fichier de ressources d'interpréteur de commandes (par exemple, .bash_profile ou .shrc) : 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export LD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${LD_LIBRARY_PATH}

      où <client_utility> est le chemin d'accès de l'utilitaire client. Après avoir ajouté ces lignes, rechargez votre interpréteur de commandes avant d'utiliser le plug-in d'analyse statique.

    9. macOS uniquement : Dans une invite de commande, accédez au répertoire qui contient l'utilitaire client et ajoutez les lignes suivantes à votre fichier de ressources d'interpréteur de commandes (par exemple, .bash_profile ou .shrc) : 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export DYLD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${DYLD_LIBRARY_PATH}

      où <client_utility> est le chemin d'accès de l'utilitaire client. Après avoir ajouté ces lignes, rechargez votre interpréteur de commandes avant d'utiliser le plug-in d'analyse statique.

      Important : La définition de la variable d'environnement dans le fichier de ressources d'interpréteur de commandes affecte uniquement les applications lancées sur le terminal. Vous devez donc utiliser la fenêtre de terminal pour lancer les environnements de développement intégrés (IDE) à partir desquels vous souhaitez exécuter l'analyse statique.

Que faire ensuite

Si vous exécutez l'utilitaire client sur un ordinateur placé derrière un proxy, utilisez l'une des méthodes suivantes pour définir le proxy en autorisant l'utilitaire client à se connecter au cloud :

  • Interface de ligne de commande de l'utilitaire client et environnements de développement intégrés pris en charge : Définissez la variable d'environnement système ou globale suivante afin que le proxy soit automatiquement reconnu :
    • Windows : APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>
    • Linux et macOS : APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"

    Où <proxy> est le nom d'hôte du serveur proxy, et <port> est le numéro de port que le serveur proxy utilise.

    A chaque utilisation de l'interface de ligne de commande ou de l'environnement de développement intégré (lancés à partir d'une invite de commande ou d'un terminal), vous pouvez également faire en sorte que l'utilitaire client utilise le proxy en émettant une commande.

    • Windows : set "APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
    • Linux et macOS : export APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
  • Maven : Si vous avez ajouté le plug-in d'analyse statique à Maven, vous pouvez ajouter la propriété à la variable d'environnement MAVEN_OPTS de manière globale, ou l'ajouter à chaque commande.

Suppression de l'utilitaire client Static Analyzer

Cette tâche décrit comment supprimer les plug-in de l'utilitaire client Static Analyzer dans votre environnement de développement intégré (IDE). Des instructions détaillant comment supprimer l'utilitaire client sont également disponibles. Suivez ces instructions uniquement si vous envisagez de supprimer de votre système les composants de l'utilitaire client Static Analyzer. Si vous mettez ces composants à niveau vers une nouvelle version de l'utilitaire client, l'installation du plug-in supprime automatiquement les anciens composants avant d'installer la nouvelle version.

Avant de commencer

Important : Si vous avez installé le plug-in de l'utilitaire client dans un environnement IDE, vous devez exécuter des fichiers ou des commandes à l'emplacement du fichier SAClientUtil_<version>_<os>.zip extrait pour supprimer complètement le plug-in de l'IDE. Il s'agit des fichiers suivants :
  • utilitaire client Static Analyzer Version 6.0.865 et versions antérieures :
    • walalib\dotnet.libs35\UnRegisterInGAC.bat
    • walalib\dotnet.libs40\UnRegisterInGAC.bat
  • Toutes les versions de l'utilitaire client Static Analyzer, notamment la version 6.0.865 et les versions antérieures : Ouvrez une invite de commande en tant qu'administrateur et accédez au répertoire dans lequel vous avez extrait le fichier SAClientUtil_<version>_<os>.zip. Accédez au dossier bin\ puis émettez la commande suivante :
    regsvr32 /u msdia80.dll

Pour pouvoir exécuter ces fichiers et ces commandes, ne supprimez pas le contenu du fichier SAClientUtil_<version>_<os>.zip extrait tant que vous n'avez pas supprimé tous les plug-in de l'IDE.

Vous pouvez exécuter ces fichiers et ces commandes avant ou après avoir suivi les instructions ci-dessous indiquant comment supprimer les plug-in de votre environnement IDE.

Procédure

  • Eclipse :
    1. Dans Eclipse, sélectionnez Aide > Installer un nouveau logiciel dans la barre de menus principale du plan de travail.
    2. Dans la sélection Détails de la boîte de dialogue Installer, cliquez sur le lien permettant d'accéder à la liste des éléments déjà installés.
    3. Dans la liste des logiciels installés, recherchez et sélectionnez Static Analyzer puis sélectionnez Désinstaller.
    4. Dans la boîte de dialogue qui s'affiche, cliquez sur Terminer.
    5. Lorsque vous y êtes invité, redémarrez Eclipse.
  • IntelliJ IDEA :
    1. Dans la barre de menus principale d'IntelliJ IDEA, sélectionnez Fichier > Paramètres.
    2. Dans la boîte de dialogue Paramètres, sélectionnez Plug-in.
    3. Dans la liste qui s'affiche, recherchez et sélectionnez le plug-in Static Analyzer.
    4. Cliquez sur Désinstaller.
    5. Cliquez sur l'option permettant de redémarrer IntelliJ IDEA.
  • Visual Studio :
    1. Dans le menu principal de Visual Studio, sélectionnez Outils > Extensions et mises à jour.
    2. Dans la liste des mises à jour et des extensions installées, recherchez et sélectionnez IBM Static Analyzer.
    3. Cliquez sur Désinstaller puis lorsque vous êtes invité à confirmer la désinstallation d'IBM Static Analyzer, cliquez sur Oui.
    4. Sélectionnez Redémarrer maintenant pour redémarrer Visual Studio ou sélectionnez Fermer si vous souhaitez redémarrer Visual Studio ultérieurement.
  • Utilitaire client : Pour supprimer l'utilitaire client, il vous suffit de supprimer sur votre système le contenu du fichier SAClientUtil_<version>_<os>.zip extrait. Si vous avez ajouté l'emplacement du répertoire \bin de l'utilitaire client à votre variable d'environnement PATH, retirez-le.

Génération d'un fichier IRX à l'aide de l'interface de ligne de commande

Pour pouvoir lancer une analyse de vos fichiers, vous devez générer un fichier IRX que vous soumettez au cloud. Si vous voulez utiliser l'interface de ligne de commande, suivez ces instructions pour créer ce fichier.

Procédure

Pour générer le fichier IRX :

  1. Vous devez ouvrir une nouvelle fenêtre d'invite de commande (n'utilisez pas la fenêtre d'invite de commande qui a été utilisée pour installer l'utilitaire client).
  2. Dans l'invite de commande, émettez une commande pour changer de répertoire et vous placer dans celui qui contient vos fichiers. Voir Langages pris en charge pour plus d'informations sur les types de fichier pris en charge. Si vous examinez les projets Java dans Eclipse, changez de répertoire pour vous placer dans votre espace de travail Eclipse.
    Remarque :
    • Cette étape n'est pas nécessaire si utilisez un fichier de configuration pour générer le fichier IRX. Voir Configuration de la génération de fichier IRX via l'interface de ligne de commande.
    • Avec l'interface de ligne de commande, vous ne pouvez pas examiner de projets individuels dans un espace de travail Eclipse ; vous ne pouvez examiner que l'espace de travail entier. Si vous souhaitez examiner des projets individuels, vous devez générer le fichier IRX dans Eclipse.
    • Les fichiers .dll et .exe doivent être des assemblages .NET valides.
  3. Exécutez la commande appscan prepare (Windows) ou appscan.sh prepare (Linux et macOS). Cette commande est utilisée avec des options répertoriées dans les rubriques Commandes de configuration (Windows) et Commandes de configuration (Linux et macOS). Par exemple, vous pouvez utiliser les options -d et -n pour indiquer un répertoire spécifique dans lequel sauvegarder le fichier IRX et sauvegarder le fichier sous un nom de fichier particulier. De plus, vous pouvez exécuter la commande avec un fichier de configuration afin de spécifier des informations supplémentaires permettant de générer un fichier IRX complet (voir Configuration de la génération de fichier IRX via l'interface de ligne de commande).
    Conseil : Par défaut, le code tiers Java et .NET n'est pas examiné au cours de la génération du fichier IRX. Vous pouvez modifier les paramètres d'exclusion de code tiers en suivant les instructions de la section Gestion des exclusions Java et .NET tierces. Pour inclure le code tiers, spécifiez l'option -t ou --thirdParty lorsque vous entrez la commande prepare.

    Si vous êtes développeur de code tiers qui devrait normalement être inclus dans un examen, vous devez utiliser ce paramètre pour l'inclure.

    Remarque : Lorsque vous examinez du code source ou que vous générez un fichier IRX, il est possible que vous receviez un message sur l'installation de la dernière version de l'utilitaire client. Voir Compatibilité de la version de l'utilitaire client et de la version du service de cloud.
  4. Une fois la commande émise, l'utilitaire client rassemble des informations pour tous les fichiers pris en charge dans le répertoire depuis lequel la commande a été passé, et dans tous ses sous-répertoires. Un fichier IRX est créé dans le répertoire. Le nom de ce fichier est basé sur le répertoire à partir duquel la commande a été exécutée et comprend un horodatage (si vous n'avez pas utilisé d'option de configuration pour spécifier un nom de fichier différent).
  5. Soumettez le fichier IRX au cloud pour analyse. Pour ce faire, exécutez la commande appscan queue_analysis (Windows) ou la commande appscan.sh queue_analysis (Linux et macOS). Cette commande est utilisée avec des options répertoriées dans les rubriques Commandes d'analyse (Windows) ou Commandes d'analyse (Linux et macOS).

Génération d'un fichier IRX pour un projet Maven

Pour pouvoir lancer une analyse de vos fichiers, vous devez générer un fichier IRX que vous soumettez au cloud. Si vous voulez examiner un projet Maven, suivez les instructions ci-après pour créer ce fichier.

Procédure

Pour générer le fichier IRX :

  1. Vous devez ouvrir une nouvelle fenêtre d'invite de commande (n'utilisez pas la fenêtre d'invite de commande qui a été utilisée pour installer l'utilitaire client).
  2. A l'invite de commande, exécutez une commande pour changer le répertoire pour l'emplacement contenant votre projet Maven (le dossier contenant le fichier pom.xml).
  3. Emettez l'une de ces commandes :
    • Si votre projet Maven n'est pas encore généré, entrez la commande mvn package com.ibm.appscan:appscan-maven-plugin:prepare. Une fois la commande émise, un fichier IRX est créé dans le répertoire target du projet Maven. Le nom de ce fichier est basé sur le projet Maven de niveau supérieur (le nom du projet dans le fichier pom.xml), sauf si vous avez utilisé une option de configuration pour spécifier un autre nom de fichier.
    • Si votre projet Maven est déjà généré, entrez mvn com.ibm.appscan:appscan-maven-plugin:prepare. Une fois la commande émise, un fichier IRX est créé dans le répertoire target du projet Maven. Le nom de ce fichier est basé sur le projet Maven de niveau supérieur (le nom du projet dans le fichier pom.xml), sauf si vous avez utilisé une option de configuration pour spécifier un autre nom de fichier.
      Remarque : Si votre projet Maven possède des dépendances de compilation sur un autre projet dans la même génération de réacteur, un bogue Maven existant (https://issues.apache.org/jira/browse/MNG-3283) entraîne l'échec de la commande mvn com.ibm.appscan:appscan-maven-plugin:prepare. Pour contourner ce problème, effectuez l'une des opérations suivantes :
      • Exécutez la commande mvn install à la place de mvn package avant d'exécuter la commande prepare de l'utilitaire client.
      • Exécutez la commande mvn package et la commande prepare de l'utilitaire client en même temps (entrez mvn package com.ibm.appscan:appscan-maven-plugin:prepare).
    • Si vous voulez créer le fichier IRX dans un autre répertoire, ou avec un nom de fichier différent, émettez les commandes avec l'option -Doutput :

      Sous Windows, entrez cette commande :

      mvn
package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<emplacement_sortie>\<nom_fichier>.irx

      Sous Linux et macOS, exécutez la commande suivante :

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<emplacement_sortie>/<nom_fichier>.irx

      où <emplacement_sortie> est le répertoire dans lequel sauvegarder le fichier, et nom_fichier est le nom du fichier.

    • L'utilitaire client inclut un compilateur JSP Tomcat. Ce compilateur JSP est utilisé par défaut au cours de la génération du fichier IRX. Si vous générez un fichier IRX pour un fichier WAR ou EAR et voulez utiliser un compilateur JSP différent, émettez la commande avec l'option -DjspCompiler. Exemple :
      mvn com.ibm.appscan:appscan-maven-plugin:prepare -DjspCompiler=<compilateur_jsp>

      où <compilateur_jsp> est le chemin d'accès au compilateur JSP à utiliser (spécifiez le répertoire racine du serveur).

    Conseil : Par défaut, le code tiers n'est pas examiné lors de la génération du fichier IRX. Vous pouvez modifier les paramètres d'exclusion de code tiers en suivant les instructions de la section Gestion des exclusions Java et .NET tierces. Pour inclure le code tiers, entrez la commande avec l'option -DthirdParty.

    Si vous êtes développeur de code tiers qui devrait normalement être inclus dans un examen, vous devez utiliser ce paramètre pour l'inclure.

    Remarque : Lorsque vous examinez du code source ou que vous générez un fichier IRX, il est possible que vous receviez un message sur l'installation de la dernière version de l'utilitaire client. Voir Compatibilité de la version de l'utilitaire client et de la version du service de cloud.
  4. Soumettez le fichier IRX au cloud pour analyse. Pour ce faire, exécutez la commande appscan queue_analysis (Windows) ou la commande appscan.sh queue_analysis (Linux et macOS). Cette commande est utilisée avec des options répertoriées dans les rubriques Commandes d'analyse (Windows) ou Commandes d'analyse (Linux et macOS).

Utilisation des fichier des configuration

Un fichier de configuration peut être utilisé pour indiquer les cibles à inclure ou exclure lors de la génération d'un fichier IRX. Vous pouvez aussi spécifier des répertoires qui contiennent des artefacts de génération, et vous pouvez spécifier des informations de génération à utiliser au cours de la génération du fichier IRX (par exemple, vous pouvez utiliser ce paramètre pour spécifier un logiciel JDK ou un compilateur particulier).

Lorsque vous exécutez la commande appscan prepare (Windows) ou appscan.sh prepare (Linux et macOS) pour générer un fichier IRX, l'utilitaire client Static Analyzer recherche automatiquement un fichier appscan-config.xml dans le répertoire en cours. Si le fichier est trouvé, il est utilisé automatiquement.

Si votre fichier de configuration se trouve dans un autre répertoire ou avec un autre nom de fichier, vous pouvez utiliser le fichier en incluant l'option -c lorsque vous exécutez la commande appscan prepare ou appscan.sh prepare. Dans ce cas, vous pouvez exécuter appscan prepare -c <fichier_configuration> sous (Windows) ou appscan.sh prepare -c <fichier_configuration> sous (Linux et macOS), où <fichier_configuration> est le chemin complet et le nom du fichier de configuration.

Modèle de fichier de configuration et paramètres

Lors de la préparation du fichier de configuration, utilisez ce modèle :

<Configuration>
  <Targets>
    <Target outputs-only="true/false" path="chemin_cible_examen">
      <CustomBuildInfo build_info="info"/>
      <Include>string_pattern</Include>
      <Exclude>string_pattern</Exclude>
    </Target>
  </Targets>
</Configuration>

Où vous pouvez utiliser l'élément Targets pour indiquer les cibles à examiner lors de la génération du fichier IRX. Les attributs de cible (Target) et les éléments enfant sont utilisés comme suit :

  • L'attribut outputs-only de l'élément Target est utilisé uniquement pour examiner un répertoire et forcer l'utilitaire client à ne trouver que les fichiers de sortie de génération (comme les fichiers .jar, .war et .class). Par défaut, cet attribut a pour valeur false. Cela signifie que l'utilitaire client examine le répertoire pour déterminer s'il est une cible (comme un serveur d'applications ou un espace de travail Eclipse) ou s'il contient des éléments tels que des scripts de génération, des fichiers .pom Maven et des fichiers make. Si vous voulez que la cible de l'examen soit traitée comme un simple répertoire, et si vous n'avez localisé que les fichiers de sortie, spécifiez outputs-only="true" dans l'élément Target.

    Par exemple, si vous spécifiez <Target outputs-only="false" path="C:\Tomcat">, l'utilitaire client suppose que la cible est votre serveur d'applications Tomcat et recherche ses fichiers .war déployés. Toutefois, si vous spécifiez <Target outputs-only="true" path="C:\Tomcat">, l'utilitaire client traite cet emplacement comme un répertoire et localise tous les fichiers de sortie de génération qu'il contient.

  • Utilisez l'attribut path de l'élément Target pour spécifier le chemin d'accès à une cible d'examen ou un répertoire de cibles d'examen (<scan_target_path>). Lorsqu'un répertoire est spécifié, tous ses sous-répertoires sont inclus lors de l'examen.

    Par exemple, si vous spécifiez <Target outputs-only="false" path="C:\WebSphere\AppServer85\profiles\AppSrv01">, l'utilitaire client localise tous les fichiers .ear déployés dans le profil AppSrv01. Si vous spécifiez <Target outputs-only="false" path="C:\WebSphere\AppServer85">, l'utilitaire client localise tous les fichiers .ear déployés dans tous les profils.

  • Vous pouvez aussi spécifier les sous-éléments Target facultatifs suivants :
    • Vous pouvez utiliser l'élément CustomBuildInfo pour définir les attributs. A l'aide de cette partie du modèle,
      <CustomBuildInfo build_info="info"/>
      vous pouvez spécifier les information de génération en fonction de la langue cible. Pour certaines langues, vous pouvez définir plusieurs attributs. Par exemple, vous pouvez définir <CustomBuildInfo build_info_1="info_1" build_info_2="info_2" build_info_3="info_3"/> en fonction de la langue cible.
      • Pour Java, vous pouvez définir ces attributs :
        <CustomBuildInfo additional_classpath="dependency_path"
  jdk_path="JDK_path" jsp_compiler="JSP_compiler_path"/>
        • Spécifiez des chemins d'accès aux classes supplémentaires avec l'attribut additional_classpath. Sous Windows, séparez plusieurs chemins d'accès aux classes par un point-virgule. Sur Linux et macOS, séparez les différents chemins d'accès aux classes avec deux points.
        • Utilisez l'attribut jdk_path pour spécifier le chemin d'accès à votre installation JDK.
        • Utilisez l'attribut jsp_compiler pour spécifier le chemin d'accès à votre compilateur JSP, par exemple jsp_compiler="C:\Tomcat".
      • Pour .NET (Windows uniquement), vous pouvez définir ces attributs :
        <CustomBuildInfo reference="assembly_reference"
  configuration="build_configuration"/>
        • Ajoutez des références d'assemblage à l'aide de l'attribut reference. Séparez les différentes références par des points-virgules.
        • Incluez une configuration de génération pour la détection de la solution Visual Studio à l'aide de l'attribut configuration.
      • Pour C/C++ (Windows uniquement), vous pouvez définir ces attributs :
        <CustomBuildInfo configuration="build_configuration"
  include_paths="include_directories" macros="macros" compiler_opts=/>
        • Utilisez l'attribut configuration pour inclure une configuration d'une génération.
        • Spécifiez des chemins d'inclusion en utilisant l'attribut include_paths. Séparez les différents chemins d'inclusion par des points-virgules.
        • Incluez des macros à l'aide de l'attribut macros. Séparez les différentes macros par des points-virgules.
        • Spécifiez des options de compilation à l'aide de l'attribut compiler_opts. Séparez les différentes options par des points-virgules.
    Remarque : Les sous-cibles héritent des valeurs définies par ces attributs. Par exemple, si votre cible est un fichier EAR qui inclut des sous-cibles WAR et JAR, on suppose que WAR et JAR possèdent les mêmes valeurs que celles définies pour le fichier EAR qui utilise ces attributs.
  • Utilisez l'élément enfant Include pour spécifier des masques de fichiers (<string_pattern>) à inclure lors de la génération du fichier IRX. Le comportement Include dépend du type de cible, comme expliqué à la section Comportement d'inclusion et d'exclusion des cibles. Pour spécifier plusieurs masques include, ajoutez chaque masque dans sa propre balise <Include></Include>. Par exemple :
    <Include>string_pattern_1</Include>
<Include>string_pattern_2</Include>
    Remarque : Si vous spécifiez des masques include et exclude qui entrent en conflit, les masques exclude ont la priorité.
  • Utilisez l'élément enfant Exclude pour spécifier des masques de fichiers à exclure lors de la génération du fichier IRX. Le comportement Exclude dépend du type de cible, comme expliqué à la section Comportement d'inclusion et d'exclusion des cibles. Pour spécifier plusieurs masques exclude, ajoutez chaque masque dans sa propre balise <Exclude></Exclude>.
    Remarque : Si vous spécifiez des masques include et exclude qui entrent en conflit, les masques exclude ont la priorité.

  • test/*.jar : inclut ou exclut tous les fichiers .jar qui se trouvent dans le répertoire test/, mais pas ceux qui se trouvent dans ses sous-répertoires.
serveur Web (voir Configuration requise pour en savoir plus sur les serveurs Web pris en charge) Si votre cible est un serveur Web, vous pouvez inclure ou exclure le nom de fichier de toute application qui est déployée sur le serveur.
  • JSPWiki.war
  • MyApp.ear
  • *.war (Lorsque la cible est un serveur Tomcat)
.sln (.NET) Si la cible est un fichier de la solution Visual Studio qui contient des projets .NET, vous pouvez inclure ou exclure le nom de fichier d'un assemblage .NET produit par un projet dans la solution.
  • *\MyLib.dll
  • *\MyApp.exe
.sln (C/C++) Si la cible est un fichier de la solution Visual Studio qui contient des projets C/C++, vous pouvez inclure ou exclure le nom de fichier d'un projet.
  • *\myProject.vcxproj

Exemple de fichier de configuration

<Configuration>
  <Targets>
    <Target path="C:\test\MyApplication.ear">
      <CustomBuildInfo additional_classpath="C:\java\lib;C:\java2\lib"
        jdk_path="C:\Program Files\Java\jdk1.7" jsp_compiler="C:\Tomcat"/>
      <Include>ShoppingCart.war</Include>
      <Include>*Account.jar</Include>
      <Exclude>thirdPartyJars/</Exclude>
      <Exclude>*test*</Exclude>
    </Target>
  </Targets>
</Configuration>
Dans ce fichier de configuration :
  • C:\test\MyApplication.ear est la cible.
  • Les dépendances de fichier de classe nécessaires pour la compilation sont recherchées dans C:\java\lib et C:\java2\lib.
  • C:\Program Files\Java\jdk1.7 est utilisé pour localiser les dépendances.
  • Le compilateur C:\Tomcat est utilisé pour la compilation JSP à la place de l'application Apache Tomcat Version 7 incluse dans l'utilitaire client Static Analyzer.
  • Lorsque le fichier IRX est généré, le fichier ShoppingCart.war est inclus, ainsi que tous les fichiers dont le nom se termine par *Account.jar. Tous les éléments du répertoire thirdPartyJars/ dans le fichier EAR sont exclus, ainsi que tous les fichiers dont le nom comporte la chaîne test. En cas de conflits, les masques exclude ont la priorité. Par exemple, si le fichier EAR contient un fichier testCustomerAccount.jar, ce fichier est exclus car le masque test exclude a la priorité sur le masque *Account.jar include.

Des erreurs ou des avertissements se produisent lors de la génération d'un fichier IRX d'analyse statique

Symptômes

Lors de la génération d'un fichier IRX, il est possible que vous receviez des erreurs ou des avertissements ou que la génération du fichier IRX échoue. Lorsque la génération du fichier IRX échoue, la chaîne _failed est ajoutée à la fin du nom du fichier généré (par exemple my_irx_file_failed.irx). Lorsque cela se produit, plusieurs possibilités sont à votre disposition pour traiter les incidents relatifs à la génération des fichiers IRX.

Résolution de l'incident

  • Utilisation d'un fichier de configuration pour traiter les incidents liés à la génération d'un fichier IRX

    Si vous générez un fichier IRX sans utiliser de fichier de configuration et si l'opération n'aboutit pas (ou si elle aboutit avec une erreur), l'utilitaire client Static Analyzer génère un fichier de configuration temporaire que vous pouvez utiliser pour les examens suivants. Ce fichier ressemble au modèle de fichier de configuration dans le sens où il contient un élément Targets avec les informations connues concernant vos cibles. En revanche, il contient également un élément <TargetSettings> que vous pouvez modifier en ajoutant des informations qui permettront à l'utilitaire client de générer un fichier IRX complet. Ce fichier s'appelle appscan-config.xml et est sauvegardé dans le répertoire de travail. Cela signifie que le fichier est automatiquement utilisé pour l'examen suivant.

    Par exemple, si un fichier IRX est généré pour une cible contenant des dépendances externes qui sont introuvables, il est possible que vous receviez un message du type :

    C:\myTargetDirectory>appscan prepare
An IRX file was created, but it may be incomplete.
C:\myTargetDirectory\myTarget.jar requires the following packages:
      org.apache.jasper.*
An appscan-config.xml file was created in: C:\myTargetDirectory\appscan-config.xml
Please update the file and rerun the command.

    Le fichier appscan-config.xml généré ressemble à ceci :

    <Configuration>
  <Targets>
    <Target outputs-only="true" path="C:\myTargetDirectory"/>
  </Targets>
  <Settings/>
  <TargetSettings>
    <Target path="C:\myTargetDirectory\myTarget.jar"/>
      <!--Please add the path of the following dependencies to the attribute "additional_classpath" below, separated by semi-colon.
			  org.apache.jasper.*-->
      <CustomBuildInfo additional_classpath=""/>
    </Target>
  </TargetSettings>
</Configuration>

    Pour résoudre le problème, mettez à jour <CustomBuildInfo additional_classpath=""/> pour inclure les classes org.apache.jasper.* dans le chemin d'accès aux classes, sauvegardez le fichier appscan-config.xml, puis réexécutez la commande appscan prepare (Windows) ou appscan.sh prepare (Linux et macOS). Lorsque vous spécifiez le chemin d'accès aux fichiers JAR, la syntaxe standard du chemin d'accès aux classes Java est admise (par exemple, les caractères génériques sont acceptés).

    Si vous rencontrez des problèmes similaires alors que vous utilisez déjà un fichier de configuration, vous recevez le message signalant les problèmes. Utilisez ce message pour mettre à jour votre fichier de configuration existant, puis réexécutez la commande.

    Ce fichier de configuration est décrit à la rubrique Configuration de la génération de fichier IRX via l'interface de ligne de commande.

  • Génération de fichiers IRX en mode débogage

    Si vous générez un fichier IRX à partir de l'interface de ligne de commande, vous pouvez exécuter la commande appscan prepare (Windows) ou appscan.sh prepare (Linux et macOS) avec une option de débogage. Davantage de fichiers journaux sont alors générés.

  • Fichiers journaux associés à la génération des fichiers IRX

    Pour afficher les erreurs ou les avertissements, vous pouvez consulter les fichiers journaux qui sont également générés. Les fichiers journaux sont sauvegardés dans un fichier d'archive .zip. Par défaut, le fichier est sauvegardé dans le même emplacement que le fichier IRX ; vous pouvez indiquer un autre emplacement en utilisant l'option -l de la commande appscan prepare (Windows) ou appscan.sh prepare (Linux et macOS). Le nom du fichier .zip est basé sur le nom du fichier IRX qui est généré.

L'analyse de la sécurité a échoué avec l'erreur L'application avec l'ID x n'existe pas

Symptômes

L'exécution d'une analyse qui utilise le plug-in Jenkins ou IDE ou l'interface de ligne de commande a échoué avec cette erreur :

L'application avec l'ID x n'existe pas
L'exécution de l'analyse de la sécurité a échoué : des problèmes se sont produits lors de l'envoi du fichier IRX au service d'analyse.

Causes

Auparavant, les applications Application Security on Cloud étaient associées en fonction de leur app_id. Une mise à jour récente a remplacé cette association app_id par guid. Pour l'essentiel, le service ASoC vous invitera à réassocier des applications en fonction de cette mise à jour. Pour les plug-ins Jenkins et IDE et l'interface de ligne de commande, vous devez exécuter des étapes supplémentaires.

Résolution de l'incident

Pour le plug-in Jenkins, réenregistrez la configuration du plug-in Jenkins en fonction des instructions sous Ajout d'une analyse de sécurité à votre serveur d'automatisation Jenkins et réessayez d'effectuer l'analyse.

Pour le plug-in IDE, lorsque vous y êtes invité, resélectionnez la même application à analyser e t cliquez sur OK.

Pour l'interface de ligne de commande, le nouveau guid remplace app_id. Le guid est un identificateur unique comportant 32 caractères. Par exemple, 33c3281a-6d7c-e711-b334-002590731623. A l'aide de la commande appscan list_apps, déterminez le guid de chaque application et remplacez app_id par ce GUID dans la commande queue_analysis dans les scripts ou la commande de ligne.

Utilisation des annotations

L'utilisation des annotations est décrite ci-dessous :

  1. Les annotations sont prises en charge par défaut. Le fichier .jar d'annotations réside dans le répertoire \annotation du fichier SAClientUtil_<version>_<os>.zip décompressé.
  2. Si vous examinez des classes précompilées ou des fichiers .war ou .jar, ajoutez com.ibm.security.annotation.jar au projet Java qui contient les sources annotées et régénérez-le.
  3. Si vous effectuez des examens dans un environnement de développement intégré (IDE) où le plug-in d'analyse statique est installé, ajoutez com.ibm.security.annotation.jar au chemin d'accès aux classes Java.
  4. Si vous utilisez l'interface de ligne de commande de l'analyse statique, créez un fichier de configuration et définissez-y CustomBuildInfo additional_classpath="dependency_path" afin que dependency_path inclue l'emplacement du fichier com.ibm.security.annotation.jar.

Les annotations peuvent être ajoutées au code source avant l'examen, ou après l'examen et pendant le triage pour identifier et éliminer les faux positifs. Si vous effectuez des examens dans un environnement de développement intégré où le plug-in d'analyse statique est installé, un bouton M'aider à résoudre ce problème est disponible dans le rapport d'examen. Lorsque vous sélectionnez ce bouton, les points au niveau desquels des annotations peuvent être ajoutées sont mis en évidence.

Les annotations permettent d'insérer directement des connaissances dans le code source sous la forme d'annotations de sécurité. Les annotations pouvant être utilisées pour déclarer fiables des portions de code, elles doivent être utilisées avec beaucoup de précautions. Elles ne doivent pas être utilisées pour du code dans lequel des vulnérabilités de sécurité doivent être recherchées. Si vous utilisez des annotations, un analyste de sécurité peut choisir de les ignorer en désactivant la fonction dans le fichier irgen-engine.apsettings qui se trouve dans le répertoire \config du fichier SAClientUtil_<version>_<os>.zip décompressé. Dans ce fichier, repérez le paramètre suivant :

<Setting 
  name="process_security_annotations"
  value="true"
  default_value="true"
  description="When turned on, method annotations will be processed - used by DotNet and Java LSPs"
  display_name="Process Security Annotations"
  type="bool"
/>

Pour désactiver la fonction, remplacez value="true" par value="false".

Remarque : Si le paramètre est défini sur value="false", définissez-le sur value="true" si vous souhaitez utiliser des annotations.

@ValidatorMethod

Les valideurs sont des méthodes qui vérifient les données en entrée et renvoient généralement une valeur booléenne qui indique si l'entrée est valide. Plutôt que d'accepter ou de rejeter les entrées utilisateur à l'aide de valideurs, vous pouvez les modifier pour leur donner un format acceptable. Ces méthodes s'appellent des nettoyeurs.

L'annotation @ValidatorMethod permet d'identifier toutes les méthodes de validation et de nettoyage dans le code source de l'application. Au cours des examens d'analyse statique, ces informations sont utilisées pour supprimer les flux de données qui transitent par ces méthodes car ces données sont alors considérées comme fiables.

Remarque : Il n'est actuellement pas possible de spécifier quels paramètres de la méthode annotée doivent être considérés comme validés. Pendant les examens d'analyse statique, tous les paramètres d'entrée sont considérés comme validés.

@SuppressSecurityTrace

Toutes les traces qui passent par une méthode marquée avec cette annotation sont supprimées. Cette annotation est utile lorsqu'un groupe de traces est identifié comme des faux positifs, ou est moins important ou intéressant que d'autres. Vous pouvez l'utiliser pour éliminer ces traces, ou pour les masquer afin de réduire l'encombrement.

@CallbackMethod

Cette annotation est utilisée pour identifier des rappels ou les points d'entrée d'une application. Tous les arguments sont considérés comme tachés.

Les valideurs sont des méthodes qui vérifient les données en entrée et renvoient généralement une valeur booléenne qui indique si l'entrée est valide. Plutôt que d'accepter ou de rejeter les entrées utilisateur à l'aide de valideurs, vous pouvez les modifier pour leur donner un format acceptable. Ces méthodes s'appellent des nettoyeurs.

L'attribut [ValidatorMethod] permet d'identifier toutes les méthodes de validation et de nettoyage dans le code source de l'application. Au cours des examens d'analyse statique, ces informations sont utilisées pour supprimer les flux de données qui transitent par ces méthodes car ces données sont alors considérées comme fiables.

Remarque : Il n'est actuellement pas possible de spécifier quels paramètres de la méthode annotée doivent être considérés comme validés. Pendant les examens d'analyse statique, tous les paramètres d'entrée sont considérés comme validés.

[SuppressSecurityTrace] ou [SuppressSecurityTrace()]

Toutes les traces qui passent par une méthode marquée avec cet attribut sont supprimées. Cette annotation est utile lorsqu'un groupe de traces est identifié comme des faux positifs, ou est moins important ou intéressant que d'autres. Vous pouvez l'utiliser pour éliminer ces traces, ou pour les masquer afin de réduire l'encombrement.

[CallbackMethod] ou [CallbackMethod()]

Cet attribut est utilisé pour identifier des rappels ou les points d'entrée d'une application. Tous les arguments sont considérés comme tachés.

Configuration des expressions include PHP non résolues et des classes PHP inconnues

La génération d'un fichier IRX pour PHP déclenche la création d'un fichier permettant de configurer la génération IRX lors de l'examen suivant des fichiers.

Le fichier de configuration, php-scan-config.xml, est généré à l'emplacement à partir duquel vous avez émis la commande prepare. Il inclut deux sections, <UserInputs> et <Php-Unknown-Configuration>.

La section <UserInputs> comporte des paramètres de configuration pouvant être modifiés pour l'examen PHP. Cette section contient par défaut des valeurs exemple. Vous pouvez ajouter des valeurs supplémentaires en utilisant un point virgule (;). Les paramètres de cette section sont les suivants :

  • <RegularExpression> : Fonction avancée permettant d'indiquer une expression régulière comme texte à rechercher.
  • <ReplaceText> : Ce paramètre est utilisé pour l'opération de recherche/remplacement de texte simple. Le texte à rechercher est utilisé en l'état et est remplacé par le texte de remplacement.
  • <Function> : Ce paramètre est utilisé lorsqu'il est nécessaire de remplacer un appel de fonction. Le texte à rechercher doit correspondre au nom de la fonction sans parenthèse. Ce texte sera augmenté pour rechercher un appel de fonction portant le nom spécifié suivi de parenthèses et renverra tout élément correspondant à celui entre parenthèses.
  • <SimpleText> : Texte de remplacement simplifié. Paramètre à utiliser si le texte est une chaîne unique.
  • <SimpleRegEx> : Expression régulière de remplacement simplifiée. Paramètre à utiliser si l'expression est une chaîne unique.
  • <AdditionalIncludePaths> : Utilisez ce paramètre pour ajouter des chemins include contenant des fichiers référencés.

La section <Php-Unknown-Configuration> contient toutes les expressions include non résolues et les classes inconnues détectées lors du dernier examen. Indiquez ces informations dans la section <UserInputs>.

  • <Summary> : Récapitule le nombre d'expressions include inconnues (<NumberOfUnknownIncludeExprs>) et de classes inconnues (<NumberOfUnknownClasses>) du dernier examen.
  • <SourceFile> : Toutes les expressions include et les classes inconnues sont répertoriées par ordre alphabétique du nom de fichier source. Pour chaque fichier source, le chemin du fichier (<FilePath>) inclut le chemin absolu de la source, la classe inconnue (<UnknownClass>) et l'expression include inconnue (<UnknownIncludes>).

Gestion des exclusions Java et .NET tierces

Par défaut, le code tiers Java et .NET n'est pas examiné lors de la génération de fichier IRX. Vous pouvez gérer le code tiers exclu en suivant les instructions de cette rubrique.

Pourquoi et quand exécuter cette tâche

Lorsque vous générez un fichier IRX pour votre code, le code tiers est exclu de sorte que votre évaluation ne contienne ni informations ni résultats concernant le code que vous ne souhaitez pas analyser. Pour modifier la liste des éléments à exclure, suivez la procédure décrite dans cette rubrique. Si, au contraire, vous souhaitez inclure un code tiers lors de l'examen, consultez les rubriques suivantes :

  • Génération d'un fichier IRX à l'aide de l'interface de ligne de commande
  • Génération d'un fichier IRX pour un projet Maven
  • Examen dans les environnements de développement intégrés

Procédure

  1. Recherchez le répertoire config du fichier SAClientUtil_<version>_<os>.zip extrait (où <version> correspond à la version actuelle de l'utilitaire client).
  2. Dans ce répertoire, recherchez le fichier dot_net.exclusions (.NET) ou java.exclusions (Java).
    Remarque : D'autres fichiers .exclusions de ce répertoire sont expérimentaux et leur modification n'aura aucune conséquence sur les exclusions tierces.
  3. Ouvrez le fichier dans un éditeur de texte vous permettant de voir que le format XML est utilisé : 
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<Exclusions>
  <packages>
    <package name="package_name"/>
  </packages>
  ...
</Exclusions>
  4. Editez le fichier afin d'ajouter les packages à exclure.
    Remarque :
    • Les noms de package spécifiés correspondront à tous les packages ayant le même préfixe (ils seront traités comme s'ils étaient suivis d'un caractère générique). Par exemple, si vous ajoutez <package name="com.mycompany.common"/> comme exclusion, <package name="com.mycompany.common.action1"/> et <package name="com.mycompany.common.action2"/> seront exclus.
    • Seules les entrées ajoutées au conteneur <packages> affectent les exclusions tierces. L'ajout de classes et de méthodes n'aura aucune conséquence sur les exclusions.
  5. Sauvegardez le ficher puis générez un fichier IRX pour votre code source.

Que faire ensuite

Si vous modifiez les fichiers d'exclusion, vous devez appliquer les meilleures pratiques suivantes :
  • Conservez une copie de sauvegarde des fichiers d'exclusion modifiés. Gardez à l'esprit que si vous effectuez une mise à niveau vers une nouvelle version de l'utilitaire client Static Analyzer, les fichiers d'exclusion téléchargés n'incluront pas vos mises à jour. Lors de la mise à jour de l'utilitaire client, conservez une copie des fichiers d'exclusion modifiés puis copiez-les dans le nouvel utilitaire client.
  • Conservez les modifications dans un seul bloc d'entrées XML afin que vous puissiez facilement les copier et les coller, si nécessaire.

Commandes de résultats (Linux et macOS)

Utilisez les commandes de résultats pour extraire les résultats d'examen du service d'analyse.

appscan.sh get_result

Syntaxe :

appscan.sh get_result -d <chemin_fichier> -i <id_travail> -t <type>

Description :

Procédez à l'extraction des résultats d'examen.

Avant d'utiliser ces commandes, vous devez vous connecter au service d'analyse.

Options requises :

  • -i : spécifiez -i <job_id>, où <job_id> est l'ID du travail d'analyse.

Options facultatives :

  • -d : spécifiez -d <file_path>, où <file_path> est le chemin d'accès complet au fichier de destination et/ou son nom. Si aucun nom de fichier n'est spécifié, le nom de fichier dépend du nom du travail d'examen. Si aucun chemin n'est spécifié, le fichier est sauvegardé dans le répertoire de travail. Si cette option n'est pas incluse, le fichier est sauvegardé dans le répertoire de travail avec un nom de fichier reposant sur le nom du travail d'examen.
  • -t : spécifiez -t <type>, où <type> est html ou zip. Les résultats sont sauvegardés dans un fichier HTML ou dans un fichier .zip contenant les résultats au format HTML. Si cette option n'est pas incluse, les résultats sont sauvegardés dans un fichier HTML.

    Si les résultats de l'examen concernent un fichier IRX qui a été généré par la commande package, l'option -t zip sauvegarde les résultats avec un nouveau fichier .ozasmt susceptible d'être chargé dans votre produit AppScan Source version 9.0 ou ultérieure.

Conseil : Pour toutes les commandes, les options peuvent être utilisées dans n'importe quel ordre.
