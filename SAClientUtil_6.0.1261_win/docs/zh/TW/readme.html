開始使用 Static Analyzer

使用靜態分析來掃描原始碼是否有安全漏洞。為了達到此目標，需要下載一個小型公用程式。 如果您將公用程式解壓縮到本端磁碟，則可以使用其指令行介面 (CLI)，在所有支援的語言上執行安全分析。或者，您可以執行內含的安裝程式，將靜態分析外掛程式新增至 Eclipse、Maven、IntelliJ IDEA 或 Visual Studio。在 Eclipse 和 IntelliJ IDEA 中， 您可以掃描 Java 專案 - 而在 Visual Studio 中，您可以掃描 .NET（C#、ASP.NET、VB.NET）。

設定 Static Analyzer 用戶端公用程式

若為靜態分析，您要下載小型的 用戶端公用程式。如果您將公用程式解壓縮到本端磁碟，則可以使用其指令行介面 (CLI) 來執行安全分析。或者，您可以執行內含的安裝程式，將靜態分析外掛程式新增至 Maven 建置環境 - 或 Eclipse、IntelliJ IDEA 或 Visual Studio 整合開發環境 (IDE)。

關於這項作業

當您使用靜態分析時，您會掃描原始碼以產生要上傳至雲端的檔案。該檔案是個加密的 IRX (.irx) 檔，其會利用追蹤分析加以掃描來找出安全漏洞。如果產生編譯語言的 IRX 檔，應用程式位元組碼檔案會轉換成程式碼的 中間表示法。如果產生 Scripting 語言的 IRX 檔，原始檔會併入加密的 IRX 檔中。

當您使用 Eclipse、IntelliJ IDEA 或 Visual Studio 外掛程式時，會在使用靜態分析動作時，自動從 IDE 產生和上傳 IRX 檔案。

提示： 執行安裝程式時，如果您不要在特定 IDE 中使用靜態分析，但是已安裝該 IDE 且預期將來要使用它，則您現在可能需要將外掛程式安裝到該 IDE，這樣稍後就不需要再重新執行安裝程式。
捷徑： 在安裝程式中輸入 yes 或 no 時，您可以改為輸入 y 或 n 縮寫。如果您在非英文機器上執行的安裝程式中輸入縮寫，則輸入的縮寫必須是 y 或 n。
重要： 用戶端公用程式 不應放在包含非 ASCII 字元的目錄中。

程序

若要設定「用戶端公用程式」，請執行下列動作：

  1. 選取您打算在其上執行「用戶端公用程式」的平台，然後按一下下載。這會下載 SAClientUtil_<version>_<os>.zip 檔案（其中 <version> 是「用戶端公用程式」的現行版本，而 <os> 是「用戶端公用程式」適用的作業系統）。
    註： 您可以選取表單中的如何建立 IRX 檔案？，並選取 IRX 檔，以重新開啟歡迎使用表單。
  2. 將檔案解壓縮至本端磁碟機。
  3. 如果您打算使用 CLI 來進行 IRX 檔案產生、上傳 IRX 檔案，或是管理掃描：請將解壓縮之「SAClientUtil_<version>_<os>.zip」檔案的 \bin 目錄的位置新增至 PATH 環境變數。如果您沒有這麼做，則每次發出此指令時，都需要使用所解壓縮之 SAClientUtil_<version>_<os>.zip 檔案的 \bin 目錄來限定所有指令。
    重要： 如果您打算以 CLI 或 Microsoft Visual Studio 掃描 .NET，這三個 .bat 檔（位於解壓縮的 SAClientUtil_<version>_<os>.zip 檔中）必須以管理者身分執行：
    • bin\register.bat
    • walalib\dotnet.libs40\RegisterInGAC.bat
    • walalib\dotnet.libs35\RegisterInGAC.bat

    這些檔案可以手動執行 - 您也可以用管理者身分執行 install-plugins.bat 檔（如果您只使用 CLI， 請執行 install-plugins.bat，但是當系統提示您安裝靜態分析外掛程式時，請輸入 no）。

    如果沒有以管理者身分執行這些檔案，每次使用 CLI 時，主控台中都會出現警告訊息。如果您是掃描 Java 目標，可以忽略這則訊息。

    提示： 變更 PATH 之後，請在命令提示字元中發出 appscan version (Windows) 或 appscan.sh version（Linux 和 macOS）。如果傳回 Static Analyzer 用戶端公用程式 版本、起始目錄和其他資訊，表示已正確設定 PATH。
  4. 如果您打算將靜態分析 外掛程式新增至 Eclipse、Microsoft Visual Studio（限 Windows）、IntelliJ IDEA 或 Maven：在解壓縮的「用戶端公用程式」中尋找並執行 install-plugins.bat (Windows) 或 install-plugins.sh （Linux 和 macOS）。在 Windows 上，以管理者身分執行檔案。如果您打算在 Linux 或 macOS 上將外掛程式新增至 Eclipse，請確定登入的帳戶有修改安裝目標 Eclipse 實例的許可權。
    1. 依預設，「用戶端公用程式」將會連接至位於「IBM Cloud Marketplace」的 ASoC 服務。而如果您將連接至位於 IBM Bluemix® 的服務，請在提示時輸入 yes。
    2. 如果您打算在 Eclipse 中掃描原始碼，請遵循安裝程式中的提示。如果您選擇安裝 Eclipse 外掛程式，則系統會提示您輸入 Eclipse 安裝的位置。 請輸入含有 eclipse.exe (Windows) 或 eclipse （Linux 或 macOS）檔案之目錄的完整路徑。
    3. 如果您打算使用 Maven 以產生「IRX」檔案，請遵循安裝程式中的提示。
    4. 如果您打算在 IntelliJ IDEA 中掃描原始碼，請遵循安裝程式中的提示。如果您選擇安裝 IntelliJ IDEA 外掛程式， 則系統會提示您輸入 IntelliJ IDEA 配置目錄的位置。請輸入此目錄的完整路徑。
    5. 僅限 Windows：如果您打算在 Visual Studio 中掃描原始碼，請遵循安裝程式中的提示。如果您選擇安裝 Visual Studio 外掛程式，系統會提示您選擇應在其中安裝外掛程式的 Visual Studio 版本。請輸入對應於要將外掛程式安裝到其中的 Visual Studio 版本的版本。
    6. 按任意鍵，以關閉命令提示字元。
    7. 針對您將外掛程式安裝到其中的所有 IDE，重新啟動 IDE。
    8. 僅限 Linux：在命令提示字元中，瀏覽至包含用戶端公用程式的目錄，並且將下列程式碼行新增至您的 Shell 資源檔（例如，.bash_profile 或 .shrc）： 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export LD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${LD_LIBRARY_PATH}

      其中，<client_utility> 是 用戶端公用程式 的路徑。在新增這些程式碼行之後，請重新載入您的 Shell，然後再使用靜態分析外掛程式。

    9. 僅限 macOS：在命令提示字元，瀏覽至包含用戶端公用程式的目錄，並且將下列程式碼行新增至您的 Shell 資源檔（例如，.bash_profile 或 .shrc）： 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export DYLD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${DYLD_LIBRARY_PATH}

      其中，<client_utility> 是 用戶端公用程式 的路徑。在新增這些程式碼行之後，請重新載入您的 Shell，然後再使用靜態分析外掛程式。

      重要： 在 Shell 資源檔中設定環境變數，只會影響在終端機中啟動的應用程式。因此，您必須使用終端機視窗來啟動要執行靜態分析的整合開發環境 (IDE)。

下一步

如果您要在受 Proxy 保護的電腦上執行 用戶端公用程式 - 使用下列其中一個方法來指定 Proxy，使得 用戶端公用程式 可以連接至雲端：

  • 用戶端公用程式 CLI 和支援的整合開發環境 (IDE)：設定這個廣域或系統環境變數，以便自動辨識 Proxy：
    • Windows：APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>
    • Linux 和 macOS：APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"

    其中 <proxy> 是 Proxy 伺服器的主機名稱，而 <port> 是 Proxy 伺服器使用的埠號。

    或者，每當您使用 CLI 或 IDE（從命令提示字元或終端機啟動），您可以發出下列指令，將 用戶端公用程式 設定為使用 Proxy：

    • Windows：set "APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
    • Linux 和 macOS：export APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
  • Maven：如果您已在 Maven 中新增靜態分析外掛程式，您可以在 MAVEN_OPTS 環境變數中廣域新增該內容， 或是在每一個指令中新增該內容。

移除 Static Analyzer 用戶端公用程式

這項作業說明如何從整合開發環境 (IDE) 移除 Static Analyzer 用戶端公用程式 外掛程式。此外，也提供了移除 用戶端公用程式 的指示。僅當您想要從系統中移除 Static Analyzer 用戶端公用程式 元件時，才需要遵循這些指示。如果您要將這些元件升級至新的 用戶端公用程式 版本，外掛程式安裝作業將會先自動移除舊的元件，再安裝新的版本。

開始之前

重要： 如果您已將 用戶端公用程式 外掛程式安裝至 IDE，您需要在解壓縮的 SAClientUtil_<version>_<os>.zip 位置中執行一或多個檔案或指令，才能完全移除 IDE 外掛程式。這些檔案如下：
  • Static Analyzer 用戶端公用程式 6.0.865 版和更舊版本：
    • walalib\dotnet.libs35\UnRegisterInGAC.bat
    • walalib\dotnet.libs40\UnRegisterInGAC.bat
  • 所有 Static Analyzer 用戶端公用程式 版本，包括 6.0.865 版和更舊版本：以管理者身分開啟命令提示字元，並將目錄切換至用來解壓縮 SAClientUtil_<version>_<os>.zip 的位置。將目錄切換至 bin\ 資料夾，然後發出以下指令：
    regsvr32 /u msdia80.dll

為了能夠執行這些檔案和指令，在刪除所有 IDE 外掛程式之前，請勿刪除解壓縮的 SAClientUtil_<version>_<os>.zip 內容。

這些檔案和指令可以在遵循以下指示來從 IDE 中移除外掛程式之前或之後執行。

使用指令行介面 (CLI) 產生 IRX 檔案

為了能夠起始檔案的分析，您必須產生要提交至雲端的 IRX 檔案。如果您要使用 CLI，請遵循這些指示來建立該檔案。

程序

若要產生 IRX 檔案，請執行下列動作：

  1. 確定開啟新的命令提示字元視窗（請不要使用用於安裝「用戶端公用程式」的命令提示字元視窗）。
  2. 在命令提示中發出指令，將目錄切換至包含您檔案的位置。請參閱語言支援， 瞭解支援的檔案類型。如果您在 Eclipse 中掃描 Java 專案，請將目錄切換至您的 Eclipse 工作區。
    註：
    • 如果您使用配置檔來產生 IRX 檔案，則不需要此步驟。請參閱 使用 CLI 配置 IRX 檔案產生。
    • 使用 CLI，您無法在 Eclipse 工作區中掃描個別專案，而只能掃描整個工作區。如果您要掃描個別專案，必須在 Eclipse 中產生 IRX 檔案。
    • .dll 和 .exe 檔必須是有效的 .NET 組件。
  3. 發出 appscan prepare (Windows) 或 appscan.sh prepare （Linux 和 macOS） 指令。這個指令是與配置指令 (Windows) 和配置指令（Linux 和 macOS）主題中列出的選項搭配使用。例如，您可以使用 -d 及 -n 選項，以指定在其中儲存 IRX 檔案的特定目錄，以及使用特定檔名來儲存檔案。此外，還可以將此指令與配置檔搭配執行，來指定可協助產生完整 IRX 檔案的其他資訊（請參閱使用 CLI 配置 IRX 檔案產生）。
    提示： 依預設，在產生 IRX 檔案期間，不會掃描第三方 Java 和 .NET 程式碼。您可以遵循管理第三方 Java 和 .NET 排除中的指示來修改第三方程式碼排除設定。 若要包括第三方程式碼，請在發出 prepare 指令時，指定 -t 或 --thirdParty 選項。

    如果您是通常會包含在掃描中的第三方程式碼的開發人員，則應使用此設定來包含第三方程式碼。

    註： 掃描原始碼或產生 IRX 檔案時，您可能會接收到關於更新至最新用戶端公用程式的訊息。請參閱用戶端公用程式 及雲端服務版本相容性。
  4. 發出指令之後，用戶端公用程式 會收集從中發出指令之目錄及其所有子目錄中任何所支援檔案的資訊。在目錄中會建立 IRX 檔案。此檔案的名稱會根據從中發出指令的目錄，並包括日期和時間戳記（假設您未使用配置選項來指定不同的檔名）。
  5. 將 IRX 檔案提交至雲端進行分析。其作法是發出 appscan queue_analysis 指令 (Windows) 或 appscan.sh queue_analysis 指令（Linux 和 macOS）。這個指令是與分析指令 (Windows) 或分析指令 （Linux 和 macOS） 主題所列出的選項搭配使用。

產生 Maven 專案的 IRX 檔案

為了能夠起始檔案的分析，您必須產生要提交至雲端的 IRX 檔案。如果您要掃描 Maven 專案，請遵循這些指示來建立該檔案。

程序

若要產生 IRX 檔案，請執行下列動作：

  1. 確定開啟新的命令提示字元視窗（請不要使用用於安裝「用戶端公用程式」的命令提示字元視窗）。
  2. 在命令提示字元中，發出指令以將目錄切換為含有 Maven 專案的位置（含有 pom.xml 檔案的資料夾）。
  3. 發出下列其中一個指令：
    • 如果尚未建置您的 Maven 專案，請發出 mvn package com.ibm.appscan:appscan-maven-plugin:prepare。發出指令之後，即會在 Maven 專案的 target 目錄中建立 IRX 檔。假設您未使用配置選項來指定不同的檔名，則此檔案的名稱會根據最上層的 Maven 專案而定（pom.xml 檔中的專案名稱）。
    • 如果已建置您的 Maven 專案，請發出 mvn com.ibm.appscan:appscan-maven-plugin:prepare。發出指令之後，即會在 Maven 專案的 target 目錄中建立 IRX 檔。假設您未使用配置選項來指定不同的檔名，則此檔案的名稱會根據最上層的 Maven 專案而定（pom.xml 檔中的專案名稱）。
      註： 如果 Maven 專案在相同的反應程式建置中，與另一個專案之間有編譯時期相依關係，則現有的 Maven 錯誤 (https://issues.apache.org/jira/browse/MNG-3283) 會導致 mvn com.ibm.appscan:appscan-maven-plugin:prepare 指令失敗。若要暫行解決此問題，請執行下列其中一個步驟：
      • 在執行 用戶端公用程式 prepare 指令之前，先執行 mvn install 指令而非 mvn package。
      • 同時發出 mvn package 指令和 用戶端公用程式 prepare 指令（發出 mvn package com.ibm.appscan:appscan-maven-plugin:prepare）。
    • 如果您要在不同目錄中建立 IRX 檔案，或者希望該檔案具有不同的檔名，請發出具有 -Doutput 選項的指令：

      在 Windows 中，發出此指令：

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<output_location>\<filename>.irx

      在 Linux 和 macOS 中，發出此指令：

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<output_location>/<filename>.irx

      其中 <output_location> 是您要在其中儲存檔案的目錄，而 filename 是檔案的名稱。

    • 「用戶端公用程式」會包括 Tomcat JSP 編譯器。 預設會在 IRX 檔案產生期間使用此 JSP 編譯器。如果您是產生 WAR 或 EAR 檔案的 IRX 檔案，而且想要使用不同的 JSP 編譯器，請發出具有 -DjspCompiler 選項的指令。例如：
      mvn com.ibm.appscan:appscan-maven-plugin:prepare -DjspCompiler=<jsp_compiler>

      其中 <jsp_compiler> 是您要使用的 JSP 編譯器的路徑（指定伺服器的根目錄）。

    提示： 依預設，在 IRX 檔案產生的期間，不會掃描第三方程式碼。您可以遵循管理第三方 Java 和 .NET 排除中的指示來修改第三方程式碼排除設定。若要包括第三方程式碼，請發出搭配 -DthirdParty 選項的指令。

    如果您是通常會包含在掃描中的第三方程式碼的開發人員，則應使用此設定來包含第三方程式碼。

    註： 掃描原始碼或產生 IRX 檔案時，您可能會接收到關於更新至最新用戶端公用程式的訊息。請參閱用戶端公用程式 及雲端服務版本相容性。
  4. 將 IRX 檔案提交至雲端進行分析。其作法是發出 appscan queue_analysis 指令 (Windows) 或 appscan.sh queue_analysis 指令（Linux 和 macOS）。這個指令是與分析指令 (Windows) 或分析指令 （Linux 和 macOS） 主題所列出的選項搭配使用。

配置檔用法

配置檔可以用於指出要在產生 IRX 檔案時包括或排除的目標。您也可以指定含有已建置構件的目錄，而且可以指定應該在 IRX 檔案產生期間使用的建置資訊（例如，您可以使用此設定來指定特定 JDK 或 JSP 編譯器）。

只要發出 appscan prepare (Windows) 或 appscan.sh prepare （Linux 和 macOS） 指令以產生 IRX 檔案，Static Analyzer 用戶端公用程式 就會自動檢查現行目錄中是否有 appscan-config.xml 檔。如果找到該檔案，就會自動使用它。

如果您的配置檔是在不同的目錄，或者檔案名稱不同，則可以在發出 appscan prepare 或 appscan.sh prepare 指令時包括 -c 選項，來使用該檔案。在此情況下，您可以在 (Windows) 上發出 appscan prepare -c <configuration_file>，或在 （Linux 和 macOS） 上發出 appscan.sh prepare -c <configuration_file>，其中 <configuration_file> 是配置檔的完整路徑和檔名。

配置檔範本及設定

在準備配置檔時，請使用此範本：

<Configuration>
  <Targets>
    <Target outputs-only="true/false" path="scan_target_path">
      <CustomBuildInfo build_info="info"/>
      <Include>string_pattern</Include>
      <Exclude>string_pattern</Exclude>
    </Target>
  </Targets>
</Configuration>

您可以使用 Targets 元素來指出要在產生 IRX 檔案時掃描的目標。目標 (Target) 屬性及子元素的用法如下所示：

  • 唯有當您想要掃描目錄並強制 用戶端公用程式 只尋找建置輸出檔（例如 .jar、.war 及 .class 檔案）時，才使用 Target 元素 outputs-only 屬性。此屬性預設是設定為 false。這表示 用戶端公用程式 會搜尋該目錄，以判定它是否為目標（例如應用程式伺服器或 Eclipse 工作區）- 或者該目錄是否包含建置 Script、Maven .pom 檔及 make 檔之類的項目。如果您希望將掃描目標當成簡單目錄，並且只尋找輸出檔，請在 Target 元素中指定 outputs-only="true"。

    例如，若您指定 <Target outputs-only="false" path="C:\Tomcat">，用戶端公用程式 會假設該目標是您的 Tomcat 應用程式伺服器 - 並搜尋其已部署的 .war 檔。不過，如果您指定 <Target outputs-only="true" path="C:\Tomcat">，用戶端公用程式 會將該位置視為目錄，並找出其中所有的建置輸出檔。

  • 使用 Target 元素 path 屬性，以指定掃描目標的路徑或掃描目標的目錄 (<scan_target_path>)。當指定目錄時，在掃描期間會包含其所有子目錄。

    例如，若您指定 <Target outputs-only="false" path="C:\WebSphere\AppServer85\profiles\AppSrv01">，用戶端公用程式 會找出 AppSrv01 設定檔中所有已部署的 .ear 檔。如果您指定 <Target outputs-only="false" path="C:\WebSphere\AppServer85">，用戶端公用程式 會找出所有設定檔中所有已部署的 .ear 檔。

  • 您也可以指定這些選用的 Target 子元素：
    • 您可以使用 CustomBuildInfo 元素來設定屬性。使用範本的 
      <CustomBuildInfo build_info="info"/>
      這個部分，您可以指定建置資訊，視目標語言而定。針對某些語言，可以設定多個屬性，例如，您可以設定 <CustomBuildInfo build_info_1="info_1" build_info_2="info_2" build_info_3="info_3"/>，視目標語言而定。
      • 如果是 Java，您可以設定這些屬性：
        <CustomBuildInfo additional_classpath="dependency_path"
  jdk_path="JDK_path" jsp_compiler="JSP_compiler_path"/>
        • 使用 additional_classpath 屬性指定其他類別路徑。在 Windows 上，使用分號區隔多個類別路徑。在 Linux 和 macOS，以冒號隔開多個類別路徑。
        • 使用 jdk_path 屬性，以指定您 JDK 安裝的路徑。
        • 使用 jsp_compiler 屬性，以指定您 JSP 編譯器的路徑，例如，jsp_compiler="C:\Tomcat"。
      • 如果是 .NET（僅限 Windows），您可以設定這些屬性：
        <CustomBuildInfo reference="assembly_reference"
  configuration="build_configuration"/>
        • 使用 reference 屬性新增組件參照。請以分號區隔多個參照。
        • 使用 configuration 屬性併入 Visual Studio 解決方案探索的建置配置。
      • 如果是 C/C++（僅限 Windows），您可以設定這些屬性：
        <CustomBuildInfo configuration="build_configuration"
  include_paths="include_directories" macros="macros" compiler_opts=/>
        • 使用 configuration 屬性併入建置配置。
        • 使用 include_paths 屬性指定併入路徑。請以分號區隔多個併入路徑。
        • 使用 macros 屬性併入巨集。請以分號區隔多個巨集。
        • 使用 compiler_opts 屬性指定編譯器選項。請以分號區隔多個選項。
    註： 這些屬性設定的值由子目標繼承。例如，若目標是 EAR 檔案，其中包含 WAR 和 JAR 子目標，則會假設 WAR 和 JAR 含有使用這些屬性對 EAR 檔案設定的相同值。
  • 使用 Include 子元素，以指定要在產生 IRX 檔案時包括的檔案型樣 (<string_pattern>)。Include 行為視目標類型而定，如目標併入與排除行為小節所概述。若要指定多個 include 型樣，請在型樣本身的 <Include></Include> 標記中新增每一個型樣。 例如，
    <Include>string_pattern_1</Include>
<Include>string_pattern_2</Include>
    註： 如果您指定的 include 型樣與 exclude 型樣發生衝突，則以 exclude 型樣為優先。
  • 使用 Exclude 子元素，以指定要在產生 IRX 檔案時排除的檔案型樣。Exclude 行為視目標類型而定，如目標併入與排除行為小節所概述。若要指定多個 exclude 型樣，請在型樣本身的 <Exclude></Exclude> 標記中新增每一個型樣。
    註： 如果您指定的 include 型樣與 exclude 型樣發生衝突，則以 exclude 型樣為優先。

      <Include>ShoppingCart.war</Include>

  • C:\test\MyApplication.ear 是目標。
  • 在 C:\java\lib 和 C:\java2\lib 中搜尋是否有編譯所需的任何類別檔相依關係。
  • 使用 C:\Program Files\Java\jdk1.7 來尋找相依關係。
  • 會使用 C:\Tomcat 編譯器來編譯 JSP， 而不是使用 Static Analyzer 用戶端公用程式 隨附的 Apache Tomcat 第 7 版應用程式。
  • 產生 IRX 檔案時，會併入 ShoppingCart.war 檔案，且結尾為 *Account.jar 的所有檔案也一樣會併入。EAR 檔案的 thirdPartyJars/ 中的一切都會被排除，且其檔名中有 test 字串的任何檔案也一樣被排除。如果發生衝突，則以 exclude 型樣為優先。例如，若 EAR 檔案包含 testCustomerAccount.jar 檔案，則會排除它，因為 test exclude 型樣的優先順序高於 *Account.jar include 型樣。

在整合開發環境中掃描

這些指示說明在靜態分析外掛程式已安裝到 Eclipse、IntelliJ IDEA 或 Visual Studio 整合開發環境 (IDE) 之後，如何使用它來掃描原始碼。在 Eclipse 和 IntelliJ IDEA 中， 您可以掃描 Java 專案 - 而在 Visual Studio 中，您可以掃描 .NET（C#、ASP.NET、VB.NET）。

開始之前

依預設，在掃描 Java 和 .NET 時，不會包含第三方程式碼。您可以遵循管理第三方 Java 和 .NET 排除中的指示來修改第三方程式碼排除設定。
  • 在 Eclipse、IntelliJ IDEA 或 Visual Studio 中掃描時，如果要包含第三方程式碼， 請使用下列其中一個方法：
    • 啟動 ID 之前，先設定這個廣域或系統環境變數： 
      APPSCAN_OPTS=-DthirdParty
    • 每當使用 IDE 時，您可以在啟動 IDE 之前發出下列指令：
      set APPSCAN_OPTS=-DthirdParty
  • 或者，在 Eclipse 中掃描時，您可以這樣做：在啟動 Eclipse 之前先修改 eclipse.ini 檔，使 -vmargs 區段包含 -DthirdParty。
  • 或者，當您在 IntelliJ IDEA 中掃描時，您可以這樣做：在啟動 IntelliJ IDEA 之前，先在 IntelliJ IDEA 安裝架構的 bin 目錄中，修改 idea.exe.vmoptions 檔。將 -DthirdParty 新增到檔案，然後儲存它。

如果您是通常會包含在掃描中的第三方程式碼的開發人員，則應使用此設定來包含第三方程式碼。

程序

若要掃描原始碼或是開啟評量或報告，請執行下列動作：

  1. 確定外掛程式已安裝到 IDE。在安裝期間，如果 IDE 已開啟，請將其重新啟動。
  2. 選取您要掃描的項目：
    • 在 Eclipse 中，選取您要掃描的一或多個專案。若要掃描整個 Eclipse 工作區，請選取所有專案。
    • 在 IntelliJ IDEA 中，選取您要掃描的專案模組。
    • 在 Visual Studio 中，選取您要掃描的解決方案、專案或網站。
  3. 用滑鼠右鍵按一下選項，然後選取安全分析 > 執行靜態分析。
    註： 掃描原始碼或產生 IRX 檔案時，您可能會接收到關於更新至最新用戶端公用程式的訊息。請參閱用戶端公用程式 及雲端服務版本相容性。
  4. 在掃描準備之後，如果您尚未登入服務，即會開啟「登入」對話框。開啟的對話框取決於啟動 install-plugins.bat (Windows) 或 install-plugins.sh (Linux 和 macOS) 安裝程式之後所做的連線設定。請在對話框中，輸入服務認證：
    • 如果您要連接至 Cloud Marketplace：在 Application Security on Cloud 服務中產生 API 金鑰時，您會接收到金鑰 ID和金鑰密碼。請在 ID 和密碼欄位中輸入這些值。如果您尚未產生 API 金鑰，請遵循對話框中的鏈結以建立 API 金鑰。
    • 如果您要連接至 Bluemix：請輸入服務認證中所包含的連結 ID 和密碼。

      如果要判斷 Bluemix 服務認證，請在服務「儀表板」的左導覽窗格中選取服務認證。請參閱啟用外部應用程式使用 Bluemix 服務。

    當您登入服務時，會建立一個加密金鑰檔。之後當與 ASoC 服務互動時，其他動作就會參照此記號檔。

    如果您已登入 Bluemix，此記號檔會在它建立之後的 2 小時到期。在此情況下，在它到期之後， 您需要再次登入服務，以繼續執行進一步動作 - 或者您可以在登入時選取儲存認證勾選框。一旦選取這個勾選框，用戶端公用程式 就會在記號到期時，自動嘗試重新接受服務的鑑別。

  5. 限IBM Cloud Marketplace： 如果您連接至 IBM Cloud Marketplace 上的 Application Security on Cloud 服務，您 IDE 中的靜態分析掃描必須與現有的 Application Security on Cloud 應用程式相關聯。啟動掃描之後，系統會藉由對話框提示您選擇要關聯的應用程式。

    選取要與掃描關聯的應用程式之後，關聯資訊會依下列方式儲存：

    • 在 Eclipse 中，應用程式是與整個工作區相關聯。此關聯的相關資訊會儲存在 <workspace>/.metadata/.plugins/com.ibm.appscan.eclipse.plugin/.scans/.app 中，其中 <workspace> 是 Eclipse 工作區的位置。
    • 在 IntelliJ IDEA 中，應用程式是與個別專案模組相關聯。此關聯的相關資訊會儲存在 <project>/.scans/.app 中，其中 <project> 是專案模組的位置。
    • 在 Visual Studio 中，應用程式是與解決方案相關聯。此關聯的相關資訊儲存在 <solution>/.scans/.app 中，其中 <solution> 是包含 .sln 檔的目錄。

    如果將 .app 檔從這些任一位置移除，下次掃描時，系統會提示您選擇關聯的應用程式。如果您選擇了應用程式關聯，但要將它變更到不同的應用程式，請刪除 .app 檔再重新掃描。

    註： 當您連接至 IBM Bluemix 上的 Application Security on Cloud 服務時，並不適用應用程式關聯。
  6. 在提交掃描之後，即會開啟「靜態分析器掃描」視圖。
  7. 當掃描完成時，會開啟一個通知，含有開啟結果或報告的鏈結。此外，還會更新「靜態分析器掃描」視圖以包含掃描。此視圖會列出掃描名稱、狀態、開始和結束的時間，以及找到之漏洞的數目和嚴重性。
    1. 結果選取通知中的鏈結來開啟結果；或是按兩下視圖中的結果直欄中的圖示（在 Visual Studio 中，也可以使用系統匣來開啟結果）。這會開啟互動式評量， 可依修正程式群組列出發現項目。修正程式群組代表分組發現項目流經的最常見節點。一般而言，如果針對修正程式群組實作了修正程式，則可以用較小量的工作達到較最佳的效果。修正程式群組也可視為可在該處同時檢閱相關發現項目的邏輯分組點。請注意，修正程式群組不一定是應放置修正程式的確切位置。您可以使用某修正程式的修正程式群組位置來防止將來的重構、程式碼實務及其他因素。

      在評量中：

      • 每一個修正程式群組都會顯示建議的修正程式位置、開啟該修正程式位置來源的鏈結、 漏洞，以及在更正原始碼後將會修正的漏洞出現數目。
      • 如果您選取漏洞，即會開啟它的說明（通常附有範例和建議事項）。
      • 如果您選取詳細資料按鈕，修正程式群組即會開啟到在更正原始碼後將會修正的所有發現項目。在詳細視圖中，如果您選取原始碼位置，則它會在原始碼編輯器中開啟。選取追蹤圖示會開啟追蹤，其中顯示流經應用程式的資料流。
      重要： 在 IntelliJ IDEA 或某些訂閱層次上無法使用互動式評量（結果）。
    2. 報告選取通知中的鏈結來開啟報告；或是按兩下視圖中的報告直欄中的圖示（在 Visual Studio 中，也可以使用系統匣來開啟報告）。這會開啟發現項目的報告。您已連接的雲端服務上也會提供此報告。

結果

重要： 在整合開發環境中不支援重新掃描。

產生靜態分析 IRX 檔時發生錯誤或警告

症狀

產生 IRX 檔案時，您可能會接收到錯誤或警告，或者產生 IRX 檔案可能會失敗。當 IRX 檔案產生失敗時，產生檔案的名稱會附加 _failed（例如，my_irx_file_failed.irx）。當發生這種情況時，有多種選項可進行 IRX 檔案產生的疑難排解。

解決問題

  • 使用配置檔進行 IRX 檔案產生的疑難排解

    如果您在未使用配置檔的情況下產生 IRX 檔案，而檔案產生未完成（或者已完成但發生錯誤），則 Static Analyzer 用戶端公用程式 會產生暫時配置檔，供您用於後續掃描。此檔案與配置檔範本類似，因為其中包含具有目標已知相關資訊的 Targets 元素。不過，此檔案也包含 <TargetSettings> 元素供您修改，您可以新增資訊，讓 用戶端公用程式 能夠產生完整的 IRX 檔案。此檔案命名為 appscan-config.xml，而且會儲存至現行目錄。這表示下一次掃描時將會自動使用此檔案。

    例如，如果目標的 IRX 檔案產生含有找不到的外部相依關係，您可能會接收到如下的訊息：

    C:\myTargetDirectory>appscan prepare
An IRX file was created, but it may be incomplete.
C:\myTargetDirectory\myTarget.jar requires the following packages:
      org.apache.jasper.*
An appscan-config.xml file was created in: C:\myTargetDirectory\appscan-config.xml
Please update the file and rerun the command.

    產生的 appscan-config.xml 檔案會與下面內容類似：

    <Configuration>
  <Targets>
    <Target outputs-only="true" path="C:\myTargetDirectory"/>
  </Targets>
  <Settings/>
  <TargetSettings>
    <Target path="C:\myTargetDirectory\myTarget.jar"/>
      <!--Please add the path of the following dependencies to the attribute "additional_classpath" below, separated by semi-colon.
			  org.apache.jasper.*-->
      <CustomBuildInfo additional_classpath=""/>
    </Target>
  </TargetSettings>
</Configuration>

    若要解決此問題，請更新 <CustomBuildInfo additional_classpath=""/>，以在類別路徑中併入 org.apache.jasper.* 類別， 儲存 appscan-config.xml 檔，然後重新執行 appscan prepare (Windows) 或 appscan.sh prepare （Linux 和 macOS） 指令。當您指定一或多個 JAR 檔的路徑時，支援標準 Java 類別路徑語法（例如，支援萬用字元）。

    如果您在已使用配置檔時發生類似問題，則會接收到指出問題的訊息。請使用此訊息來更新現有配置檔，然後重新執行指令。

    該配置檔說明於使用 CLI 配置 IRX 檔案產生。

  • 以除錯模式產生 IRX 檔案

    如果您是從 CLI 產生 IRX 檔案，則可以發出具有除錯選項的 appscan prepare (Windows) 或 appscan.sh prepare （Linux 和 macOS） 指令。這會導致產生較多的日誌檔。

  • 與 IRX 檔案產生相關聯的日誌檔

    若要查看這些錯誤或警告，您也可以參閱一併產生的日誌檔。日誌檔會儲存到 .zip 保存檔。依預設，檔案會儲存到 IRX 儲存的相同位置 - 或者您可以使用 appscan prepare (Windows) 或 appscan.sh prepare （Linux 和 macOS） 指令的 -l 選項指定另一個位置。.zip 檔案的名稱會根據所產生之 IRX 檔的名稱。

Eclipse 外掛程式安裝在設定 Static Analyzer 用戶端公用程式 時失敗

症狀

將靜態分析外掛程式安裝到 Eclipse 時，安裝失敗，帶有錯誤。

解決問題

由於 Application with id x does not exist 錯誤，安全掃描失敗

症狀

由於下列錯誤，執行使用 Jenkins 或 IDE 外掛程式或指令行介面 (CLI) 的掃描失敗：

Application with id x does not exist
Failed to run the security scan: Problems occurred submitting the IRX file to the analysis service.

原因

在先前，Application Security on Cloud 會根據 app_id 來關聯應用程式。最近的更新將此關聯從 app_id 變更為 guid。在大多數的情況下，ASoC 服務會提示您根據此更新來重新關聯應用程式。針對 Jenkins 以及 IDE 外掛程式和 CLI，您必須採取額外的步驟。

解決問題

針對 Jenkins 外掛程式，請根據將安全分析新增至 Jenkins 自動化伺服器上的指示來重新儲存 Jenkins 外掛程式配置，然後重試掃描。

針對 IDE 外掛程式，請重新選取相同的應用程式來掃描，然後按一下確定。

針對 CLI，新的 guid 會取代 app_id。guid 是一個唯一的 32 字元 ID。例如，33c3281a-6d7c-e711-b334-002590731623。請使用 appscan list_apps 指令判定各個應用程式的 guid，然後在 Script 中或指令行上的 queue_analysis 指令中，以 app_id 取代它。

使用註釋

您可以透過遵循下列步驟來使用註釋：

  1. 依預設會啟用註釋支援。註釋 .jar 檔位於解壓縮的 SAClientUtil_<version>_<os>.zip 檔的 \annotation 目錄。

@ValidatorMethod

驗證器是一些方法，可對輸入資料執行檢查，且經常會傳回布林值，以指出輸入是否有效。您可以將使用者輸入變更為可接受的格式，而不是使用驗證器來接受或拒絕輸入。這些方法稱為消毒器 (sanitizer)。

您可以使用 @ValidatorMethod 註釋來識別應用程式原始碼中的所有驗證器和消毒器方法。在靜態分析掃描期間，將會使用這項資訊來移除通過這些方法的資料流程，因為現在將資料視為安全。

註： 目前，並未提供指定應將註解方法的哪些參數視為已經過驗證。在靜態分析掃描期間，會將所有的輸入參數都認定為已經過驗證。

@SuppressSecurityTrace

系統會移除流經方法中標示此註釋的所有追蹤資料。當系統將某一組的追蹤資料識別為誤判，或是比其他追蹤資料較不重要或有意思時，這就非常有用。您可以使用此註釋來過濾掉這些追蹤資料或是隱藏它們以減少雜亂。

@CallbackMethod

此註釋用來識別應用程式的回呼點或進入點。所有引數都被視為帶有污染。

配置未解析的 PHP include 表示式及不明 PHP 類別

在產生 PHP 的 IRX 檔案之後，會建立配置檔，可讓您在下次掃描該檔案時配置 IRX 產生。

配置檔 php-scan-config.xml 會在您從中發出 prepare 指令的位置中產生。該檔案包含 <UserInputs> 和 <Php-Unknown-Configuration> 兩個區段。

<UserInputs> 區段包含可針對 PHP 掃描修改的配置設定。此區段會預先填入範例值。您可以使用分號 (;) 附加到這些值。此區段中的設定為：

  • <RegularExpression>：這是進階功能，可針對搜尋文字指定正規表示式。
  • <ReplaceText>：此設定用於簡式文字搜尋和取代。搜尋文字是依現狀使用，而如果在包含的文字中的任何位置發現它，則會以取代文字取代它。
  • <Function>：當要取代某函數呼叫時，會使用此設定。搜尋文字應為沒有括弧的函數名稱。將會擴增搜尋文字以尋找具有指定名稱且後接括弧的函數呼叫，並會比對括弧內的任何內容。
  • <SimpleText>：簡化的取代文字。當取代文字是單一字串時，請使用此設定。
  • <SimpleRegEx>：簡化的取代正規表示式。當取代正規表示式是單一字串時，請使用此設定。
  • <AdditionalIncludePaths>：請使用此設定來新增包含所參照檔案的額外 include 路徑。

<Php-Unknown-Configuration> 區段包含在前次掃描中發現的所有未解析的 include 表示式及不明類別。請使用這項資訊來完成 <UserInputs> 區段。

  • <Summary>：彙總前次掃描中的不明 include 表示式 (<NumberOfUnknownIncludeExprs>) 及不明類別 (<NumberOfUnknownClasses>) 的數目。
  • <SourceFile>：所有不明 include 表示式及不明類別會按原始檔名稱的字母順序列出。針對每一個原始檔，檔案路徑 (<FilePath>) 會列出原始檔、不明類別 (<UnknownClass>) 及不明 include 表示式 (<UnknownIncludes>) 的絕對路徑。

不明的類別 (<UnknownClass>) 資訊

標籤 說明
<ClassName> 原始碼中參照的類別名稱，連同原始檔 (<FilePath>) 中使用該類別所在的位置。您可能需要在編輯器中查看該位置， 以瞭解應如何解析不明類別。
<Status> 指出類別為已解析還是未解析。
<ResolvedBy> 指出解析不明類別的方式。值包括：
  • AutoResolver：應用程式使用內部啟發來尋找檔案。
  • SearchReplace：已將一或多個搜尋和取代規則套用到包含的文字，以產生更新的文字。

不明的 include 表示式 (<UnknownIncludes>) 資訊

標籤 說明
<IncludeText> 原始碼中存在的表示式文字，連同原始檔 (<FilePath>) 中使用該表示式所在的位置。您可能需要在編輯器中查看該位置， 以瞭解應如何解析不明 include 表示式。
<UpdatedText> 在前次掃描期間所用的更新文字。
<Status> 指出表示式為已解析還是未解析。
<ResolvedBy> 指出產生更新文字的方式。值包括：
  • AutoResolver：應用程式使用內部啟發來尋找檔案。
  • SearchReplace：已將一或多個搜尋和取代規則套用到包含的文字，以產生更新的文字。
  • SearchReplace+AutoResolver：已將一或多個搜尋和取代規則套用到包含的文字，以產生更新的文字，然後已套用內部啟發來尋找檔案。
  • SearchReplace+IncludePath：已將一或多個搜尋和取代規則套用到包含的文字，以產生更新的文字，然後與 include 路徑上的目錄結合來尋找檔案。
  • SearchReplace+RelativeDir：已將一或多個搜尋和取代規則套用到包含的文字，以產生更新的文字，然後發現與包含 include 表示式之檔案的來源目錄相關。

管理第三方 Java 和 .NET 排除

依預設，在 IRX 檔案產生的期間，不會掃描第三方 Java 和 .NET 程式碼。您可以遵循這個主題中的指示來管理執行的第三方程式碼。

關於這項作業

您產生程式碼的 IRX 檔時會排除第三方程式碼，讓您的評量不包含您無意分析之程式碼的相關資訊和發現項目。如果要修改排除的內容，請遵循這個主題中的步驟。或者，如果您要在掃描時包含第三方程式碼，請參閱下列主題：

  • 使用指令行介面 (CLI) 產生 IRX 檔案
  • 產生 Maven 專案的 IRX 檔案
  • 在整合開發環境中掃描

程序

  1. 尋找解壓縮之 SAClientUtil_<version>_<os>.zip 檔案的 config 目錄（其中 <version> 是現行版本的 用戶端公用程式）。
  2. 在此目錄中，尋找 dot_net.exclusions (.NET) 或 java.exclusions (Java) 檔。
    註： 這個目錄中的其他 .exclusions 檔案為實驗性質，修改這些檔案並不會影響第三方排除。
  3. 在文字編輯器中開啟檔案，您會看到如下格式的 XML： 
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<Exclusions>
  <packages>
    <package name="package_name"/>
  </packages>
  ...
</Exclusions>
  4. 編輯檔案，新增要排除的套件。
    註：
    • 指定的套件名稱將會比對出具有相同字首的所有套件（視為後面跟隨萬用字元）。例如，如果您新增 <package name="com.mycompany.common"/> 作為排除，則會排除 <package name="com.mycompany.common.action1"/> 和 <package name="com.mycompany.common.action2"/>。
    • 只有新增至 <packages> 儲存器的項目才會影響第三方排除。新增類別和方法不會影響排除。
  5. 儲存檔案，然後產生針對您的原始碼產生 IRX 檔。

下一步

如果您修改排除檔案，您應遵循下列最佳實務：
  • 保存修改排除檔案的備份。特別要注意的是，如果您升級至新版的 Static Analyzer 用戶端公用程式，您下載的排除檔案版本沒有您所做的更新。更新 用戶端公用程式 時，請保存一份您修改的排除檔案副本，然後將副本複製回新的 用戶端公用程式。
  • 將修改保持在單一區塊的 XML 項目，以便在需要時能夠容易地複製和貼上。

appscan.sh prepare

語法：

appscan.sh prepare -c <configuration_file> -d <save_path> -l <log_path> -n <file_name> -t,--thirdParty -v,--verbose -X,--debug

說明：

產生 IRX 檔案。

註： 掃描原始碼或產生 IRX 檔案時，您可能會接收到關於更新至最新用戶端公用程式的訊息。請參閱用戶端公用程式 及雲端服務版本相容性。

選用選項：

  • -c：此選項用於配置掃描。
  • -d：指定 -d <save_path>，其中 <save_path> 是您想要儲存 IRX 檔案的目錄。
  • -l：指定 -l <log_path>，其中 <log_path> 是您想要儲存日誌檔的目錄。
  • -n：指定 -n <file_name>，其中 <file_name> 是 IRX 檔案名稱。指定檔名時不一定要有 .irx 副檔名。 如果您指定沒有副檔名的檔名，則在產生檔案時會為您自動新增副檔名。
  • -t,--thirdParty：依預設，在產生 IRX 檔案期間，不會掃描第三方 Java 和 .NET 程式碼。您可以遵循管理第三方 Java 和 .NET 排除中的指示來修改第三方程式碼排除設定。 若要包括第三方程式碼，請在發出 prepare 指令時，指定 -t 或 --thirdParty 選項。

  • -v,--verbose：指定 -v 或 --verbose，以在 IRX 檔案產生期間顯示更多參考資訊輸出。
  • -X,--debug：指定 -X 或 --debug，以除錯模式執行整個指令。在除錯模式執行時，會產生較多日誌檔來進行疑難排解。

appscan.sh get_pubkey

語法：

appscan.sh get_pubkey -d <save_path>

說明：

下載公開加密金鑰以便用於未連接網際網路的電腦。

如果您要從連接網際網路的電腦產生 IRX 檔案，則不需要這個指令，因為在您發出 prepare 指令時會自動下載加密金鑰。如果電腦上已有加密金鑰，當您發出 prepare 指令時會視需要更新它。

然而，如果您要從未連接網際網路的電腦產生 IRX 檔案，則您可以使用這個指令下載加密金鑰。接著您可以將加密金鑰複製到未連接網際網路的電腦，以便產生 IRX 檔案時使用。若要在該電腦上使用加密金鑰，您必須保留 rsa.pub 檔名，並將檔案放在解壓縮之 SAClientUtil_<version>_<os>.zip 檔案的 config 目錄中（其中 <version> 是「用戶端公用程式」的現行版本）。

註： 使用的加密金鑰必須是最新的。如果使用過時的加密金鑰產生 IRX 檔案，則在上傳 IRX 檔案時會遭到拒絕。如果您的電腦已連接網際網路，請重新發出 prepare 指令，以便自動更新加密金鑰。如果您的電腦未連接網際網路，則需要使用 get_pubkey 指令。

選用選項：

  • -d：指定 -d <save_path>，其中 <save_path> 是您想要儲存加密金鑰的目錄。 如果未指定這個選項，金鑰將會儲存到解壓縮之 SAClientUtil_<version>_<os>.zip 檔案的 config 目錄。
