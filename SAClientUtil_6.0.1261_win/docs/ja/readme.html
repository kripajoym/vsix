Static Analyzer 入門

静的分析を使用して、セキュリティー脆弱性がないかソース・コードをスキャンします。これを行うには、小さなユーティリティーをダウンロードする必要があります。 ユーティリティーをローカル・ディスクに解凍した場合、そのコマンド・ライン・インターフェース (CLI) を使用してすべてのサポート対象言語でセキュリティー分析を実行できます。 あるいは、Eclipse、Maven、IntelliJ IDEA、または Visual Studio に静的分析プラグインを追加する、組み込み済みのインストーラーを実行できます。Eclipse および IntelliJ IDEA では、Java プロジェクトをスキャンできます。また Visual Studio では .NET (C#、ASP.NET、VB.NET) をスキャンできます。

手順

  • IBM Cloud Marketplace で Application Security on Cloud サービスを使用する方法については、http://www.ibm.com/support/knowledgecenter/SSYJJF_1.0.0/ApplicationSecurityonCloud/Welcome.htmlを参照してください。
  • Application Security on Cloud for Bluemix の使用方法については、https://console.ng.bluemix.net/docs/services/ApplicationSecurityonCloud/index.htmlを参照してください。
  • この README ファイルの残りの部分で、Static Analyzer クライアント・ユーティリティー インストールと使用法について説明しています。

Static Analyzer クライアント・ユーティリティー のセットアップ

静的分析を実行するには、小規模なクライアント・ユーティリティーをダウンロードします。ユーティリティーをローカル・ディスクに解凍した場合、そのコマンド・ライン・インターフェース (CLI) を使用してセキュリティー分析を実行できます。 あるいは、Maven ビルド環境、または Eclipse、IntelliJ IDEA、または Visual Studio の統合開発環境 (IDE) に静的分析プラグインを追加する、組み込み済みのインストーラーを実行できます。

このタスクについて

静的分析を使用する場合、クラウドにアップロードされるファイルを生成するためにソース・コードをスキャンします。 このファイル (暗号化された IRX (.irx) ファイル) は、セキュリティー脆弱性を検出するためにトレース分析を使用してスキャンされます。 IRX ファイルがコンパイル言語用に生成される場合、アプリケーションのバイト・コード・ファイルは、コードの 中間表現 に変換されます。 IRX ファイルがスクリプト言語用に生成される場合、ソース・ファイルは暗号化された IRX ファイルに組み込まれます。

Eclipse、IntelliJ IDEA、または Visual Studio のプラグインを使用する場合、静的分析アクションの使用時に、IRX ファイルの生成とアップロードが IDE から自動的に行われます。

ヒント: インストーラーの実行時、特定の IDE で静的分析を使用しないものの、その IDE がインストール済みであり、将来は静的分析を使用することが想定される場合、ここでプラグインをその IDE にインストールしておくと後でインストーラーを再実行しなくてすみます。
ファースト・パス: インストーラーで「yes」または「no」を入力する際、代わりに省略語として y または n を入力できます。英語以外のマシンで実行中のインストーラーで省略語を入力する場合、入力する省略語は y または n でなければなりません。
重要: クライアント・ユーティリティー は非 ASCII 文字を含むディレクトリーに入れてはいけません。

手順

クライアント・ユーティリティーをセットアップするには、次の手順を実行します。

  1. クライアント・ユーティリティーを実行する予定のプラットフォームを選択し、次に「ダウンロード」をクリックします。これにより、SAClientUtil_<version>_<os>.zip ファイル (<version> はクライアント・ユーティリティーの現行バージョン、<os> はクライアント・ユーティリティーの対象となるオペレーティング・システム) がダウンロードされます。
    注: IRX ファイルを選択できるフォーム内で「IRXファイルの作成方法 (How do I create an IRXfile?)」を選択することで、ウェルカム・フォームを再度開けます。
  2. ファイルをローカル・ドライブに解凍します。
  3. IRX ファイルの生成、IRX ファイルのアップロード、またはスキャンの管理に CLI を使用する予定の場合、解凍後の SAClientUtil_<version>_<os>.zip ファイルの ¥bin ディレクトリーの場所を PATH 環境変数に追加してください。これを行わないと、コマンドを実行するたびに、解凍後の SAClientUtil_<version>_<os>.zip ファイルの ¥bin ディレクトリーを使用して、すべてのコマンドを修飾しなければなりません。
    重要: CLI または Microsoft Visual Studio を使用して .NET をスキャンする予定の場合、以下の 3 つの .bat ファイル (解凍された SAClientUtil_<version>_<os>.zip ファイル内にある) を管理者として実行する必要があります。
    • bin¥register.bat
    • walalib¥dotnet.libs40¥RegisterInGAC.bat
    • walalib¥dotnet.libs35¥RegisterInGAC.bat

    これらのファイルは手動で実行できます。あるいは install-plugins.bat ファイルを管理者として実行できます (CLI のみ使用する場合、install-plugins.bat を実行しますが、静的分析プラグインのインストールを求めるプロンプトが出されたら no を入力します)。

    これらのファイルを管理者として実行しなかった場合、CLI を使用するたびにコンソールに警告メッセージが表示されます。このメッセージは、Java ターゲットをスキャンする場合は無視できます。

    ヒント: PATH を変更した後、コマンド・プロンプトで appscan version (Windows) または appscan.sh version (Linux および macOS) を発行してください。Static Analyzer クライアント・ユーティリティー バージョン、ホーム、および他の情報が返される場合、 PATH は正しく設定されています。
  4. 静的分析プラグインを、Eclipse、Microsoft Visual Studio (Windows の場合のみ)、IntelliJ IDEA、または Maven に追加する場合、 解凍された クライアント・ユーティリティー で、install-plugins.bat (Windows) または install-plugins.sh (Linux) を見つけて実行します。Windows では、管理者として以下のファイルを実行してください。プラグインを Linux または macOS 上の Eclipse に追加する予定の場合、インストール先の Eclipse インスタンスを変更できる権限を持つアカウントにログインしていることを確認してください。
    1. デフォルトでは、クライアント・ユーティリティー は IBM Cloud Marketplace で ASoC サービスに接続します。 代わりに、IBM Bluemix® のサービスに接続する場合、プロンプトが出されたら yes を入力します。
    2. Eclipse のソース・コードをスキャンする予定の場合、インストーラーのプロンプトに従います。Eclipse プラグインをインストールすることを選択した場合、Eclipse のインストール・ロケーションを入力するよう求めるプロンプトが出されます。eclipse.exe (Windows) または eclipse (Linux および macOS) ファイルを含むディレクトリーの絶対パスを入力してください。
    3. Maven for IRX ファイル生成を使用する予定場合は、インストーラーのプロンプトに従います。
    4. IntelliJ IDEA のソース・コードをスキャンする予定の場合、インストーラーのプロンプトに従います。IntelliJ IDEA プラグインをインストールすることを選択した場合、IntelliJ IDEA 構成ディレクトリーの場所を入力するよう求めるプロンプトが出されます。このディレクトリーの絶対パスを入力してください。
    5. Windows のみ: Visual Studio のソース・コードをスキャンする予定の場合、インストーラーのプロンプトに従います。 Visual Studio プラグインをインストールすることを選択した場合、プラグインのインストール先となる Visual Studio のバージョンを選択するよう求めるプロンプトが出されます。 プラグインのインストール先となる Visual Studio バージョンに該当するバ ージョンを入力します。
    6. 任意のキーを押してコマンド・プロンプトを閉じます。
    7. プラグインをインストールした IDE では、IDE を再始動します。
    8. Linux のみ: コマンド・プロンプトで、クライアント・ユーティリティーを含むディレクトリーを参照し、ご使用のシェル・リソース・ファイル (例えば .bash_profile または .shrc) に以下の行を追加します。 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export LD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${LD_LIBRARY_PATH}

      ここで <client_utility> は。クライアント・ユーティリティーへのパスです。 これらの行を追加したら、静的分析プラグインを使用する前にシェルを再ロードしてください。

    9. macOS のみ: コマンド・プロンプトで、クライアント・ユーティリティーを含むディレクトリーまでブラウズし、ご使用のシェル・リソース・ファイル ( 例えば .bash_profile または .shrc) に以下の行を追加します。 
      export APPSCAN_INSTALL_DIR=<client_utility>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export DYLD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${DYLD_LIBRARY_PATH}

      ここで <client_utility> は。クライアント・ユーティリティーへのパスです。 これらの行を追加したら、静的分析プラグインを使用する前にシェルを再ロードしてください。

      重要: シェル・リソース・ファイルの環境変数の設定は、端末で起動されるアプリにのみ影響します。 そのため、統合開発環境 (IDE) を起動するためには、静的分析を実行したい端末ウィンドウを使用する必要があります。

次のタスク

プロキシーの背後にあるコンピューターで クライアント・ユーティリティー を実行している場合 - 以下のいずれかの方法を使用して、クライアント・ユーティリティー がクラウドに接続できるようにプロキシーを指定します。

  • クライアント・ユーティリティー CLI およびサポートされる統合開発環境 (IDE): プロキシーが自動的に認識されるように、このグローバル変数またはシステム環境変数を設定します。
    • Windows: APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>
    • Linux および macOS: APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"

    ここで <proxy> はプロキシー・サーバーのホスト名、<port> はプロキシー・サーバーが使用するポート番号です。

    あるいは、CLI または IDE (コマンド・プロンプトまたは端末から起動) を使用するたびに、以下のコマンドを発行してプロキシーを使用するようにクライアント・ユーティリティー を設定できます。

    • Windows: set "APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
    • Linux および macOS: export APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
  • Maven: 静的分析プラグインを Maven に追加している場合、プロパティーを MAVEN_OPTS 環境変数全体に追加するか、プロパティーを各コマンドに追加することができます。

Static Analyzer クライアント・ユーティリティー の削除

このタスクでは、統合開発環境 (IDE) から Static Analyzer クライアント・ユーティリティー プラグインを削除する方法を説明します。 また、クライアント・ユーティリティー を削除する手順も示します。 この手順を実行する必要があるのは、ご使用のシステムから Static Analyzer クライアント・ユーティリティー コンポーネントを削除する予定がある場合のみです。これらのコンポーネントを新規 クライアント・ユーティリティー バージョンにアップグレードする場合、プラグイン・インストールによって自動的に古いコンポーネントが削除された後、新規バージョンがインストールされます。

コマンド・ライン・インターフェース (CLI) を使用した IRX ファイルの生成

ファイルの分析を開始できるようにするには、IRX ファイルを生成する必要があります (このファイルをクラウドにサブミットします)。CLI を使用したい場合、以下の手順に従ってファイルを作成します。

手順

IRX ファイルを生成するには、次のようにします。

  1. 新しいコマンド・プロンプト・ウィンドウを開くようにします (クライアント・ユーティリティーのインストールに使用したコマンド・プロンプト・ウィンドウは使用しないでください)。
  2. コマンド・プロンプトで、コマンドを発行して、ファイルが含まれるロケーションにディレクトリーを変更します。 サポート対象ファイル・タイプについては、言語サポート を参照してください。 Eclipse で Java プロジェクトをスキャンしている場合は、ディレクトリーを Eclipse ワークスペースに変更してください。
    注:
    • IRX ファイルの生成に構成ファイルを使用する場合、このステップは不要です。CLI での IRX ファイル生成の構成を参照してください。
    • CLI を使用して、Eclipse ワークスペース内の個々のプロジェクトをスキャンすることはできません。 ワークスペース全体のみをスキャンできます。個々のプロジェクトをスキャンしたい場合は、Eclipse 内に IRX ファイルを生成する必要があります。
    • .dll ファイルおよび .exe ファイルは有効な .NET アセンブリーでなければなりません。
  3. appscan prepare (Windows) または appscan.sh prepare (Linux および macOS) コマンドを発行します。 このコマンドは、構成コマンド (Windows) または 構成コマンド (Linux および macOS) トピックで示されるオプションと共に使用されます。 例えば、-d オプションおよび -n オプションを使用して、 IRX ファイルを保存するための特定のディレクトリーを指定し、特定のファイル名でファイルを保存することができます。さらに、このコマンドは、構成ファイルを指定して実行することができます。このファイルには、完全な IRX ファイルを生成する上で有用となる追加の情報を指定します (「CLI での IRX ファイル生成の構成」参照)。
    ヒント: デフォルトでは、IRX ファイルの生成中にサード・パーティーの Java および .NET コードはスキャンされません。サード・パーティーの Java と .NET の除外の管理 の手順に従って、サード・パーティーのコードの除外設定を変更できます。 サード・パーティー・コードを組み込むには、prepare コマンドの実行時に -t オプションまたは --thirdParty オプションを指定してください。

    サード・パーティー・コードがスキャンに標準的に含まれるようにする開発者の場合、この設定を使用してサード・パーティー・コードをスキャンに含める必要があります。

    注: ソース・コードをスキャンしたり、IRX ファイルを生成する場合、最新の クライアント・ユーティリティー への更新に関するメッセージが表示される場合があります。クライアント・ユーティリティー とクラウド・サービスのバージョンの互換性を参照してください。
  4. コマンドの実行後、クライアント・ユーティリティー は、コマンドが実行されたディレクトリー内、およびそのすべてのサブディレクトリー内にある、サポートされるすべてのファイルの情報を収集します。このディレクトリー内に IRX ファイルが作成されます。このファイルの名前は、コマンドが実行されたディレクトリーに基づいて付けられ、 日時タイム・スタンプが組み込まれます (異なるファイル名を指定する構成オプションを使用しなかった場合)。
  5. 分析のために IRX ファイルをクラウドにサブミットします。そのために、appscan queue_analysis コマンド (Windows) または appscan.sh queue_analysis コマンド (Linux および macOS) を発行します。 このコマンドは、「分析コマンド (Windows)」や「分析コマンド (Linux および macOS)」の各トピックにリストされているオプションを指定して使用します。

Maven プロジェクトの IRX ファイルの生成

ファイルの分析を開始できるようにするには、IRX ファイルを生成する必要があります (このファイルをクラウドにサブミットします)。Maven プロジェクトをスキャンしたい場合、以下の手順に従ってファイルを作成します。

手順

IRX ファイルを生成するには、次のようにします。

  1. 新しいコマンド・プロンプト・ウィンドウを開くようにします (クライアント・ユーティリティーのインストールに使用したコマンド・プロンプト・ウィンドウは使用しないでください)。
  2. コマンド・プロンプトでディレクトリー変更コマンドを実行して、Maven プロジェクトが保存されている場所 (pom.xml ファイルが入っているフォルダー) に移動します。
  3. 以下のいずれかのコマンドを実行します。
    • Maven プロジェクトがまだビルドされていない場合、mvn package com.ibm.appscan:appscan-maven-plugin:prepare を実行します。このコマンドの実行後、 Maven プロジェクトの target ディレクトリーに IRX ファイルが作成されます。 このファイルの名前は、 最上位の Maven プロジェクト (pom.xml ファイル内のプロジェクトの名前) に基づいて付けられます。これは、異なるファイル名を指定するために構成オプションを使用しなかったことを前提としています。
    • Maven プロジェクトが既にビルド済みの場合、mvn com.ibm.appscan:appscan-maven-plugin:prepare を実行します。このコマンドの実行後、 Maven プロジェクトの target ディレクトリーに IRX ファイルが作成されます。 このファイルの名前は、 最上位の Maven プロジェクト (pom.xml ファイル内のプロジェクトの名前) に基づいて付けられます。これは、異なるファイル名を指定するために構成オプションを使用しなかったことを前提としています。
      注: Maven プロジェクトに、同じリアクター・ビルド内の他のプロジェクトに対するコンパイル時依存関係がある場合、既存の Maven のバグ (https://issues.apache.org/jira/browse/MNG-3283) が原因で mvn com.ibm.appscan:appscan-maven-plugin:prepare コマンドが失敗します。この問題を回避するには、以下のステップのいずれかを実行します。
      • クライアント・ユーティリティー prepare コマンドを実行する前に、mvn package コマンドの代わりに mvn install コマンドを実行します。
      • mvn package コマンドと クライアント・ユーティリティー prepare コマンドを同時に発行します (mvn package com.ibm.appscan:appscan-maven-plugin:prepare を発行)。
    • 異なるディレクトリーに IRX ファイルが作成されるようにしたい場合、 または異なるファイル名にしたい場合は、次のように -Doutput オプションを指定してコマンドを実行します。

      Windows では、次のコマンドを発行します。

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<output_location>¥<filename>.irx

      Linux および macOS では、次のコマンドを発行します。

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<output_location>/<filename>.irx

      ここで、<output_location> はファイルを保存したいディレクトリー、filename はファイルの名前です。

    • クライアント・ユーティリティーには Tomcat JSP コンパイラーが含まれています。この JSP コンパイラーは、IRX ファイル生成中にデフォルトで使用されます。WAR ファイルまたは EAR ファイルの IRX ファイルを生成していて、 異なる JSP コンパイラーが使用されるようにしたい場合は、-DjspCompiler オプションを指定してコマンドを実行します。以下に例を示します。
      mvn com.ibm.appscan:appscan-maven-plugin:prepare -DjspCompiler=<jsp_compiler>

      ここで、<jsp_compiler> は、 使用したい JSP コンパイラーへのパスです (サーバーのルート・ディレクトリーを指定してください)。

    ヒント: デフォルトでは、IRX ファイルの生成中にサード・パーティー・コードはスキャンされません。サード・パーティーの Java と .NET の除外の管理 の手順に従って、サード・パーティーのコードの除外設定を変更できます。 サード・パーティー・コードを組み込むには、-DthirdParty オプションを指定してこのコマンドを実行します。

    サード・パーティー・コードがスキャンに標準的に含まれるようにする開発者の場合、この設定を使用してサード・パーティー・コードをスキャンに含める必要があります。

    注: ソース・コードをスキャンしたり、IRX ファイルを生成する場合、最新の クライアント・ユーティリティー への更新に関するメッセージが表示される場合があります。クライアント・ユーティリティー とクラウド・サービスのバージョンの互換性を参照してください。
  4. 分析のために IRX ファイルをクラウドにサブミットします。そのために、appscan queue_analysis コマンド (Windows) または appscan.sh queue_analysis コマンド (Linux および macOS) を発行します。 このコマンドは、「分析コマンド (Windows)」や「分析コマンド (Linux および macOS)」の各トピックにリストされているオプションを指定して使用します。

構成ファイルの使用法

どのターゲットを IRX ファイルの生成時に包含または除外するのかを指示するために、構成ファイルを使用できます。また、 ビルド成果物を含むディレクトリーも指定でき、IRX ファイル生成中に使用される必要のあるビルド情報を指定することもできます (例えば、この設定を使用して、特定の JDK コンパイラーまたは JSP コンパイラーを指定できます)。

IRX ファイルを生成するよう appscan prepare (Windows)またはappscan.sh prepare (Linux および macOS) コマンドを発行するたびに、現行ディレクトリー内に appscan-config.xml ファイルがあるかどうかが Static Analyzer クライアント・ユーティリティー によって自動的にチェックされます。ファイルが検出された場合は、そのファイルが自動的に使用されます。

異なるディレクトリーに構成ファイルがあるか、異なるファイル名である場合は、appscan prepare または appscan.sh prepare コマンドを実行するときに -c オプションを組み込むことによってそのファイルを使用できます。この場合、(Windows) の場合は appscan prepare -c <configuration_file> を実行し、(Linux および macOS)の場合は appscan.sh prepare -c <configuration_file> を実行します。 ここで、<configuration_file> は、構成ファイルの絶対パスおよびファイル名です。

構成ファイル・テンプレートおよび設定

構成ファイルを作成するときには以下のテンプレートを使用してください。

<Configuration>
  <Targets>
    <Target outputs-only="true/false" path="scan_target_path">
      <CustomBuildInfo build_info="info"/>
      <Include>string_pattern</Include>
      <Exclude>string_pattern</Exclude>
    </Target>
  </Targets>
</Configuration>

Targets エレメントを使用して、IRX ファイルの生成中にスキャンするターゲットを指示できます。ターゲット (Target) の属性および子エレメントは次のように使用されます。

  • Target エレメントの outputs-only 属性は、ディレクトリーをスキャンしたい場合で、かつ、クライアント・ユーティリティー がビルド出力ファイル (例えば、.jar ファイル、.war ファイル、および .class ファイル) のみを検出するようにしたい場合にのみ使用されます。デフォルトでは、この属性は false に設定されます。これが意味するのは、クライアント・ユーティリティー はディレクトリーを検索して、そのディレクトリーがターゲット (例えば、アプリケーション・サーバーまたは Eclipse ワークスペース) かどうか、または、そのディレクトリーに、ビルド・スクリプト、Maven .pom ファイル、make ファイルなどの項目が含まれているかどうかを判別するということです。スキャン・ターゲットが単純なディレクトリーとして扱われるようにして、 出力ファイルのみを見つけるようにしたい場合、 Target エレメントに outputs-only="true" を指定します。

    例えば、<Target outputs-only="false" path="C:¥Tomcat"> を指定すると、クライアント・ユーティリティー は、ターゲットは Tomcat アプリケーション・サーバーだと想定し、そこにデプロイされている .war ファイルを検索します。しかし、<Target outputs-only="true" path="C:¥Tomcat"> を指定すると、クライアント・ユーティリティー はそのロケーションをディレクトリーとして扱い、その中にあるすべてのビルド出力ファイルを探します。

  • Target エレメントの path 属性を使用して、 スキャン・ターゲットへのパスまたはスキャン・ターゲットのディレクトリーを指定します (<scan_target_path>)。ディレクトリーが指定されると、 そのディレクトリーのすべてのサブディレクトリーがスキャン中に包含されます。

    例えば、<Target outputs-only="false" path="C:¥WebSphere¥AppServer85¥profiles¥AppSrv01"> と指定すると、クライアント・ユーティリティー は、AppSrv01 プロファイル内のすべてのデプロイされた .ear ファイルを探します。<Target outputs-only="false" path="C:¥WebSphere¥AppServer85"> と指定すると、クライアント・ユーティリティー は、すべてのプロファイル内のすべてのデプロイされた .ear ファイルを探します。

  • 以下に示す、オプションの Target サブエレメントも指定できます。
    • CustomBuildInfo エレメントを使用して、属性を設定できます。テンプレートの 
      <CustomBuildInfo build_info="info"/>
      の部分を使用して、ターゲット言語に応じてビルド情報を指定できます。 一部の言語では、複数の属性を設定できます。例えば、ターゲット言語に応じて <CustomBuildInfo build_info_1="info_1" build_info_2="info_2" build_info_3="info_3"/> を設定することができます。
      • Java の場合、以下の属性を設定できます。
        <CustomBuildInfo additional_classpath="dependency_path"
  jdk_path="JDK_path" jsp_compiler="JSP_compiler_path"/>
        • additional_classpath 属性を使用して、追加のクラスパスを指定します。 Windows では、複数のクラスパスはセミコロンで区切ります。 Linux および macOS では、複数のクラスパスはコロンで区切ります。
        • jdk_path 属性を使用して、JDK がインストールされた場所へのパスを指定します。
        • JSP コンパイラーのパスを指定するには、jsp_compiler 属性を使用します。例えば、jsp_compiler="C:¥Tomcat" とします。
      • .NET (Windows のみ) の場合、以下の属性を設定できます。
        <CustomBuildInfo reference="assembly_reference"
  configuration="build_configuration"/>
        • reference 属性を使用してアセンブリー参照を追加します。 複数の参照はセミコロンで区切ります。
        • configuration 属性を使用して、Visual Studio ソリューション・ディスカバリーのビルド構成を組み込みます。
      • C/C++ (Windows のみ) の場合、以下の属性を設定できます。
        <CustomBuildInfo configuration="build_configuration"
  include_paths="include_directories" macros="macros" compiler_opts=/>
        • configuration 属性を使用して、ビルド構成を組み込みます。
        • include_paths 属性を使用して、インクルード・パスを指定します。 複数のインクルード・パスはセミコロンで区切ります。
        • macros 属性を使用して、マクロを組み込みます。 複数のマクロはセミコロンで区切ります。
        • compiler_opts 属性を使用して、コンパイラー・オプションを指定します。 複数のオプションはセミコロンで区切ります。
    注: これらの属性で設定された値は、サブターゲットに継承されます。例えば、ターゲットが EAR ファイルで、このファイルに WAR および JAR の各サブターゲットが含まれている場合、 この WAR と JAR は、これらの属性を使用する EAR ファイルに対して設定したのと同じ値を持っていると想定されます。
  • Include 子エレメントを使用して、IRX ファイルの生成時に包含するファイル・パターン (<string_pattern>) を指定します。Include 動作は、『ターゲットの包含と除外の動作』セクションに概説されているように、ターゲット・タイプによって異なります。複数の include パターンを指定するには、それぞれのパターンを個々の <Include></Include> タグに追加します。 例: 
    <Include>string_pattern_1</Include>
<Include>string_pattern_2</Include>
    注: 競合する include パターンと exclude パターンを指定した場合、exclude パターンが優先されます。
  • Exclude 子エレメントを使用して、IRX ファイルの生成時に除外するファイル・パターンを指定します。Exclude 動作は、『ターゲットの包含と除外の動作』セクションに概説されているように、ターゲット・タイプによって異なります。複数の exclude パターンを指定するには、それぞれのパターンを個々の <Exclude></Exclude> タグに追加します。
    注: 競合する include パターンと exclude パターンを指定した場合、exclude パターンが優先されます。

構成ファイルの例

<Configuration>
  <Targets>
    <Target path="C:¥test¥MyApplication.ear">
      <CustomBuildInfo additional_classpath="C:¥java¥lib;C:¥java2¥lib"
        jdk_path="C:¥Program Files¥Java¥jdk1.7" jsp_compiler="C:¥Tomcat"/>
      <Include>ShoppingCart.war</Include>
      <Include>*Account.jar</Include>
      <Exclude>thirdPartyJars/</Exclude>
      <Exclude>*test*</Exclude>
    </Target>
  </Targets>
</Configuration>
この構成ファイルでは以下のことが行われます。
  • C:¥test¥MyApplication.ear はターゲットです。
  • C:¥java¥lib および C:¥java2¥lib で、クラス・ファイルの依存関係がすべて検索されます。この依存関係はコンパイルに必要です。
  • C:¥Program Files¥Java¥jdk1.7 が依存関係を見つけるために使用されます。
  • C:¥Tomcat コンパイラーが JSP コンパイル用に使用されます。Static Analyzer クライアント・ユーティリティー に同梱されている Apache Tomcat バージョン 7 アプリケーションの代わりです。
  • IRX ファイルの生成時に、名前が *Account.jar で終わるファイルがすべて包含されます。同様に、ShoppingCart.war ファイルも包含されます。 ファイル名に test というストリングがあるファイルがすべて除外されます。同様に、EAR ファイルの thirdPartyJars/ に含まれるものもすべて除外されます。 競合が発生した場合は、exclude パターンが優先されます。例えば、EAR ファイルに testCustomerAccount.jar ファイルが含まれている場合、test exclude パターンは *Account.jar include パターンより優先されるため、testCustomerAccount.jar ファイルは除外されます。

統合開発環境でのスキャン

以下の手順では、静的分析プラグインを Eclipse、IntelliJ IDEA、または Visual Studio 統合開発環境 (IDE) にインストールした後、そのプラグインを使用してソース・コードをスキャンする方法を説明します。Eclipse および IntelliJ IDEA では、Java プロジェクトをスキャンできます。また Visual Studio では .NET (C#、ASP.NET、VB.NET) をスキャンできます。

始める前に

デフォルトでは、Java および .NET をスキャンする場合にサード・パーティー・コードはスキャンに含まれません。 サード・パーティーの Java と .NET の除外の管理 の手順に従って、サード・パーティーのコードの除外設定を変更できます。
  • Eclipse、IntelliJ IDEA、または Visual Studio でスキャンする場合、サード・パーティー・コードを含めるには、以下のいずれかの方法を使用します。
    • ID を開始するまえに、以下のグローバル変数またはシステム環境変数を設定します。 
      APPSCAN_OPTS=-DthirdParty
    • IDE を使用するたびに、IDE の開始前に以下のコマンドを発行します。 
      set APPSCAN_OPTS=-DthirdParty
  • あるいは、Eclipse でのスキャン時に以下を実行します: Eclipse を始動する前に eclipse.ini ファイルを変更し、-vmargs セクションに -DthirdParty が含まれるようにします。
  • あるいは、IntelliJ IDEA でのスキャン時に以下を実行します。 IntelliJ IDEA を始動する前に、IntelliJ IDEA インストール済み環境の bin ディレクトリーで、idea.exe.vmoptions ファイルを変更します。 -DthirdParty をファイルに追加してからファイルを保存します。

サード・パーティー・コードがスキャンに標準的に含まれるようにする開発者の場合、この設定を使用してサード・パーティー・コードをスキャンに含める必要があります。

手順

ソース・コードをスキャンし、評価またはレポートを開くには、次のようにします。

  1. プラグインが IDE にインストールされていることを確認します。 インストール中に IDE が開いていた場合は、再始動してください。
  2. スキャンする項目を選択します。
    • Eclipse では、スキャンするプロジェクト (複数可) を選択します。 Eclipse ワークスペース全体をスキャンするには、すべてのプロジェクトを選択します。
    • IntelliJ IDEA では、スキャンするプロジェクト・モジュールを選択します。
    • Visual Studio では、スキャンするソリューション、プロジェクト、または Web サイトを選択します。
  3. 選択項目を右クリックし、「セキュリティー分析」 > 「静的分析の実行」をクリックします。
    注: ソース・コードをスキャンしたり、IRX ファイルを生成する場合、最新の クライアント・ユーティリティー への更新に関するメッセージが表示される場合があります。クライアント・ユーティリティー とクラウド・サービスのバージョンの互換性を参照してください。
  4. スキャンの準備の完了後、サービスにまだログインしていない場合には、「ログイン」ダイアログ・ボックスが開きます。開くダイアログ・ボックスは、install-plugins.bat (Windows) または install-plugins.sh (Linux および macOS) インストーラーの起動後に行われる接続設定により決まります。 ダイアログ・ボックスに、サービス資格情報を入力します。
    • Cloud Marketplace に接続している場合: Application Security on Cloud サービスで API キーを作成した場合、「Key Id」 と「Key Secret」を受け取ります。それらの値を「ID」フィールドと「Secret」フィールドに入力します。 API キーをまだ生成していない場合は、ダイアログ・ボックス内のリンクに従ってキーを作成してください。
    • Bluemix に接続している場合: サービス資格情報に指定されている「Binding ID」と「Password」を入力します。

      Bluemix サービス資格情報を判別するには、サービス・ダッシュボードの左側のナビゲーション・ペインで「サービス資格情報」を選択します。外部アプリが Bluemix サービスを使用できるようにするを参照してください。

    サービスにログインすると暗号化された鍵ファイルが作成されます。 このトークン・ファイルは、それ以降、ASoC サービスと対話するときに他のアクションによって参照されます。

静的分析 IRX ファイルの生成時にエラーまたは警告が表示される

症状

IRX ファイルの生成でエラーまたは警告が出されたり、IRX ファイルの生成が失敗したりすることがあります。IRX ファイルの生成に失敗すると、生成されたファイルの名前には _failed が付加されます (例: my_irx_file_failed.irx)。 この状況が発生した場合、IRX ファイルの生成のトラブルシューティングを行うための各種のオプションを使用できます。

問題の解決

  • IRX ファイル生成のトラブルシューティングのために構成ファイルを使用する

    構成ファイルを使用せずに IRX ファイルを生成し、 ファイル生成が完了しない (またはエラーで完了する) 場合、それ以降のスキャンに使用できる一時構成ファイルが Static Analyzer クライアント・ユーティリティー によって生成されます。 このファイルは、ターゲットに関する既知の情報が入った Targets エレメントが含まれるという点で、構成ファイル・テンプレートに似たものになります。 ただし、このファイルには <TargetSettings> エレメントも含まれ、 このエレメントを変更して情報を追加すると、クライアント・ユーティリティー が完全な IRX ファイルを生成できるようになります。 このファイルは appscan-config.xml と名付けられ、 現行ディレクトリーに保存されます。これは、このファイルが自動的に次のスキャンに使用されることを意味します。

    例えば、IRX ファイル生成のターゲットに場所を探索できない外部依存関係が含まれていると、次のようなメッセージが出されることがあります。

    C:¥myTargetDirectory>appscan prepare
An IRX file was created, but it may be incomplete.
C:¥myTargetDirectory¥myTarget.jar requires the following packages:
      org.apache.jasper.*
An appscan-config.xml file was created in: C:¥myTargetDirectory¥appscan-config.xml
Please update the file and rerun the command.

    生成される appscan-config.xml ファイルは、次のようなものになります。

    <Configuration>
  <Targets>
    <Target outputs-only="true" path="C:¥myTargetDirectory"/>
  </Targets>
  <Settings/>
  <TargetSettings>
    <Target path="C:¥myTargetDirectory¥myTarget.jar"/>
      <!--Please add the path of the following dependencies to the attribute "additional_classpath" below, separated by semi-colon.
			  org.apache.jasper.*-->
      <CustomBuildInfo additional_classpath=""/>
    </Target>
  </TargetSettings>
</Configuration>

    この問題を解決するには、<CustomBuildInfo additional_classpath=""/> を、クラスパスの org.apache.jasper.* クラスを含むように更新し、appscan-config.xml ファイルを保存して、appscan prepare (Windows) コマンドまたは appscan.sh prepare (Linux および macOS) コマンドを再実行します。JAR ファイルへのパスを指定する際、標準の Java クラスパスの構文がサポートされています (例えばワイルドカードはサポートされています)。

    既に構成ファイルを使用しているときに似たような問題が発生する場合は、 問題を示すメッセージが出されます。 そのメッセージを使用して既存の構成ファイルを更新し、その後でコマンドを再実行してください。

    構成ファイルについては、CLI での IRX ファイル生成の構成に説明があります。

  • デバッグ・モードで IRX ファイルを生成する

    CLI から IRX ファイルを生成している場合、 debug オプションを指定して appscan prepare (Windows) または appscan.sh prepare (Linux および macOS) コマンドを実行できます。これにより、より多くのログ・ファイルが生成されます。

  • IRX ファイルの生成に関連するログ・ファイル

    エラーまたは警告を確認するには、 生成されるログ・ファイルを調べます。ログ・ファイルは、.zip アーカイブ・ファイルに保存されます。デフォルトで、このファイルの保存先は、ご使用の IRX が保存されているのと同じ場所になります。あるいは、appscan prepare (Windows) コマンドまたは appscan.sh prepare (Linux および macOS) コマンドの -l オプションを使用して別の場所を指定することもできます。.zip ファイルの名前は、生成される IRX ファイルの名前に基づいて付けられます。

Application with id x does not exist エラーでのセキュリティー・スキャンの失敗

症状

Jenkins プラグインまたは IDE プラグインあるいはコマンド・ライン・インターフェース (CLI) を使用するスキャンの実行が次のエラーで失敗します。

Application with id x does not exist
Failed to run the security scan: Problems occurred submitting the IRX file to the analysis service.

原因

以前は、Application Security on Cloud は、app_id に基づいてアプリケーションを関連付けました。最近の更新で、この関連付けが app_id から guid に変更されました。ほとんどの場合、ASoC サービスは、この更新に基づいてアプリケーションを再度関連付けるようにプロンプトを出します。 Jenkins プラグインおよび IDE プラグイン、および CLI の場合は、追加ステップを実行する必要があります。

問題の解決

Jenkins プラグインの場合、Jenkins 自動化サーバーへのセキュリティー分析の追加 の説明に従って Jenkins プラグイン構成を再保存してから、スキャンを再試行します。

IDE プラグインの場合、プロンプトが表示されたらスキャン対象として同じアプリケーションを再選択して、「OK」をクリックします。

CLI の場合、新規の guid で app_id を置き換えます。guid は、固有の 32 文字の ID です。例えば、33c3281a-6d7c-e711-b334-002590731623 などです。appscan list_apps コマンドを使用して、各アプリケーションの guid を判別し、スクリプトあるいはコマンド・ラインの queue_analysis コマンドで app_id をその guid に置き換えます。

サポート対象の Java ソース・コードのアノテーション

Java をスキャンするために静的分析を使用する際は、@ValidatorMethod、@CallbackMethod、および @SuppressSecurityTrace のメソッド・レベル・アノテーションがサポートされています。

  • アノテーションの使用
  • @ValidatorMethod
  • @SuppressSecurityTrace
  • @CallbackMethod

アノテーションの使用

アノテーションは、以下のステップを実行することにより使用できます。

  1. アノテーションのサポートはデフォルトで有効になっています。アノテーション .jar ファイルは、解凍された SAClientUtil_<version>_<os>.zip ファイルの ¥annotation ディレクトリー内にあります。
  2. プリコンパイル・クラス、.war ファイル、または .jar ファイルをスキャンする場合、アノテーション付きソースが含まれる Java プロジェクトに com.ibm.security.annotation.jar を追加して、プロジェクトを再作成します。
  3. 静的分析プラグインがインストールされている統合開発環境 (IDE) でスキャンする場合、Java クラス・パスに com.ibm.security.annotation.jar を追加します。
  4. 静的分析 CLI を使用している場合、構成ファイルを作成し、その中に CustomBuildInfo additional_classpath="dependency_path" を設定して dependency_path に com.ibm.security.annotation.jar ファイルの場所が含まれるようにします。

アノテーションは、スキャン前にソース・コードに追加できます。あるいは、スキャン後や誤検出を特定して除外するためのトリアージ中に追加することもできます。静的分析プラグインがインストールされている IDE でスキャンする場合、スキャン・レポートでは「問題解決のヘルプ」ボタンが使用可能です。 このボタンを選択すると、アノテーションを追加できるポイントが表示されます。

アノテーションは、セキュリティー・アノテーション形式のナレッジをソース・コードに直接挿入できるようにするために提供されています。 アノテーションはコードの一部を安全と宣言するために使用する場合があるので、慎重に使用する必要があります。 セキュリティー脆弱性をスキャンするコードには、アノテーションを使用しないでください。 アノテーションを使用する場合、セキュリティー・アナリストは、解凍された SAClientUtil_<version>_<os>.zip ファイルの ¥config ディレクトリーにある irgen-engine.apsettings ファイル内の機能を無効にすることで、アノテーションを無視するように選択できます。このファイルで、以下の設定を見つけます。

<Setting 
  name="process_security_annotations"
  value="true"
  default_value="true"
  description="When turned on, method annotations will be processed - used by DotNet and Java LSPs"
  display_name="Process Security Annotations"
  type="bool"
/>

注: value="false" に設定されている場合、アノテーションを使用したい場合はこの設定を value="true" にします。

@ValidatorMethod

バリデーターは、入力データのチェックを行うメソッドであり、多くの場合、入力が有効か無効かを示す boolean 値を返します。 バリデーターを使用して入力を受け入れまたは拒否するのではなく、ユーザー入力を受け入れ可能なフォーマットに変更できます。 これらのメソッドは、サニタイザーと呼ばれます。

@SuppressSecurityTrace

このアノテーションが付けられたメソッドを経由するすべてのトレースは削除されます。 これは、特定グループのトレースを誤検出として、または重要度や注目度が低いものとして識別する場合に便利です。 このアノテーションを使用して、これらのトレースをフィルターで除外するか、乱雑になるのを防ぐためにトレースを非表示にすることができます。

@CallbackMethod

このアノテーションを使用してコールバックやアプリケーションへのエントリー・ポイントを特定します。 すべての引数は、汚染の伝搬元とみなされます。

バリデーターは、入力データのチェックを行うメソッドであり、多くの場合、入力が有効か無効かを示す boolean 値を返します。 バリデーターを使用して入力を受け入れまたは拒否するのではなく、ユーザー入力を受け入れ可能なフォーマットに変更できます。 これらのメソッドは、サニタイザーと呼ばれます。

[ValidatorMethod] 属性を使用することで、アプリケーション・ソース・コードのすべてのバリデーターとサニタイザーのメソッドを識別できます。 静的分析スキャン中に、この情報を使用して、これらのメソッドを通過するデータ・フローを除去するので、データは安全と見なされるようになります。

注: 現在、アノテーション付きメソッドの検証済みと見なされるパラメーターを指定するために提供されているものはありません。 静的分析のスキャン中、すべての入力パラメーターは検証済みとみなされます。

[SuppressSecurityTrace] または [SuppressSecurityTrace()]

この属性を持つメソッドを経由するすべてのトレースは削除されます。 これは、特定グループのトレースを誤検出として、または重要度や注目度が低いものとして識別する場合に便利です。 この属性を使用して、これらのトレースをフィルターで除外するか、乱雑になるのを防ぐためにトレースを非表示にすることができます。

[CallbackMethod] または [CallbackMethod()]

この属性を使用してコールバックやアプリケーションへのエントリー・ポイントを特定します。 すべての引数は、汚染の伝搬元とみなされます。

未解決の PHP include 式と不明な PHP クラスの構成

PHP 用に IRX ファイルを生成すると、構成ファイルが作成され、次回ファイルをスキャンするときに IRX 生成を構成するために使用できます。

構成ファイル php-scan-config.xml は、prepare コマンドが発行された場所に生成されます。 ファイルには <UserInputs> と <Php-Unknown-Configuration> の 2 つのセクションが含まれています。

<UserInputs> セクションには、PHP スキャン用に変更できる構成設定が含まれています。 このセクションは、サンプル値が事前に入力されています。 セミコロン (;) を使用して、その値に追加できます。 このセクションの設定は次のとおりです。

  • <RegularExpression>: これは、検索テキスト用に正規表現を指定できる拡張機能です。
  • <ReplaceText>: この設定は、単純なテキスト検索と置換に使用されます。 検索テキストはそのまま使用され、インクルード・テキスト内のどこかで検出されると、置換テキストに置き換えられます。
  • <Function>: この設定は、関数呼び出しを置き換えする場合に使用されます。 検索テキストは括弧なしの関数名でなければなりません。 検索テキストは、指定された名前とそれに続く括弧を使用して、関数呼び出しを検索できるように拡張される場合があり、括弧内に指定するものと照合されます。
  • <SimpleText>: 簡易化された置換テキスト。 これは、置換テキストが単一ストリングの場合に使用します。
  • <SimpleRegEx>: 簡易化された置換正規表現。 これは、置換正規表現が単一ストリングの場合に使用します。
  • <AdditionalIncludePaths>: この設定は、参照ファイルが含まれる include パスを追加するために使用します。

<Php-Unknown-Configuration> セクションには、前回のスキャンで検出された未解決の include 式および不明クラスがすべて含まれます。 この情報を使用して <UserInputs> セクションに入力します。

  • <Summary>: 前回スキャンの不明な include 式 (<NumberOfUnknownIncludeExprs>) および不明クラス (<NumberOfUnknownClasses>) の数を集計します。
  • <SourceFile>: 不明な include 式と不明クラスのすべてが、ソース・ファイル名のアルファベット順にリストされます。 ソース・ファイルごとに、ファイル・パス (<FilePath>) には、ソースの絶対パス、不明クラス (<UnknownClass>)、および不明な include 式 (<UnknownIncludes>) がリストされます。

不明クラス (<UnknownClass>) 情報

タグ 説明
<ClassName> ソース・コード内で参照されたクラス名。クラスが使用されるソース・ファイル (<FilePath>) の場所と共に示されます。 エディター内でこの場所を示して、不明クラスがどのように解決すべきかを確認できます。
<Status> クラスが Resolved か Not Resolved であるかを示します。
<ResolvedBy> 不明クラスが解決された方法を示します。 以下の値が含まれます。
  • AutoResolver: アプリケーションは、内部ヒューリスティックを使用して、ファイルを検出しました。
  • SearchReplace: 1 つ以上の検索ルールおよび置換ルールをインクルード・テキストに適用して、更新済みテキストを生成しました。

不明な include 式 (<UnknownIncludes>) 情報

タグ 説明
<IncludeText> ソース・コード内に、そのままの状態で存在する式テキスト。式が使用されるソース・ファイル (<FilePath>) の場所と共に示されます。 エディター内でこの場所を示して、不明な include 式がどのように解決すべきかを確認できます。
<UpdatedText> 前回のスキャン中に使用された更新済みテキスト。
<Status> 式が Resolved か Not Resolved であるかを示します。
<ResolvedBy> 更新済みテキストが生成された方法を示します。 以下の値が含まれます。
  • AutoResolver: アプリケーションは、内部ヒューリスティックを使用して、ファイルを検出しました。
  • SearchReplace: 1 つ以上の検索ルールおよび置換ルールをインクルード・テキストに適用して、更新済みテキストを生成しました。
  • SearchReplace+AutoResolver: 1 つ以上の検索および置換ルールをインクルード・テキストに適用して、更新済みテキストを生成しました。その後、更新済みテキストに内部ヒューリスティックを適用して、ファイルを検出しました。
  • SearchReplace+IncludePath: 1 つ以上の検索および置換ルールをインクルード・テキストに適用して、更新済みテキストを生成しました。その後、更新済みテキストを include パス上のディレクトリーと結合して、ファイルを検出しました。
  • SearchReplace+RelativeDir: 1 つ以上の検索および置換ルールをインクルード・テキストに適用して、更新済みテキストを生成しました。その後、include 式が含まれるファイルのソース・ディレクトリーを基準として、ファイルを検出しました。

サード・パーティーの Java と .NET の除外の管理

デフォルトでは、IRX ファイルの生成中にサード・パーティーの Java と .NET のコードはスキャンされません。このトピックの手順に従って、除外されるサード・パーティーのコードを管理することができます。

このタスクについて

IRX ファイルをご使用のコード用に生成する場合、分析する必要がないコードに関する情報と検出結果は評価に含まれないように、サード・パーティーのコードは除外されます。 除外する対象を変更するには、このトピックの手順に従ってください。 そうではなく、スキャン時にサード・パーティーのコードを組み込みたい場合には、以下のトピックを参照してください。

  • コマンド・ライン・インターフェース (CLI) を使用した IRX ファイルの生成
  • Maven プロジェクトの IRX ファイルの生成
  • 統合開発環境でのスキャン

手順

  1. 抽出された SAClientUtil_<version>_<os>.zip ファイル (ここで <version> は クライアント・ユーティリティー の現行バージョン) の config ディレクトリーを見つけます。
  2. このディレクトリーで、dot_net.exclusions (.NET) ファイルまたは java.exclusions (Java) ファイルを見つけます。
    注: このディレクトリー内の他の .exclusions ファイルは試験用であり、変更してもサード・パーティーの除外に影響を与えません。
  3. テキスト・エディターでファイルを開くと、次のフォーマットで XML が表示されています。 
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<Exclusions>
  <packages>
    <package name="package_name"/>
  </packages>
  ...
</Exclusions>
  4. ファイルを編集して除外するパッケージを追加します。
    注:
    • 指定するパッケージ名は、同じ接頭部を持つすべてのパッケージと一致するとみなされます (後ろにワイルドカードが付くものとして処理されます）。例えば、<package name="com.mycompany.common"/> を除外対象として追加した場合、<package name="com.mycompany.common.action1"/> と <package name="com.mycompany.common.action2"/> が除外されます。
    • サード・パーティーの除外に影響するのは、<packages> コンテナーに追加される項目だけです。 クラスとメソッドの追加は除外に影響しません。
  5. このファイルを保存して、ご使用のソース・コード用の IRX ファイルを生成します。

次のタスク

除外ファイルを変更する場合、以下のベスト・プラクティスに従う必要があります。
  • 変更された除外ファイルのバックアップを保持します。 特に、新規バージョンの Static Analyzer クライアント・ユーティリティー にアップグレードする場合、更新されていない除外ファイルのバージョンをダウンロードすることに注意してください。 クライアント・ユーティリティー を更新する場合、変更された除外ファイルのコピーを保持して、それを新規 クライアント・ユーティリティー にコピーします。
  • 変更内容は、XML 項目の単一ブロックで保持し、それらを必要に応じて簡単にコピーして貼り付けられるようにします。

説明:

分析のために IRX ファイルをクラウドにサブミットします。スキャンが完了すると、(分析サービスにログインする際に使用したアカウントと関連付けられた E メール・アドレスで) E メール通知を受信します。 この E メールにはクラウドへのリンクが含まれており、ログインしてスキャンをダウンロードできるようになっています。

    注: このオプションが必要なのは、以下の記述のいずれかまたは両方が当てはまる場合のみです。
    • コマンドを発行するディレクトリーに複数の IRX ファイルが含まれている。 ディレクトリーに含まれている IRX ファイルが 1 つのみであれば、 -f オプションが使用されていない場合はそのファイルがサブミットされます。
    • コマンドを発行するディレクトリーに IRX ファイルが 1 つも含まれていない。 この場合、 -f オプションを使用して、サブミットする IRX ファイルのパスおよびファイル名を指定する必要があります。
  • -n: -n <scan_name> と指定します。ここで、<scan_name> は、クラウドで実行されるスキャンの名前です。
  • -a (IBM Cloud Marketplace のみ): IBM Cloud Marketplace の Application Security on Cloud サービスに接続している場合、クラウドにサブミットする IRX ファイルは、既存の Application Security on Cloud アプリケーションと関連付ける必要があります。 このオプションを使用する際は、-a <app_id> と指定します。ここで <app_id> は、関連付けるアプリケーションの ID です。 ID を判別するには、list_apps コマンドを使用します。
    注: Bluemix の ASoC サービスに接続している場合には、アプリケーションの関連付けは適用されません。
ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。

例:

appscan queue_analysis -f my_irx.irx -n my_scan

このコマンドは、現行ディレクトリーにある my_irx.irx を、スキャンのためにクラウドにサブミットします。クラウド・ユーザー・インターフェースまたは list コマンドを使用してすべての現行スキャンを表示すると、リストに my_scan が含まれます。

appscan prepare

構文:

appscan prepare -c <configuration_file> -d <save_path> -l <log_path> -n <file_name> -t,--thirdParty -v,--verbose -X,--debug

説明:

IRX ファイルを生成します。

注: ソース・コードをスキャンしたり、IRX ファイルを生成する場合、最新の クライアント・ユーティリティー への更新に関するメッセージが表示される場合があります。クライアント・ユーティリティー とクラウド・サービスのバージョンの互換性を参照してください。

任意指定のオプション:

  • -c: このオプションは、スキャンを構成するために使用されます。
  • -d: -d <save_path> と指定します。 ここで、<save_path> は IRX ファイルを保存したいディレクトリーです。
  • -l: -l <log_path> と指定します。 ここで、<log_path> はログ・ファイルを保存したいディレクトリーです。
  • -n: -n <file_name> と指定します。 ここで、<file_name> は IRX ファイル名です。.irx ファイル拡張子付きまたはファイル拡張子なしでファイル名を指定できます。ファイル拡張子なしで指定すると、ファイル生成時に自動的に拡張子が付けられます。
  • -t,--thirdParty: デフォルトでは、IRX ファイルの生成中にサード・パーティーの Java および .NET コードはスキャンされません。サード・パーティーの Java と .NET の除外の管理 の手順に従って、サード・パーティーのコードの除外設定を変更できます。 サード・パーティー・コードを組み込むには、prepare コマンドの実行時に -t オプションまたは --thirdParty オプションを指定してください。

    サード・パーティー・コードがスキャンに標準的に含まれるようにする開発者の場合、この設定を使用してサード・パーティー・コードをスキャンに含める必要があります。

  • -v,--verbose: IRX ファイル生成中により多くの出力情報を表示するには、-v または --verbose を指定します。
  • -X,--debug: コマンド全体をデバッグ・モードで実行するには、-X または --debug を指定します。デバッグ・モードで実行すると、より多くのログ・ファイルがトラブルシューティング用に生成されます。
ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。

例:

構成ファイル c:¥my_config_files¥my_config.xml を使用する IRX ファイルを生成し、IRX ファイルを c:¥my_irx_files¥my_scan.irx に保存するには、以下のコマンドを実行します。

appscan prepare -c c:¥my_config_files¥my_config.xml -d c:¥my_irx_files -n my_scan.irx

appscan package

構文:

appscan package -d <save_path> -f <assessment_file> -n <file_name>

説明:

IBM Security AppScan® Source バージョン 9.0 以降の製品で作成されたアセスメント (.ozasmt) ファイルを含む IRX ファイルを生成します。

注: ソース・コードをスキャンしたり、IRX ファイルを生成する場合、最新の クライアント・ユーティリティー への更新に関するメッセージが表示される場合があります。クライアント・ユーティリティー とクラウド・サービスのバージョンの互換性を参照してください。

任意指定のオプション:

  • -d: -d <save_path> と指定します。 ここで、<save_path> は IRX ファイルを保存したいディレクトリーです。
  • -f: -f <assessment_file> を指定します。 ここで、<assessment_file> は、スキャンするためにパッケージする .ozasmt ファイルです。<assessment_file> ファイルが現行ディレクトリーにない場合、 このオプションを使用してアセスメント・ファイルのパスおよびファイル名を指定します。
    注: このオプションが必要なのは、以下の記述のいずれかまたは両方が当てはまる場合のみです。
    • コマンドを発行するディレクトリーに複数のアセスメント・ファイルが含まれている。ディレクトリーに含まれているアセスメント・ファイルが 1 つのみであれば、-f オプションが使用されていない場合はそのファイルがパッケージされます。
    • コマンドを発行するディレクトリーにアセスメント・ファイルが 1 つも含まれていない。この場合、-f オプションを使用して、パッケージするアセスメント・ファイルのパスおよびファイル名を指定する必要があります。
  • -n: -n <file_name> と指定します。 ここで、<file_name> は IRX ファイル名です。.irx ファイル拡張子付きまたはファイル拡張子なしでファイル名を指定できます。ファイル拡張子なしで指定すると、ファイル生成時に自動的に拡張子が付けられます。
ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。

例:

c:¥my_irx_files ディレクトリー内で、次のコマンドを実行することで、アセスメント・ファイル c:¥my_assessment_files¥my_assessment.ozasmt を含む IRX ファイルを生成することができます。

appscan package -f c:¥my_assessment_files¥my_assessment.ozasmt

結果の IRX ファイルは c:¥my_irx_files に保存されます。

appscan get_pubkey

構文:

appscan get_pubkey -d <save_path>

説明:

インターネットに接続されていないコンピューターで使用する公開暗号鍵をダウンロードします。

インターネットに接続されているコンピューターから IRX ファイルを生成する場合は、prepare コマンドを発行すると暗号鍵が自動的にダウンロードされるため、このコマンドは必要ありません。コンピューターに暗号鍵が既に存在する場合は、prepare コマンドを発行すると、必要に応じて暗号鍵が更新されます。

ただし、インターネットに接続されていないコンピューターから IRX ファイルを生成する場合は、このコマンドを使用して暗号鍵をダウンロードできます。その後、IRX ファイルを生成するときに、インターネットに接続されていないコンピューターに暗号鍵をコピーして使用します。そのコンピューターで暗号鍵を使用するには、rsa.pub のファイル名を保持し、解凍された SAClientUtil_<version>_<os>.zip (<version> はクライアント・ユーティリティーの現行バージョン) ファイルの config ディレクトリーに置く必要があります。

注: 使用する暗号鍵は最新である必要があります。 古い暗号鍵を使用して IRX ファイルを生成すると、IRX ファイルはアップロード時に拒否されます。ご使用のコンピューターがインターネットに接続されている場合は、prepare コマンドを再発行して、暗号鍵が自動的に更新されるようにします。ご使用のコンピューターがインターネットに接続されていない場合は、get_pubkey コマンドを使用する必要があります。

任意指定のオプション:

  • -d: -d <save_path> (<save_path> は暗号鍵を保存するディレクトリー) を指定します。このオプションを指定しない場合、鍵は、解凍された SAClientUtil_<version>_<os>.zip ファイルの configディレクトリーに保存されます。

appscan api_login

構文:

appscan api_login -P <key_secret> -u <key_id> -persist

説明:

生成された API キーを使用して IBM Cloud Marketplace の ASoC サービスにログインします。

重要: api_login コマンドは、IBM Cloud Marketplace においてのみ、CLI から ASoC サービスにログインするのに使用します。 Bluemix の ASoC サービスにログインするには、login コマンドを使用します。

このサービスにログインすると、cli.token という名前の暗号化された鍵ファイルが home ディレクトリーの .appscan フォルダーに保存されます。 このトークン・ファイルは、それ以降、ASoC サービスと対話するときに他のコマンドおよびアクションによって参照されます。このトークン・ファイルは、作成後 2 時間で有効期限が切れます。有効期限が切れたら、以降のコマンドやアクションを続行するためにサービスに再度ログインする必要があります。- あるいは、ログイン時に -persist オプションを組み込むことができます。 このオプションの使用時、クライアント・ユーティリティー はトークンの有効期限が切れるときに、サービスに対する再認証を自動的に試みます。

必須指定のオプション:

  • -P: -P <key_secret> を指定します。ここで、<key_secret> は、API キーの生成時に作成された「Key Secret」です。
  • -u: -u <key_id> を指定します。ここで、<key_id> は、API キーの生成時に作成された「Key Id」です。

任意指定のオプション:

  • -persist: ログイン・トークン・ファイルの有効期限が切れるときに、サービスに対する再認証を自動的に試みます。
ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。

appscan get_result

構文:

説明:

スキャン結果を取得します。

以下のコマンドを使用する前に、分析サービスにログインするようにしてください。

必須指定のオプション:

  • -i: -i <job_id> と指定します。 ここで、<job_id> は分析ジョブの ID です。

任意指定のオプション:

  • -d: -d <file_path> と指定します。ここで、<file_path> は、宛先ファイルの完全修飾パスまたは宛先ファイルのファイル名、あるいはその両方です。ファイル名が指定されていない場合、ファイル名はスキャン・ジョブ名に基づいた名前になります。パスが指定されていない場合、 ファイルは現行ディレクトリーに保存されます。このオプションが組み込まれていない場合、 ファイルは、スキャン・ジョブ名に基づいたファイル名で現行ディレクトリーに保存されます。

appscan.sh queue_analysis

構文:

appscan.sh queue_analysis -a <app_id> -f <irx_file> -n <scan_name>

説明:

分析のために IRX ファイルをクラウドにサブミットします。スキャンが完了すると、(分析サービスにログインする際に使用したアカウントと関連付けられた E メール・アドレスで) E メール通知を受信します。 この E メールにはクラウドへのリンクが含まれており、ログインしてスキャンをダウンロードできるようになっています。

注: ソース・コードをスキャンしたり、IRX ファイルを生成する場合、最新の クライアント・ユーティリティー への更新に関するメッセージが表示される場合があります。クライアント・ユーティリティー とクラウド・サービスのバージョンの互換性を参照してください。

必須指定のオプション:

  • -f: -f <irx_file> と指定します。 ここで、<irx_file> は、スキャンするためにサブミットする IRX ファイルです。IRX ファイルが現行ディレクトリーにない場合、 このオプションを使用して IRX ファイルのパスおよびファイル名を指定します。
    注: このオプションが必要なのは、以下の記述のいずれかまたは両方が当てはまる場合のみです。
    • コマンドを発行するディレクトリーに複数の IRX ファイルが含まれている。 ディレクトリーに含まれている IRX ファイルが 1 つのみであれば、 -f オプションが使用されていない場合はそのファイルがサブミットされます。
    • コマンドを発行するディレクトリーに IRX ファイルが 1 つも含まれていない。 この場合、 -f オプションを使用して、サブミットする IRX ファイルのパスおよびファイル名を指定する必要があります。
  • -n: -n <scan_name> と指定します。ここで、<scan_name> は、クラウドで実行されるスキャンの名前です。
  • -a (IBM Cloud Marketplace のみ): IBM Cloud Marketplace の Application Security on Cloud サービスに接続している場合、クラウドにサブミットする IRX ファイルは、既存の Application Security on Cloud アプリケーションと関連付ける必要があります。 このオプションを使用する際は、-a <app_id> と指定します。ここで <app_id> は、関連付けるアプリケーションの ID です。 ID を判別するには、list_apps コマンドを使用します。
    注: Bluemix の ASoC サービスに接続している場合には、アプリケーションの関連付けは適用されません。
ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。

例:

Bluemix のサービスに接続されている場合、次のコマンドを使用して、

appscan.sh queue_analysis -f my_irx.irx -n my_scan

このコマンドは、現行ディレクトリーにある my_irx.irx を、スキャンのためにクラウドにサブミットします。クラウド・ユーザー・インターフェースまたは list コマンドを使用してすべての現行スキャンを表示すると、リストに my_scan が含まれます。

IBM Cloud Marketplace のサービスに接続されている場合、上記のコマンドを発行しますが、-a 12345 を組み込んでください。ここで、12345 は、スキャンを関連付けるアプリケーションの ID です (このアプリケーション ID は list_apps コマンドを使用して判別できます)。

appscan.sh prepare

構文:

appscan.sh prepare -c <configuration_file> -d <save_path> -l <log_path> -n <file_name> -t,--thirdParty -v,--verbose -X,--debug

説明:

IRX ファイルを生成します。

注: ソース・コードをスキャンしたり、IRX ファイルを生成する場合、最新の クライアント・ユーティリティー への更新に関するメッセージが表示される場合があります。クライアント・ユーティリティー とクラウド・サービスのバージョンの互換性を参照してください。

任意指定のオプション:

  • -c: このオプションは、スキャンを構成するために使用されます。
  • -d: -d <save_path> と指定します。 ここで、<save_path> は IRX ファイルを保存したいディレクトリーです。
  • -l: -l <log_path> と指定します。 ここで、<log_path> はログ・ファイルを保存したいディレクトリーです。
  • -n: -n <file_name> と指定します。 ここで、<file_name> は IRX ファイル名です。.irx ファイル拡張子付きまたはファイル拡張子なしでファイル名を指定できます。ファイル拡張子なしで指定すると、ファイル生成時に自動的に拡張子が付けられます。
  • -t,--thirdParty: デフォルトでは、IRX ファイルの生成中にサード・パーティーの Java および .NET コードはスキャンされません。サード・パーティーの Java と .NET の除外の管理 の手順に従って、サード・パーティーのコードの除外設定を変更できます。 サード・パーティー・コードを組み込むには、prepare コマンドの実行時に -t オプションまたは --thirdParty オプションを指定してください。

    サード・パーティー・コードがスキャンに標準的に含まれるようにする開発者の場合、この設定を使用してサード・パーティー・コードをスキャンに含める必要があります。

  • -v,--verbose: IRX ファイル生成中により多くの出力情報を表示するには、-v または --verbose を指定します。
  • -X,--debug: コマンド全体をデバッグ・モードで実行するには、-X または --debug を指定します。デバッグ・モードで実行すると、より多くのログ・ファイルがトラブルシューティング用に生成されます。
ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。

例:

構成ファイル /root/Desktop/my_config_files/my_config.xml を使用する IRX ファイルを生成し、その IRX ファイルを /root/Desktop/my_irx_files -n my_scan.irx に保存するには、以下のコマンドを発行します。

appscan.sh prepare -c /root/Desktop/my_config_files/my_config.xml -d /root/Desktop/my_irx_files -n my_scan.irx

appscan.sh package

構文:

appscan.sh package -d <save_path> -f <assessment_file> -n <file_name>

説明:

IBM Security AppScan Source バージョン 9.0 以降の製品で作成されたアセスメント (.ozasmt) ファイルを含む IRX ファイルを生成します。

注: ソース・コードをスキャンしたり、IRX ファイルを生成する場合、最新の クライアント・ユーティリティー への更新に関するメッセージが表示される場合があります。クライアント・ユーティリティー とクラウド・サービスのバージョンの互換性を参照してください。

任意指定のオプション:

  • -d: -d <save_path> と指定します。 ここで、<save_path> は IRX ファイルを保存したいディレクトリーです。
  • -f: -f <assessment_file> を指定します。 ここで、<assessment_file> は、スキャンするためにパッケージする .ozasmt ファイルです。<assessment_file> ファイルが現行ディレクトリーにない場合、 このオプションを使用してアセスメント・ファイルのパスおよびファイル名を指定します。
    注: このオプションが必要なのは、以下の記述のいずれかまたは両方が当てはまる場合のみです。
    • コマンドを発行するディレクトリーに複数のアセスメント・ファイルが含まれている。ディレクトリーに含まれているアセスメント・ファイルが 1 つのみであれば、-f オプションが使用されていない場合はそのファイルがパッケージされます。
    • コマンドを発行するディレクトリーにアセスメント・ファイルが 1 つも含まれていない。この場合、-f オプションを使用して、パッケージするアセスメント・ファイルのパスおよびファイル名を指定する必要があります。
  • -n: -n <file_name> と指定します。 ここで、<file_name> は IRX ファイル名です。.irx ファイル拡張子付きまたはファイル拡張子なしでファイル名を指定できます。ファイル拡張子なしで指定すると、ファイル生成時に自動的に拡張子が付けられます。
ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。

例:

/root/Desktop/my_irx_files ディレクトリー内で、次のコマンドを実行することで、アセスメント・ファイル /root/Desktop/my_assessment_files/my_assessment.ozasmt を含む IRX ファイルを生成することができます。

appscan.sh package -f /root/Desktop/my_assessment_files/my_assessment.ozasmt

結果の IRX ファイルは /root/Desktop/my_irx_files に保存されます。

appscan.sh get_pubkey

構文:

appscan.sh get_pubkey -d <save_path>

説明:

インターネットに接続されていないコンピューターで使用する公開暗号鍵をダウンロードします。

インターネットに接続されているコンピューターから IRX ファイルを生成する場合は、prepare コマンドを発行すると暗号鍵が自動的にダウンロードされるため、このコマンドは必要ありません。コンピューターに暗号鍵が既に存在する場合は、prepare コマンドを発行すると、必要に応じて暗号鍵が更新されます。

ただし、インターネットに接続されていないコンピューターから IRX ファイルを生成する場合は、このコマンドを使用して暗号鍵をダウンロードできます。その後、IRX ファイルを生成するときに、インターネットに接続されていないコンピューターに暗号鍵をコピーして使用します。そのコンピューターで暗号鍵を使用するには、rsa.pub のファイル名を保持し、解凍された SAClientUtil_<version>_<os>.zip (<version> はクライアント・ユーティリティーの現行バージョン) ファイルの config ディレクトリーに置く必要があります。

注: 使用する暗号鍵は最新である必要があります。 古い暗号鍵を使用して IRX ファイルを生成すると、IRX ファイルはアップロード時に拒否されます。ご使用のコンピューターがインターネットに接続されている場合は、prepare コマンドを再発行して、暗号鍵が自動的に更新されるようにします。ご使用のコンピューターがインターネットに接続されていない場合は、get_pubkey コマンドを使用する必要があります。

任意指定のオプション:

  • -d: -d <save_path> (<save_path> は暗号鍵を保存するディレクトリー) を指定します。このオプションを指定しない場合、鍵は、解凍された SAClientUtil_<version>_<os>.zip ファイルの configディレクトリーに保存されます。

appscan.sh api_login

構文:

appscan.sh api_login -P <key_secret> -u <key_id> -persist

説明:

生成された API キーを使用して IBM Cloud Marketplace の ASoC サービスにログインします。

重要: api_login コマンドは、IBM Cloud Marketplace においてのみ、CLI から ASoC サービスにログインするのに使用します。 Bluemix の ASoC サービスにログインするには、login コマンドを使用します。

このサービスにログインすると、cli.token という名前の暗号化された鍵ファイルが home ディレクトリーの .appscan フォルダーに保存されます。 このトークン・ファイルは、それ以降、ASoC サービスと対話するときに他のコマンドおよびアクションによって参照されます。このトークン・ファイルは、作成後 2 時間で有効期限が切れます。有効期限が切れたら、以降のコマンドやアクションを続行するためにサービスに再度ログインする必要があります。- あるいは、ログイン時に -persist オプションを組み込むことができます。 このオプションの使用時、クライアント・ユーティリティー はトークンの有効期限が切れるときに、サービスに対する再認証を自動的に試みます。

必須指定のオプション:

  • -P: -P <key_secret> を指定します。ここで、<key_secret> は、API キーの生成時に作成された「Key Secret」です。
  • -u: -u <key_id> を指定します。ここで、<key_id> は、API キーの生成時に作成された「Key Id」です。

任意指定のオプション:

  • -persist: ログイン・トークン・ファイルの有効期限が切れるときに、サービスに対する再認証を自動的に試みます。
ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。

appscan.sh login

構文:

appscan.sh login -P <password> -u <user_name> -persist

説明:

Bluemix の ASoC サービスにログインします。

重要: login コマンドは、Bluemix においてのみ、CLI から ASoC サービスにログインするのに使用します。IBM Cloud Marketplace の ASoC サービスにログインするには、scx_login コマンドまたは api_login コマンドを使用します。

必須指定のオプション:

  • -P: -P <password> と指定します。ここで、<password> は、サービス資格情報で指定されているパスワードです。
  • -u: -u <user_name> と指定します。ここで、<user_name> は、サービス資格情報で指定されているバインディング ID です。

Bluemix サービス資格情報を判別するには、サービス・ダッシュボードの左側のナビゲーション・ペインで「サービス資格情報」を選択します。外部アプリが Bluemix サービスを使用できるようにするを参照してください。

結果コマンド (Linux および macOS)

結果コマンドを使用して、分析サービスからスキャン結果を取得します。

appscan.sh get_result

構文:

appscan.sh get_result -d <file_path> -i <job_id> -t <type>

説明:

スキャン結果を取得します。

以下のコマンドを使用する前に、分析サービスにログインするようにしてください。

必須指定のオプション:

  • -i: -i <job_id> と指定します。 ここで、<job_id> は分析ジョブの ID です。

任意指定のオプション:

  • -d: -d <file_path> と指定します。ここで、<file_path> は、宛先ファイルの完全修飾パスまたは宛先ファイルのファイル名、あるいはその両方です。ファイル名が指定されていない場合、ファイル名はスキャン・ジョブ名に基づいた名前になります。パスが指定されていない場合、 ファイルは現行ディレクトリーに保存されます。このオプションが組み込まれていない場合、 ファイルは、スキャン・ジョブ名に基づいたファイル名で現行ディレクトリーに保存されます。
  • -t: -t <type> と指定します。ここで、<type> は、 html または zip のいずれかです。結果は HTML ファイルとして、または、HTML 結果を含んでいる .zip ファイルとして保存されます。 このオプションが組み込まれていない場合、 結果は HTML ファイルとして保存されます。

    スキャン結果が package コマンドにより生成された IRX ファイルに対するものである場合、-t zip を指定すると、AppScan Source バージョン 9.0 以降の製品にロード可能な新規の .ozasmt ファイルを含む結果が保存されます。

ヒント: すべてのコマンドで、オプションは任意の順序で使用できます。
