Cómo empezar con Static Analyzer

Utilice el análisis estático para escanear código fuente para vulnerabilidades de seguridad. Para ello se debe descargar un pequeño programa de utilidad. Si extrae el programa de utilidad en el disco local, puede utilizar su interfaz de línea de mandatos (CLI) para realizar análisis de seguridad en todos los idiomas soportados. O bien, puede ejecutar un instalador incluido que añade plug-ins de análisis estático a Eclipse, Maven, IntelliJ IDEA o Visual Studio. En Eclipse e IntelliJ IDEA, puede escanear proyectos de Java, y en Visual Studio, puede escanear .NET (C#, ASP.NET, VB.NET).

Procedimiento

  • Para aprender a utilizar el servicio de Application Security on Cloud en IBM Cloud Marketplace, consulte http://www.ibm.com/support/knowledgecenter/SSYJJF_1.0.0/ApplicationSecurityonCloud/Welcome.html.
  • Para aprender a utilizar Application Security on Cloud for Bluemix, consulte https://console.ng.bluemix.net/docs/services/ApplicationSecurityonCloud/index.html.
  • El resto de este archivo readme cubre la instalación y el uso del Programa de utilidad de cliente de Static Analyzer.

Configuración del Programa de utilidad de cliente de Static Analyzer

Para el análisis estático, descargue un pequeño programa de utilidad cliente. Si extrae el programa de utilidad en el disco local, puede utilizar su interfaz de línea de mandatos (CLI) para realizar análisis de seguridad. O bien, puede ejecutar un instalador incluido que añade el plug-in de análisis estático al entorno de creación de Maven - o a los entornos de desarrollo integrado (IDE) de Eclipse, IntelliJ IDEA o Visual Studio.

Acerca de esta tarea

Cuando utilice el análisis estático, escanee código fuente para generar un archivo que se sube a la nube. El archivo, IRX (.irx) cifrado, se escanea utilizando un análisis de rastreo para detectar vulnerabilidades de seguridad. Cuando el archivo IRX se genera para idiomas compilados, los archivos de código de bytes de la aplicación se convierten en un Intermediate Representation del código. Cuando un archivo IRX se genera para idiomas de script, los archivos de origen se incluyen en el archivo cifrado IRX.

Cuando se utilizan los plug-ins de Eclipse, IntelliJ IDEA o Visual Studio, la generación y carga del archivo IRX tiene lugar automáticamente desde el IDE cuando se utilizan acciones de análisis estático.

Consejo: Cuando se ejecuta el instalador, si no va a utilizar análisis estático en un IDE determinado, pero dicho IDE está instalado y espera utilizarlo en el futuro, es posible que desee instalar ahora el plug-in en dicho IDE de forma que no necesite volver a ejecutar el instalador más adelante.
Vía de acceso rápida: Cuando se especifica sí o no en el instalador, puede especificar y o n como abreviaturas. Si especifica abreviaturas en un instalador que se ejecuta en máquinas en otro idioma, las abreviaturas que especifique deben ser y o n.
Importante: El programa de utilidad cliente no debe colocarse en el directorio que contiene caracteres que no sean ASCII.

Procedimiento

Para configurar el programa de utilidad cliente:

  1. Seleccione la plataforma en la que tiene pensado ejecutar el programa de utilidad cliente y a continuación pulse Descargar. Esto descargará un archivo SAClientUtil_<version>_<os>.zip (donde <versión> es la versión actual del programa de utilidad cliente y <os> es el sistema operativo para el que es programa de utilidad cliente).
    Nota: Puede volver a abrir el formulario de bienvenida seleccionando ¿Cómo se crea un archivo IRX? en el formulario para seleccionar un archivo IRX.
  2. Extraiga el archivo en una unidad local.
  3. Si tiene pensado utilizar la CLI para generar archivos IRX, cargar archivos IRX o gestionar escaneos: añada la ubicación del directorio \bin del archivo SAClientUtil_<version>_<os>.zip extraído en la variable de entorno PATH. Si no lo hace, todos los mandatos se tendrán que cualificar utilizando el directorio \bin del archivo extraído SAClientUtil_<version>_<os>.zip cada vez que se emita el mandato.
    Importante: Si tiene pensado escanear .NET con la CLI o Microsoft Visual Studio, estos tres archivos .bat (en el archivo SAClientUtil_<version>_<os>.zip extraído) se deben ejecutar como administrador:
    • bin\register.bat
    • walalib\dotnet.libs40\RegisterInGAC.bat
    • walalib\dotnet.libs35\RegisterInGAC.bat

    Estos archivos se pueden ejecutar manualmente, o bien puede ejecutar el archivo install-plugins.bat como administrador (si utiliza solo la CLI, ejecute install-plugins.bat pero especifique no cuando se le solicite instalar el plug-in de análisis estático).

    Si estos archivos no se ejecutan como administrador, aparecerá un mensaje de aviso en la consola cada vez que se utilice la CLI. Este mensaje se puede omitir si está escaneando destinos Java.

    Consejo: Tras cambiar la PATH, emita appscan version (Windows) o appscan.sh version (Linux y macOS) en un indicador de mandatos. Si se devuelve la versión, la página inicial u otra información de Programa de utilidad de cliente de Static Analyzer, PATH está configurado correctamente.
  4. Si tiene pensado añadir el plug-in de análisis estático a Eclipse, Microsoft Visual Studio (solo en Windows), IntelliJ IDEA o Maven: localice y ejecute install-plugins.bat (Windows) o install-plugins.sh (Linux y macOS) en el programa de utilidad cliente extraído. En Windows, ejecute el archivo como administrador. Si tiene pensado añadir el plug-in a Eclipse en Linux o macOS, asegúrese de que ha iniciado la sesión en una cuenta que tiene permisos para modificar la instancia de Eclipse en la que está instalando.
    1. De forma predeterminada, el programa de utilidad cliente se conectará al servicio de ASoC en IBM Cloud Marketplace. Si se va a conectar al servicio en IBM Bluemix, especifique sí cuando se le solicite.
    2. Si tiene pensado escanear código fuente en Eclipse, siga las indicaciones del instalador. Si desea instalar el plug-in de Eclipse, se le solicitará que especifique la ubicación de la instalación de Eclipse. Especifique la vía de acceso completa del directorio que contenga el archivo eclipse.exe (Windows) o eclipse (Linux y macOS).
    3. Si tiene pensado utilizar Maven para la generación de archivos IRX, siga las indicaciones del instalador.
    4. Si tiene pensado escanear código fuente en IntelliJ IDEA, siga las indicaciones del instalador. Si opta por instalar el plug-in de IntelliJ IDEA, se le solicitará que especifique la ubicación del directorio de configuración de IntelliJ IDEA. Especifique la vía de acceso completa de este directorio.
    5. Windows solamente: si tiene pensado escanear código fuente en Visual Studio, siga las indicaciones del instalador. Si opta por instalar el plug-in de Visual Studio, se le solicitará que elija la versión de Visual Studio para la que va a instalar el plug-in. Especifique la versión correspondiente a la versión de Visual Studio en la que desee instalar el plug-in.
    6. Cierre el indicador de mandatos pulsando cualquier tecla.
    7. Para cualquier IDE en el que haya instalado el plug-in, reinicie el IDE.
    8. Solo Linux: en un indicador de mandatos, vaya al directorio que contiene el programa de utilidad del cliente y añada estas líneas al archivo de recursos de shell (por ejemplo, .bash_profile o .shrc): 
      export APPSCAN_INSTALL_DIR=<programa_utilidad_cliente>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export LD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${LD_LIBRARY_PATH}

      donde <programa_utilidad_cliente> es la vía de acceso al programa de utilidad cliente. Después de añadir estas líneas, vuelva a cargar el shell antes de utilizar el plug-in de análisis estático.

    9. Solo macOS: En un indicador de mandatos, vaya al directorio que contiene el programa de utilidad del cliente y añada estas líneas al archivo de recursos de shell (por ejemplo, .bash_profile o .shrc): 
      export APPSCAN_INSTALL_DIR=<programa_utilidad_cliente>
export PATH=${APPSCAN_INSTALL_DIR}/bin:${PATH}
export DYLD_LIBRARY_PATH=${APPSCAN_INSTALL_DIR}/bin:${DYLD_LIBRARY_PATH}

      donde <programa_utilidad_cliente> es la vía de acceso al programa de utilidad cliente. Después de añadir estas líneas, vuelva a cargar el shell antes de utilizar el plug-in de análisis estático.

      Importante: La definición de la variable de entorno en el archivo de recursos de shell afecta a las aplicaciones que se inician en el terminal. Por lo tanto, debe utilizar la ventana de terminal para iniciar los entornos de desarrollo integrado (IDE) desde los que desee ejecutar el análisis estático.

Qué hacer a continuación

Si ejecuta el programa de utilidad cliente en un sistema que está detrás de un proxy: Utilice uno de estos métodos para especificar el proxy de forma que programa de utilidad cliente se pueda conectar a la nube:

  • CLI y entornos de desarrollo integrado (IDE) soportados del programa de utilidad cliente: Defina esta variable de entorno de sistema o global de modo que el proxy se reconozca automáticamente:
    • Windows: APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>
    • Linux y macOS: APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"

    Donde <proxy> es el nombre de host del servidor proxy y <port> es el número de puerto que utiliza el servidor proxy.

    O bien, cada vez que utilice la CLI o el IDE (que inicia desde una línea de mandatos o un terminal) puede establecer que el programa de utilidad cliente utilice el proxy con el mandato siguiente:

    • Windows: set "APPSCAN_OPTS=-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
    • Linux y macOS: export APPSCAN_OPTS="-Dhttps.proxyHost=<proxy> -Dhttps.proxyPort=<port>"
  • Maven: Si ha añadido el plug-in de análisis estático en Maven, puede añadir la propiedad a la variable de entorno MAVEN_OPTS de modo global o puede añadir la propiedad a cada uno de los mandatos.

Eliminación del Programa de utilidad de cliente de Static Analyzer

Esta tarea describe cómo eliminar los plug-ins de Programa de utilidad de cliente de Static Analyzer desde el entorno de desarrollo integrado (IDE). Además, se proporcionan instrucciones para eliminar el programa de utilidad cliente. Estas instrucciones solo deben seguirse si pretende eliminar los componentes del Programa de utilidad de cliente de Static Analyzer desde el sistema. Si está actualizando estos componentes a una nueva versión del programa de utilidad cliente, la instalación del plug-in eliminará componentes antiguos automáticamente antes de instalar la versión nueva.

Antes de empezar

Importante: Si ha instalado el plug-in de programa de utilidad cliente en un IDE, deberá ejecutar uno o varios archivos o mandatos en la ubicación extraída de SAClientUtil_<version>_<os>.zip para eliminar completamente el plug-in de IDE. Estos archivos son:
  • Programa de utilidad de cliente de Static Analyzer Versión 6.0.865 y anterior:
    • walalib\dotnet.libs35\UnRegisterInGAC.bat
    • walalib\dotnet.libs40\UnRegisterInGAC.bat
  • Todas las versiones de Programa de utilidad de cliente de Static Analyzer, incluida la Versión 6.0.865 y anterior: Abra un indicador de mandatos como administrador y cambie el directorio a la ubicación a la que ha extraído SAClientUtil_<version>_<os>.zip. Cambie el directorio a la carpeta bin\ y, a continuación, emita este mandato:
    regsvr32 /u msdia80.dll

Para poder ejecutar estos archivos y mandatos, no suprima el contenido extraído de SAClientUtil_<version>_<os>.zip hasta que se eliminen todos los plug-ins de IDE.

Puede ejecutar estos archivos y mandatos antes o después de seguir las instrucciones siguientes para eliminar los plug-ins del IDE.

Procedimiento

  • Eclipse:
    1. En Eclipse, seleccione Ayuda > Instalar software nuevo en la barra de menús del entorno de trabajo principal.
    2. En la selección Detalles del recuadro de diálogo Instalar, pulse el enlace ¿Qué hay instalado ya?.
    3. En la lista de software instalado, ubique y seleccione Static Analyzer y, a continuación, seleccione Desinstalar.
    4. En el recuadro de diálogo Desinstalar, pulse Finalizar.
    5. Cuando se le solicite, reinicie Eclipse.
  • IntelliJ IDEA:
    1. En IntelliJ IDEA, seleccione Archivo > Configuración desde la barra del menú principal.
    2. En el recuadro de diálogo Configuración, seleccione Plugins.
    3. En la lista de plugins, ubique y seleccione Plugin de Static Analyzer.
    4. Pulse Desinstalar.
    5. Pulse Reiniciar IntelliJ IDEA.
  • Visual Studio:
    1. En Visual Studio, seleccione Herramientas > Extensiones y actualizaciones desde el menú principal.
    2. En la lista de extensiones y actualizaciones instaladas, ubique y seleccione IBM Static Analyzer.
    3. Pulse Desinstalar y, a continuación, cuando se le solicite desinstalar IBM Static Analyzer, pulse Sí.
    4. Seleccione Reiniciar ahora para reiniciar Visual Studio, o seleccione Cerrar si va a reiniciar Visual Studio más tarde.
  • programa de utilidad cliente: Para eliminar el programa de utilidad cliente, solo necesita suprimir el contenido extraído de SAClientUtil_<version>_<os>.zip del sistema. Si ha añadido la ubicación del directorio \bin del programa de utilidad cliente a la variable de entorno PATH, elimínela.

Soporte del sistema operativo

El Programa de utilidad de cliente de Static Analyzer está soportado en Windows, macOS y Linux.

Generación de un archivo IRX con la interfaz de línea de mandatos (CLI)

Para poder iniciar un análisis de los archivos, debe generar un archivo IRX para enviarlo a la nube. Si desea utilizar la CLI, siga estas instrucciones para crear el archivo.

Procedimiento

Para generar el archivo IRX:

  1. Asegúrese de que ha abierto una nueva ventana de indicador de mandatos (no utilice la ventana de indicador de mandatos que se ha utilizado para instalar el programa de utilidad cliente).
  2. En el indicador de mandatos, emita un mandato para cambiar el directorio por la ubicación que contiene los archivos. Consulte Soporte de idiomas para obtener más información sobre los tipos de archivos soportados. Si está escaneando proyectos Java en Eclipse, cambie el directorio al espacio de trabajo de Eclipse.
    Nota:
    • Este paso no es necesario si utiliza un archivo de configuración para generar el archivo IRX. Consulte Configuración de la generación de archivos IRX con la CLI.
    • Si utiliza la CLI, no puede escanear proyectos individuales en un espacio de trabajo Eclipse, debe escanear todo el espacio. Si quiere escanear proyectos individuales, debe generar el archivo IRX en Eclipse.
    • Los archivos .dll y .exe deben ser ensamblados .NET válidos.
  3. Emita el mandato Appscan prepare en (Windows) o Appscan.sh prepare en (Linux y macOS). Este mandato se utiliza con las opciones que se muestran en los temas Mandatos de configuración (Windows) y Mandatos de configuración (Linux y macOS). Por ejemplo, puede utilizar las opciones -d y -n para especificar un directorio determinado donde guardar el archivo IRX y el archivo con un determinado nombre de archivo. Además, puede ejecutar el mandato con un archivo de configuración para especificar información adicional que le ayudará a generar un archivo IRX completo (consulte Configuración de la generación de archivos IRX con la CLI).
    Consejo: De forma predeterminada, el código de terceros Java y .NET no se escanea durante la generación de archivos de IRX. Puede modificar los valores de exclusión de código de terceros siguiendo las instrucciones de Gestión de exclusiones Java y .NET de terceros. Para incluir código de terceros, especifique la opción -t o --thirdParty cuando emita el mandato prepare.

    Si es un desarrollador de código de terceros que normalmente se incluiría en un escaneo, debe utilizar este parámetro para incluir el código de terceros.

    Nota: Cuando escanee código fuente o genere un archivo IRX, es posible que reciba un mensaje sobre actualizar a la última versión del programa de utilidad cliente. Consulte Compatibilidad entre el programa de utilidad cliente y la versión del servicio de nube.
  4. Después de emitir el mandato, programa de utilidad cliente recopilará información sobre los archivos soportados en el directorio desde el que se emitió el mandato y todos sus subdirectorios. Se creará un archivo IRX en el directorio. El nombre de este archivo se basa en el directorio desde el que se ha emitido el mandato y se incluye una indicación de fecha y hora, suponiendo que no ha utilizado una opción de configuración para especificar un nombre de archivo diferente.
  5. Envíe el archivo IRX a la nube para analizarlo. Para ello, emita el mandato appscan queue_analysis en (Windows) o el mandato appscan.sh queue_analysis en (Linux y macOS). Este mandato se utiliza con las opciones que se muestran en los temas Mandatos de análisis (Windows) o Mandatos de análisis (Linux y macOS).

Generación de un archivo IRX para un proyecto Maven

Para poder iniciar un análisis de los archivos, debe generar un archivo IRX para enviarlo a la nube. Si desea escanear un proyecto Maven, siga estas instrucciones para crear el archivo.

Procedimiento

Para generar el archivo IRX:

  1. Asegúrese de que ha abierto una nueva ventana de indicador de mandatos (no utilice la ventana de indicador de mandatos que se ha utilizado para instalar el programa de utilidad cliente).
  2. En el indicador de mandatos, emita un mandato para cambiar el directorio por la ubicación que contiene los archivos del proyecto Maven, la carpeta que contiene el archivo pom.xml.
  3. Emita uno de los mandatos siguientes:
    • Si el proyecto de Maven aún no se ha creado, emita el mandato mvn package com.ibm.appscan:appscan-maven-plugin:prepare. Una vez que se haya emitido el mandato, se creará un archivo IRX en el directorio target del proyecto Maven. El nombre de este archivo se basará en el proyecto Maven de nivel superior (el nombre del proyecto del archivo pom.xml), suponiendo que no haya utilizado una opción de configuración para especificar un nombre de archivo diferente.
    • Si el proyecto de Maven aún no se ha creado, emita mvn com.ibm.appscan:appscan-maven-plugin:prepare. Una vez que se haya emitido el mandato, se creará un archivo IRX en el directorio target del proyecto Maven. El nombre de este archivo se basará en el proyecto Maven de nivel superior (el nombre del proyecto del archivo pom.xml), suponiendo que no haya utilizado una opción de configuración para especificar un nombre de archivo diferente.
      Nota: Si el proyecto Maven tiene dependencias de tiempo de compilación de otro proyecto en la misma compilación de reactor, un error de Maven existente (https://issues.apache.org/jira/browse/MNG-3283) provocará que el mandato mvn com.ibm.appscan:appscan-maven-plugin:prepare falle. Para evitar este problema, realice uno de los siguientes pasos:
      • Ejecute el mandato mvn install en lugar de mvn package antes de ejecutar el mandato prepare de programa de utilidad cliente.
      • Emita el mandato mvn package y el mandato prepare de programa de utilidad cliente simultáneamente (emita mvn package com.ibm.appscan:appscan-maven-plugin:prepare).
    • Si desea que el archivo IRX se cree en otro directorio, o tiene un nombre de directorio distinto, emita los mandatos con la opción -Doutput:

      En Windows, emita este mandato:

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<ubicación_salida>\<nombre_archivo>.irx

      En Linux y macOS, emita este mandato:

      mvn package com.ibm.appscan:appscan-maven-plugin:prepare -Doutput=<ubicación_salida>/<nombre_archivo>.irx

      donde <ubicación_salida> es el directorio donde desea guardar el archivo y nombre_archivo es el nombre del archivo.

    • La instalación del programa de utilidad cliente incluye un compilador JSP de Tomcat. Este compilador JSP se utiliza de forma predeterminada durante la generación de archivos IRX. Si está generando un archivo IRX para un archivo WAR o EAR y desea que se utilice un compilador JSP distinto, emita el mandato con una opción -DjspCompiler. Por ejemplo:
      mvn com.ibm.appscan:appscan-maven-plugin:prepare -DjspCompiler=<compilador_jsp>

      donde <compilador_jsp> es la vía de acceso del compilador JSP que desea utilizar (especifique el directorio raíz del servidor).

    Consejo: De forma predeterminada, el código de terceros no se escanea durante la generación de archivos de IRX. Puede modificar los valores de exclusión de código de terceros siguiendo las instrucciones de Gestión de exclusiones Java y .NET de terceros. Para incluir código de terceros, emita el mandato con la opción -DthirdParty.

    Si es un desarrollador de código de terceros que normalmente se incluiría en un escaneo, debe utilizar este parámetro para incluir el código de terceros.

    Nota: Cuando escanee código fuente o genere un archivo IRX, es posible que reciba un mensaje sobre actualizar a la última versión del programa de utilidad cliente. Consulte Compatibilidad entre el programa de utilidad cliente y la versión del servicio de nube.
  4. Envíe el archivo IRX a la nube para analizarlo. Para ello, emita el mandato appscan queue_analysis en (Windows) o el mandato appscan.sh queue_analysis en (Linux y macOS). Este mandato se utiliza con las opciones que se muestran en los temas Mandatos de análisis (Windows) o Mandatos de análisis (Linux y macOS).

Uso del archivo de configuración

Se puede utilizar un archivo de configuración para indicar qué destinos se incluirán o excluirán al generar un archivo IRX. También puede especificar directorios que contengan artefactos de compilación, y especificar información de compilación que se utilizará durante la generación de archivos IRX (por ejemplo, puede utilizar este valor para especificar un determinado JDK o compilador JSP).

Cada vez que emita el mandato Appscan prepare (Windows) o Appscan.sh prepare (Linux y macOS) para generar un archivo IRX, ele Programa de utilidad de cliente de Static Analyzer comprobará automáticamente si hay un archivo appscan-config.xml en el directorio actual. Si se encuentra el archivo, se utilizará automáticamente.

Si tiene un archivo de configuración en otro directorio, o con un nombre de archivo distinto, puede utilizar el archivo incluyendo la opción -c al emitir el mandato Appscan prepare o Appscan.sh prepare. En este caso, debe emitir appscan prepare -c <archivo_configuración> en (Windows) o appscan.sh prepare -c <configuration_file> en (Linux y macOS), donde <archivo_configuración> es la vía de acceso completa y el nombre del archivo de configuración.

Plantilla y valores del archivo de configuración

Cuando prepare el archivo de configuración, utilice esta plantilla:

<Configuration>
  <Targets>
    <Target outputs-only="true/false" path="scan_target_path">
      <CustomBuildInfo build_info="info"/>
      <Include>string_pattern</Include>
      <Exclude>string_pattern</Exclude>
    </Target>
  </Targets>
</Configuration>

Puede utilizar el elemento Targets para indicar qué destinos deben escanearse al generar el archivo IRX. Los atributos de destino (Target) y los elementos hijo se utilizan como se indica a continuación:

  • El atributo Target del elemento outputs-only solo se utiliza cuando se desea escanear un directorio y forzar que programa de utilidad cliente busque solo archivos de salida de compilación (por ejemplo, los archivos .jar, .war y .class). De forma predeterminada, este atributo se establece en false. Esto significa que programa de utilidad cliente buscará el directorio para determinar si éste es un destino, por ejemplo, un servidor de aplicaciones o un espacio de trabajo de Eclipse, o si el directorio contiene elementos como scripts de compilación, archivos Maven .pom o archivos make. Si desea que el destino de escaneo se trate como un directorio simple y solo ha localizado los archivos de salida, especifique outputs-only="true" en el elemento Target.

    Por ejemplo, si especifica <Target outputs-only="false" path="C:\Tomcat">, programa de utilidad cliente supondrá que el destino es el servidor de aplicaciones Tomcat y buscará los archivos .war desplegados. Sin embargo, si especifica <Target outputs-only="true" path="C:\Tomcat">, programa de utilidad cliente tratará la ubicación como directorio y localizará todos los archivos de salida de la ubicación.

  • Utilice el atributo path del elemento Target para especificar la vía de acceso de un destino de escaneo o un directorio de destinos de escaneo (<vía_acceso_destino_escaneo>). Al especificar un directorio, durante el escaneo se incluirán todos sus subdirectorios.

    Por ejemplo, si especifica <Target outputs-only="false" path="C:\WebSphere\AppServer85\profiles\AppSrv01">, el programa de utilidad cliente localizará todos los archivos .ear desplegados en el perfil AppSrv01. Si especifica <Target outputs-only="false" path="C:\WebSphere\AppServer85">, el programa de utilidad cliente localizará todos los archivos .ear desplegados en todos los perfiles.

  • También puede especificar estos subelementos opcionales de Target:
    • Puede utilizar el elemento CustomBuildInfo para establecer atributos. Si utiliza esta parte de la plantilla,
      <CustomBuildInfo build_info="info"/>
      puede especificar la información de compilación dependiendo del idioma de destino. Para algunos idiomas, puede establecer más de un atributo, por ejemplo, puede establecer <CustomBuildInfo build_info_1="info_1" build_info_2="info_2" build_info_3="info_3"/>, dependiendo del idioma de destino.
      • Para Java, puede establecer estos atributos:
        <CustomBuildInfo additional_classpath="dependency_path"
  jdk_path="JDK_path" jsp_compiler="JSP_compiler_path"/>
        • Especifique más vías de acceso de clase utilizando el atributo additional_classpath. En Windows, separe las vías de acceso de clase con un punto y coma. En Linux y macOS, separe las vías de acceso de clase con dos puntos.
        • Utilice el atributo jdk_path para especificar la vía de acceso de la instalación de JDK.
        • Utilice el atributo jsp_compiler para especificar la vía de acceso al compilador JSP, por ejemplo jsp_compiler="C:\Tomcat".
      • Para .NET (sólo Windows), puede establecer estos atributos:
        <CustomBuildInfo reference="assembly_reference"
  configuration="build_configuration"/>
        • Añada referencias de ensamblado utilizando el atributo reference. Separe varias referencias con punto y coma.
        • Utilice el atributo configuration para incluir una configuración de compilación para el descubrimiento de la solución de Visual Studio.
      • Para C/C++ (sólo Windows), puede establecer estos atributos:
        <CustomBuildInfo configuration="build_configuration"
  include_paths="include_directories" macros="macros" compiler_opts=/>
        • Utilice el atributo configuration para incluir una configuración de compilación.
        • Especifique vías de acceso de inclusión utilizando el atributo include_paths. Separe varias vías de acceso de inclusión con punto y coma.
        • Incluya macros utilizando el atributo macros. Separe varias macros con punto y coma.
        • Especifique las opciones del compilador con el atributo compiler_opts. Separe varias opciones con punto y coma.
    Nota: Los subdestinos heredan los valores que definen estos atributos. Por ejemplo, si el destino es un archivo EAR que incluye subdestinos WAR y JAR, se da por supuesto que WAR y JAR tienen los mismos valores que los establecidos para el archivo EAR que utiliza estos atributos.
  • Utilice el elemento hijo Include para especificar patrones de archivo (<string_pattern>) e incluirlos al generar el archivo IRX. El comportamiento de Include depende del tipo de destino, tal como se describe en la sección Comportamiento de los valores Incluir y Excluir de destino. Para especificar varios patrones include, añada cada patrón en su propio código <Include></Include>. Por ejemplo,
    <Include>serie_patrón_1</Include>
<Include>serie_patrón_2</Include>
    Nota: Si especifica patrones include y exclude que están en conflicto, prevalecerán los patrones exclude.
  • Utilice el elemento hijo Exclude para especificar patrones de archivo al generar el archivo IRX. El comportamiento Exclude depende del tipo de destino, tal como se describe en la sección Comportamiento de los valores Incluir y Excluir de destino. Para especificar varios patrones exclude, añada cada patrón en su propio código <Exclude></Exclude>.
    Nota: Si especifica patrones include y exclude que están en conflicto, prevalecerán los patrones exclude.

Ejemplo de archivo de configuración

<Configuration>
  <Targets>
    <Target path="C:\test\MyApplication.ear">
      <CustomBuildInfo additional_classpath="C:\java\lib;C:\java2\lib"
        jdk_path="C:\Program Files\Java\jdk1.7" jsp_compiler="C:\Tomcat"/>
      <Include>ShoppingCart.war</Include>
      <Include>*Account.jar</Include>
      <Exclude>thirdPartyJars/</Exclude>
      <Exclude>*test*</Exclude>
    </Target>
  </Targets>
</Configuration>
En este archivo de configuración:
  • C:\test\MyApplication.ear es el destino.
  • En los archivos C:\java\lib y C:\java2\lib se buscarán las dependencias de archivos de clase necesarias para la compilación.
  • Se utilizará C:\Program Files\Java\jdk1.7 para localizar dependencias.
  • Se utilizará el compilador C:\Tomcat para la compilación de JSP en lugar de la aplicación Apache Tomcat Versión 7 que se incluye con el Programa de utilidad de cliente de Static Analyzer.
  • Cuando se genere el archivo IRX, se incluirá el archivo ShoppingCart.war, al igual que todos los finales de archivo en *Account.jar. Escaneo en entornos de desarrollo integrado

Estas instrucciones explican cómo escanear código fuente utilizando el plug-in de análisis estático una vez que se ha instalado en los entornos de desarrollo integrado (IDE) de Eclipse, IntelliJ IDEA o Visual Studio. En Eclipse e IntelliJ IDEA, puede escanear proyectos de Java, y en Visual Studio, puede escanear .NET (C#, ASP.NET, VB.NET).

Antes de empezar

De forma predeterminada, el código de terceros no se incluye al escanear Java y .NET.Puede modificar los valores de exclusión de código de terceros siguiendo las instrucciones de Gestión de exclusiones Java y .NET de terceros.
  • Para incluir código de terceros al escanear en Eclipse, IntelliJ IDEA o Visual Studio, utilice uno de estos métodos:
    • Defina esta variable de entorno de sistema o global antes de iniciar el ID: 
      APPSCAN_OPTS=-DthirdParty
    • Cada vez que utilice el IDE, puede emitir un mandato antes de iniciar el IDE:
      set APPSCAN_OPTS=-DthirdParty
  • Como alternativa, al escanear en Eclipse, puede hacer esto: modifique el archivo eclipse.ini antes de iniciar Eclipse de forma que la sección -vmargs incluya -DthirdParty.
  • Como alternativa, al escanear en IntelliJ IDEA, puede hacer esto: en el directorio bin de su instalación de IntelliJ IDEA, modifique el archivo idea.exe.vmoptions antes de iniciar IntelliJ IDEA. Añada -DthirdParty al archivo y a continuación guárdelo.

Si es un desarrollador de código de terceros que normalmente se incluiría en un escaneo, debe utilizar este parámetro para incluir el código de terceros.

Para escanear código fuente y abrir evaluaciones o informes:

    • En IntelliJ IDEA, seleccione los módulos de proyecto que desee escanear.
    • En Visual Studio, seleccione las soluciones, los proyectos o los sitios web que desee escanear.
  3. Pulse con el botón derecho del ratón en la selección y elija Análisis de seguridad > Ejecutar análisis estático.
    Nota: Cuando escanee código fuente o genere un archivo IRX, es posible que reciba un mensaje sobre actualizar a la última versión del programa de utilidad cliente. Consulte Compatibilidad entre el programa de utilidad cliente y la versión del servicio de nube.

Errores o avisos que se producen al generar un archivo IRX de análisis estático

Síntomas

Al generar un archivo IRX, podría recibir errores o advertencias o podría fallar la generación de archivos IRX. Si la generación del archivo IRX falla, se añade _failed al nombre del archivo resultante (por ejemplo, my_irx_file_failed.irx). Cuando esto ocurre, tiene varias opciones para resolver la generación de archivos de IRX.

Resolución del problema

  • Uso de un archivo de configuración para resolver la generación de archivos de IRX

    Si genera un archivo IRX sin utilizar un archivo de configuración y la generación del archivo no se completa (o, si se completa con errores), el Programa de utilidad de cliente de Static Analyzer generará un archivo de configuración temporal para que lo utilice en escaneos posteriores. Este archivo será similar a la plantilla de archivo de configuración, en que contendrá un elemento Targets con información conocida sobre los destinos. Sin embargo, el archivo también contendrá un elemento <TargetSettings> que se pueda modificar agregando información y, de esta manera, programa de utilidad cliente podrá generar un archivo IRX completo. Este archivo se denominará appscan-config.xml y se guardará en el directorio actual. Esto significa que el archivo se utilizará automáticamente para el siguiente escaneo.

    Por ejemplo, una generación de archivos IRX que se proporciona para un destino que contiene dependencias externas y no se encuentran, podría recibir un mensaje como este:

    C:\myTargetDirectory>appscan prepare
An IRX file was created, but it may be incomplete.
C:\myTargetDirectory\myTarget.jar requiere los siguientes paquetes:
      org.apache.jasper.*
Se ha creado un archivo appscan-config.xml en: C:\myTargetDirectory\appscan-config.xml
Actualice el archivo y vuelva a ejecutar el mandato.

    El archivo appscan-config.xml que se genera tendrá un aspecto similar a este:

    <Configuration>
  <Targets>
    <Target outputs-only="true" path="C:\myTargetDirectory"/>
  </Targets>
  <Settings/>
  <TargetSettings>
    <Target path="C:\myTargetDirectory\myTarget.jar"/>
      <!--Añada la vía de acceso de las dependencias siguientes para el atributo "additional_classpath" que se muestra a continuación, separado por punto y coma.
			  org.apache.jasper.*-->
      <CustomBuildInfo additional_classpath=""/>
    </Target>
  </TargetSettings>
</Configuration>

    Para resolver el problema, actualice <CustomBuildInfo additional_classpath=""/> para incluir las clases org.apache.jasper.* en la vía de acceso de la clase, guarde el archivo appscan-config.xml y, a continuación, vuelva a ejecutar el mandato Appscan prepare (Windows) o Appscan.sh prepare (Linux y macOS). Cuando especifique la vía de acceso al archivo o archivos JAR, la sintaxis de vía de acceso a la clase Java estándar está soportada (por ejemplo, los caracteres comodín están soportados).

    Si encuentra problemas parecidos cuando ya está utilizando un archivo de configuración, recibirá el mensaje que indique los problemas. Utilice este mensaje para actualizar el archivo de configuración existente y, a continuación, vuelva a ejecutar el mandato.

    El archivo de configuración se describe en Configuración de la generación de archivos IRX con la CLI.

  • Generación de archivos IRX en modalidad de depuración

    Si está generando un archivo IRX desde la CLI, puede emitir el mandato Appscan prepare (Windows) o Appscan.sh prepare (Linux y macOS) con una opción de depuración. Esto hace que se generen más archivos de registro.

  • Archivos de registro asociados con la generación de archivos IRX

    Para ver los errores o avisos, consulte los archivos de registro que también se han generado. Los archivos de registro se guardan en un archivo de archivado .zip. De forma predeterminada, el archivo se guarda en la misma ubicación en la que se guarda su IRX, o puede especificar otra ubicación utilizando la opción -l del mandato Appscan prepare (Windows) o Appscan.sh prepare (Linux y macOS). El nombre del archivo .zip se basa en el nombre del archivo IRX que se genera.

Anotaciones de código fuente Java soportadas

Cuando se utiliza el análisis estático para escanear Java, se da soporte a las anotaciones a nivel de método @ValidatorMethod, @CallbackMethod y @SuppressSecurityTrace.

  • Utilización de anotaciones
  • @ValidatorMethod
  • @SuppressSecurityTrace
  • @CallbackMethod

Utilización de anotaciones

Se pueden utilizar anotaciones siguiendo estos pasos:

  1. El soporte de anotaciones está habilitado de forma predeterminado. El archivo .jar de anotaciones se encuentra en el directorio \annotation del archivo SAClientUtil_<version>_<os>.zip extraído.
  2. Si está escaneando clases precompiladas, archivos .war o archivos .jar, añada com.ibm.security.annotation.jar al proyecto Java que contiene el código fuente anotado y a continuación vuelva a crear el proyecto.
  3. Si está escaneando en un entorno de desarrollo integrado (IDE) que tiene instalado el plug-in de análisis estático, añada com.ibm.security.annotation.jar a la vía de acceso de clases Java.
  4. Si utiliza la CLI de análisis estático, cree un archivo de configuración y en él establezca CustomBuildInfo additional_classpath="dependency_path" para que dependency_path incluya la ubicación del archivo com.ibm.security.annotation.jar.

Se pueden añadir anotaciones al código fuente antes del escaneo, o bien se pueden añadir tras el escaneo y durante la selección para identificar y eliminar los falsos positivos. Si está escaneando en un IDE que tiene instalado un plug-in de análisis estático, habrá disponible un botón Ayuda para arreglar esto en el informe de escaneo. Cuando seleccione este botón, se mostrarán los posibles puntos donde se pueden añadir anotaciones.

Se proporcionan anotaciones para permitirle insertar sus conocimientos directamente en el código fuente en forma de anotaciones de seguridad. Puesto que se pueden utilizar las anotaciones para declarar seguras partes del código, se deben utilizar cuidadosamente. No se deben utilizar para código que se deba escanear en busca de vulnerabilidades de seguridad. Si utiliza anotaciones, un analista de seguridad puede optar por ignorarlas inhabilitando la característica en el archivo irgen-engine.apsettings que se encuentra en el directorio \config del archivo SAClientUtil_<version>_<os>.zip extraído. En este archivo, localice este valor:

<Setting
  name="process_security_annotations"
  value="true"
  default_value="true"
  description="Cuando esté activado, las anotaciones de método se procesarán - utilizadas por DotNet y los LSP de Java"
  display_name="Anotaciones de seguridad de proceso"
  type="bool"
/>

Para inhabilitar la característica, cambie value="true" por value="false".

Nota: Si el valor se establece en value="false", establézcalo en value="true" si desea utilizar anotaciones.

@ValidatorMethod

Los validadores son métodos que realizan comprobaciones en los datos de entrada y a menudo devuelven un valor booleano que indica si la entrada es o no válida. En lugar de aceptar o rechazar la entrada utilizando validadores, puede modificar la entrada del usuario a un formato aceptable. Estos métodos reciben el nombre de sanitizantes.

Utilizando la anotación @ValidatorMethod puede identificar todos los métodos de validador y sanitizante en el código fuente de la aplicación. Durante los escaneos de análisis estático, esta información se utilizará para eliminar los flujos de datos que pasan a través de estos métodos, ya que los datos ahora se consideran seguros.

Nota: Actualmente, no hay ninguna provisión para especificar qué parámetros del método anotado se deben considerar validados. Durante los escaneos de análisis estático, se supondrá que todos los parámetros de entrada se han validado.

@SuppressSecurityTrace

Todos los rastreos que fluyen a través de un método marcado con esta anotación se eliminan. Esto resulta útil cuando un grupo determinado de rastreos se identifica como falsos positivos o menos importantes o interesantes que otros. Puede utilizar esta anotación para filtrar estos rastreos u ocultarlos a fin de reducir el desorden.

@CallbackMethod

Esta anotación se utiliza para identificar devoluciones de llamada o puntos de entrada a una aplicación. Todos los argumentos se consideran corruptos.

Configuración de expresiones include PHP no resueltas y de clases PHP desconocidas

Tras generar un archivo IRX para PHP, se creará un archivo de configuración que le permite configurar la generación de IRX la próxima vez que escanee los archivos.

El archivo de configuración, php-scan-config.xml, se genera en la ubicación desde la que ha emitido el mandato prepare. El archivo contiene dos secciones, <UserInputs> y <Php-Unknown-Configuration>.

La sección <UserInputs> contiene valores de configuración que se pueden modificar para el escaneo de PHP. La sección está prerrellenada con valores de ejemplo. Puede añadir a estos valores utilizando un punto y coma (;). Los valores de esta sección son los siguientes:

  • <RegularExpression>: Es una característica avanzada que permite que se especifique una expresión regular para el texto de búsqueda.
  • <ReplaceText>: Este valor se utiliza para la búsqueda y sustitución de texto simple. El texto de búsqueda se utiliza tal cual y, si se encuentra en cualquier lugar del texto incluido, se sustituye con el texto de sustitución.
  • <Function>: Este valor se utiliza cuando se va a sustituir una llamada a función. El texto de búsqueda debe ser el nombre de la función sin paréntesis. El texto de búsqueda se aumentará para buscar una llamada a función con el nombre especificado seguido por paréntesis y coincidirá con cualquier elemento dentro de los paréntesis.
  • <SimpleText>: Texto de sustitución simplificado. Utilícelo cuando el texto de sustitución sea una serie única.
  • <SimpleRegEx>: Expresión regular de sustitución simplificada. Utilícela cuando la expresión regular de sustitución sea una serie única.
  • <AdditionalIncludePaths>: Utilice este valor para añadir vías de acceso include adicionales que contienen archivos referenciados.

La sección <Php-Unknown-Configuration> contiene todas las expresiones include no resueltas y las clases desconocidas que se han encontrado en el último escaneo. Utilice esta información para completar la sección <UserInputs>.

  • <Summary>: Resume el número de expresiones include desconocidas (<NumberOfUnknownIncludeExprs>) y clases desconocidas (<NumberOfUnknownClasses>) del último escaneo.
  • <SourceFile>: Todas las expresiones include y las clases desconocidas se listan de forma alfabética por nombre de archivo de origen. Para cada archivo de origen, la vía de acceso de archivos (<FilePath>) lista la vía de acceso absoluta del origen, la clase desconocida (<UnknownClass>) y la expresión include desconocida (<UnknownIncludes>).

Etiqueta Descripción

Gestión de exclusiones Java y .NET de terceros

De forma predeterminada, el código Java y .NET de terceros no se escanea durante la generación de archivos de IRX. Puede gestionar el código de terceros que se excluye siguiendo las instrucciones de este tema.

Acerca de esta tarea

Cuando genera un archivo IRX para el código, el código de terceros se excluye de manera que la evaluación no incluye la información y los hallazgos sobre el código que no le interesa analizar. Para modificar lo que se excluye, siga los pasos de este tema. Si, en su lugar, desea incluir código de terceros al escanear, consulte estos temas:

  • Generación de un archivo IRX con la interfaz de línea de mandatos (CLI)
  • Generación de un archivo IRX para un proyecto Maven
  • Escaneo en entornos de desarrollo integrado

Procedimiento

  1. Localice el directorio config del archivo SAClientUtil_<version>_<os>.zip extraído (donde <versión> es la versión actual del programa de utilidad cliente).
  2. En este directorio, localice el archivo dot_net.exclusions (.NET) o java.exclusions (Java).
    Nota: Otros archivos .exclusions de este directorio son experimentales y si los modifica las exclusiones de terceros no resultarán afectadas.
  3. Abra el archivo en un editor de texto, donde verá que es XML con el formato siguiente: 
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<Exclusions>
  <packages>
    <package name="package_name"/>
  </packages>
  ...
</Exclusions>
  4. Edite el archivo para añadir los paquetes que se excluirán.
    Nota:
    • Los nombres de paquete especificados coincidirán con todos los paquetes que tengan el mismo prefijo (se tratarán como si fueran seguidos por un comodín). Por ejemplo, si añade <package name="com.mycompany.common"/> como una exclusión, <package name="com.mycompany.common.action1"/> y <package name="com.mycompany.common.action2"/> se excluirán.
    • Solo las entradas que se añaden al contenedor <packages> afectan a las exclusiones de terceros. La adición de clases y métodos no afectará a las exclusiones.
  5. Guarde el archivo y a continuación genere un archivo IRX para el código de origen.

Qué hacer a continuación

Si modifica los archivos de exclusión, debe seguir estos métodos recomendados:
  • Guarde una copia de seguridad de los archivos de exclusión modificados. En concreto, recuerde que, si actualiza a una nueva versión del Programa de utilidad de cliente de Static Analyzer, descargará versiones de los archivos de exclusión sin las actualizaciones. Cuando actualice el programa de utilidad cliente, guarde una copia de los archivos de exclusión modificados y a continuación cópielos sobre el nuevo programa de utilidad cliente.
  • Guarde las modificaciones en un solo bloque de entradas XML de forma que pueda copiarlas y pegarlas rápidamente si lo necesita.

  • -d: Especifique -d <vía_acceso_archivo>, donde <vía_acceso_archivo> es la vía de acceso completa del archivo de destino y/o el nombre del archivo de destino. Si no se especifica ningún nombre de archivo, el nombre de archivo se basará en el nombre del trabajo de escaneo. Si no se especifica ninguna vía de acceso, el archivo se guardará en el directorio actual. Si no se incluye esta opción, el archivo se guardará en el directorio actual con un nombre que se base en el nombre del trabajo de escaneo.
  • -t: Especifique -t <tipo>, donde <tipo> es html o zip. Los resultados se guardarán como un archivo HTML o como un archivo .zip que contenga los resultados HTML. Si no se incluye esta opción, los resultados se guardarán como un archivo HTML.

    Si los resultados del escaneo corresponden a un archivo IRX generado mediante el mandato package, si especifica -t zip se guardarán los resultados que contengan un nuevo archivo .ozasmt que se pueda cargar en el producto AppScan Source Versión 9.0 o posterior.

Consejo: Para todos los mandatos, las opciones se pueden utilizar en cualquier orden.
